# Phase 02.1 Plan 04: Suggestion-Only Clarification Call Summary

**A cheap (draft/Kimi) tier suggestion-only call maps an unresolved name back to the likely roster employee (David Reyez → David Reyes) PURELY for the clarification email copy — wired strictly after the deterministic gate, degrading to {} on any failure, and structurally walled off from decide so the LLM's nickname/typo judgment can never reach a money-moving decision (D-21-05, the new Phase 2 hero).**

## Accomplishments

- **New suggestion-only stage (Task 1):** `app/pipeline/suggest.py` — `suggest_employees(unresolved_names, roster, *, llm)` makes ONE `call_structured("draft", ...)` against the new `NameSuggestionResponse` wrapper and returns `{submitted_name: roster_full_name}` for only the names the model is reasonably sure about. It degrades to `{}` on ANY failure — API error, empty content, parse/validation error, a `null`/unknown suggestion, or a suggested name that is not an actual roster employee — and **never raises** out of the stage (the WR-03 pattern applied to a structured call). An empty unresolved list makes **no LLM call**. PURE (Roster + names in, mapping out, no DB) so the Phase 4 eval can reuse it (D-21-09).
- **New suggestion prompt (Task 1):** `app/llm/prompts/suggest.py` — a JSON-mode prompt carrying the literal word `json` + an example shape (the DeepSeek convention kept across structured calls); the system prompt states the result is **email copy only and decides nothing**, and that a suggested name must be an exact roster full_name or `null`.
- **Specific clarification copy (Task 2):** `compose_clarification` gained a keyword-only `suggestions=` threaded into BOTH the draft prompt (so the model writes "did you mean David Reyes?") AND the deterministic `_template_body` floor (so the specific ask survives even a total draft failure — WR-03 preserved). Names with a suggestion get a per-name "did you mean X?" line; names without fall back to the generic bundled ask.
- **Orchestrator wiring (Task 2):** `_clarify` now takes `roster` and calls `suggest_employees(decision.unresolved_names, roster, llm=...)` **strictly after `decide` has returned** (decision is computed upstream in `_run_stages` and passed in), then composes with `suggestions=`. `suggest` is imported in the orchestrator, **never in decide**.
- **Isolation proven by tests:** the suggested employee id never leaks into the persisted Decision / reconciliation (the unresolved name stays `matched_employee_id=None`, `final_action="request_clarification"`); a source-level test asserts `suggest_employees(` appears AFTER `decide(` in the orchestrator and is never an argument to `decide`; and `decide.py` is grep-clean of "suggest".

## Decisions Made

- **Suggestion rides the existing draft (Kimi) tier** — no third tier returns; the two-tier steady state from Plan 03 holds.
- **`suggest_employees` is invoked only inside `_clarify`, strictly after `decide()` returns in `_run_stages`** — the decision is a parameter computed upstream, so the suggestion can never precede or feed it (enforced by a source-level ordering test + the decide-grep guard).
- **`llm` is passed to `suggest_employees` only when injected** (mirrors `compose_clarification`) — in production `llm` is `None`, and passing `llm=None` would force the cheap call onto a `None` client and silently template every clarification; instead the stage binds its own default client.
- **A suggested name must be an exact roster full_name or it is dropped** (a clarification can never claim a non-existent employee); a `null`/unknown suggestion yields no entry so the caller falls back to the generic ask.
