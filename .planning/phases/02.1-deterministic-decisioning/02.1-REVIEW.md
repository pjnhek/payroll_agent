# Codex Code Review — Phase 2.1 (Deterministic Decisioning)

Reviewed the 17 changed source files + the design contract against the intended deterministic design. All 5 findings verified real against the code and FIXED with regression tests (199 passed, +4).

## Findings (all confirmed + fixed)

### HIGH — cross-tier exact-vs-alias collision (reconcile_names.py)
A name that was one employee's exact `full_name` AND a *different* employee's `known_alias` resolved to the first (source=exact, resolved=True) without noticing the second candidate — uniqueness was enforced WITHIN each tier, not ACROSS them. So an ambiguous name could reach `process` and pay the wrong employee. **Fix:** `deterministic_match` now computes the set of DISTINCT candidate employees across BOTH full_name and alias matches; resolves only if exactly one, else `none`. Test: `test_name_is_one_employees_fullname_and_another_employees_alias_does_not_resolve`.

### MEDIUM — decide() didn't fail closed on a non-one-for-one match set (decide.py)
`decide` is a pure public function the eval calls with arbitrary inputs; it trusted that `matches` had exactly one entry per extracted employee. A missing record would let an employee be silently dropped from a `process` run (orchestrator `continue`s past unmatched). **Fix:** Rule 0b gates closed when the submitted-name multiset of `matches` != that of `extracted.employees`. Tests: missing-record + extra-record clarify.

### MEDIUM — NameMatchResult had no semantic invariant (roster.py)
Impossible states (`source=none, resolved=True`) were constructible and `decide` trusts `resolved`. **Fix:** `model_validator` — source=none ⟺ resolved=False + id=None; source=exact|alias ⟺ resolved=True + non-null id. Test: rejects 3 impossible combos.

### MEDIUM — PII in suggestion logs (suggest.py)
Failure log used `exc_info=True` (traceback can echo names) and the hallucinated-suggestion warning logged submitted + suggested employee names. **Fix:** log failure TYPE only, no exc_info; count dropped hallucinations instead of naming them.

### LOW — f-string SQL in bootstrap --reset (bootstrap.py)
Reset DROP used an f-string (trusted table list, no injection risk, but violates the 'never f-string SQL' rule). **Fix:** `psycopg.sql.SQL(...).format(sql.Identifier(table))`.

## Verified clean by Codex
decide.py has no LLM call/import; reconcile_names has no llm/DB seam; orchestrator branches only on final_action; suggest_employees is called only in _clarify after decide returns.
