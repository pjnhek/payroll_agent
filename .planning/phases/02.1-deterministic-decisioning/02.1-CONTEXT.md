# Phase 2.1: Deterministic Decisioning + Alias Read-Side ‚Äî Context

**Gathered:** 2026-06-21
**Status:** Ready for planning
**Type:** Decimal phase (INSERTED) ‚Äî a re-architecture of the Phase 2 decision path against finished, working code (Phase 2 = 168 tests green).

<domain>
## Why this phase exists

Phase 2 shipped a **confidence-threshold gate**: the LLM proposed `process`/`request_clarification` with a per-name confidence, and `decide.py` blocked anything below 0.8. The intended hero ("model says process but confidence < 0.8 ‚Üí code blocks") **is not a real state** for well-calibrated models: a model uncertain about a name proposes `request_clarification` on its own, so `process + sub-0.8` only occurs if the prompt is rigged. The shipped rubric literally hardcoded a nickname to 0.75 to force the gate ‚Äî a contrivance.

**This phase replaces confidence-gated LLM judgment with deterministic decisioning that never guesses on a money-moving decision.** The new thesis (recruiter-facing): *auditable, reproducible decisioning ‚Äî the system resolves names against the roster deterministically, never guesses, clarifies intelligently with a specific suggested employee, and is safe on collisions.* The "learns the client's shorthand and stops asking" payoff is real but its WRITE side moves to Phase 5 (operator-gated); the READ side lands here.

</domain>

<decisions>
## Locked Decisions

### D-21-01: Deterministic decision model (no confidence score anywhere)
`decide.py` is now **pure code over resolution facts** ‚Äî no LLM call, no confidence number. Per submitted name, resolve against the roster:
- **exact** normalized match (casefold + whitespace-normalize) to exactly one employee, no other employee shares it ‚Üí `resolved`, `source="exact"`
- matches a stored **known_alias** for exactly one employee, no collision ‚Üí `resolved`, `source="alias"`
- anything else (no match, typo, first-time nickname/alias, garbled, ambiguous) ‚Üí `unresolved`, `source="none"`

### D-21-02: Collisions always clarify, never guess (run-level checks in decide.py)
Collision detection stays a **run-level check in decide.py** (NOT folded into per-name `resolved` ‚Äî a name can be `resolved=True` while the run still clarifies on a cross-name collision). Force `request_clarification` if ANY:
- a submitted name resolves to 2+ employees (ambiguous)
- two+ submitted names resolve to the same employee
- a name is shared by 2+ roster employees (it can't be uniquely resolved)

### D-21-03: Run-level decision (final_action)
- every name `resolved` AND one-to-one mapping holds AND required fields present ‚Üí `final_action="process"`
- any `unresolved` name OR any collision OR any missing required field ‚Üí `final_action="request_clarification"`, with `gate_reasons` listing exactly what triggered it.
- `final_action` remains the **SOLE branch source** for orchestrator/dashboard/eval. There is no longer a separate model action to diverge from ‚Äî the decision is deterministic.

### D-21-04: Contract changes
**Decision (`app/models/contracts.py`):** remove `model_action`, `confidence`, `gate_triggered`, `reasons`. Keep `final_action`, `gate_reasons`, `unresolved_names`, `missing_fields`. ADD the per-name resolutions (a `list[NameMatchResult]`, or a list of resolution dicts) so the dashboard/eval can read why each name resolved/didn't ‚Äî folded into the `decision` JSONB (no separate table, see D-21-06).

**NameMatchResult (`app/models/roster.py`):** drop `confidence` and `match_type`. New shape: `submitted_name`, `matched_employee_id: UUID | None`, `source: Literal["exact","alias","none"]`, `resolved: bool`, `reason: str`. (`resolved` is explicit, not derived, for legibility.)

### D-21-05: The LLM's role shrinks ‚Äî extraction + suggestion only
- **Extraction (LLM-03):** unchanged ‚Äî strong tier (DeepSeek), the hardest task.
- **Decision LLM call: REMOVED.** `_ask_model`, `_ModelAdvice`, the reconcile-LLM layer-2 step, the `NameReconciliationResponse` wrapper, and the reconcile/decide prompts become dead code for the decision. Delete them.
- **NEW ‚Äî clarification-suggestion call (suggestion-only):** for an `unresolved` name, ONE cheap-tier LLM call suggests the most likely intended employee, used ONLY to make the clarification email specific ("did you mean David Perez?"). It NEVER feeds `decide` / `final_action`. This is where nickname knowledge (Bob‚ÜíRobert) beats plain string distance ‚Äî keeps the LLM doing real judgment beyond extraction.
- **Config: drop the mid/decision model tier entirely.** Two model IDs now: DeepSeek (extraction) + one Kimi (suggestion + drafting). Update `app/config.py` Settings + `.env.example` + `.env`.

### D-21-06: Drop the name_matches relational table
The DB has a relational `name_matches` table with a `match_confidence` column (written by `repo.persist_reconciliation`). **Drop it.** Fold per-name resolutions into the `decision` JSONB (`submitted_name`, `matched_employee_id`, `source`, `resolved`, `reason`). The eval scores the reconcile function's return value (not the DB); the dashboard reads resolutions from the JSONB. **No relational consumer remains, so the table goes.**
- **Apply the DROP to BOTH live local AND Supabase** ‚Äî `CREATE TABLE IF NOT EXISTS` will NOT alter an existing table, so the planner must include an explicit `DROP TABLE IF EXISTS name_matches` applied via bootstrap/migration to the live DBs, not just edited in `schema.sql`. The `reconciliation` JSONB column on `payroll_runs` stays (it already holds the list); `match_confidence` disappears with the table. Status-drift guard must stay green.

### D-21-07: Learning loop ‚Äî READ now, WRITE in Phase 5 (operator-gated)
- **READ side (THIS phase):** resolve a submitted name against an employee's stored `known_aliases` (already a seeded `TEXT[]` column on `employees`). The deterministic gate + eval use it; testable now against seed aliases. No write path.
- **WRITE side (Phase 5):** persist a newly-confirmed alias so the system stops asking. **Persist at the operator-approval gate** ‚Äî when the operator approves a run where a first-time shorthand was clarified, THAT approval is the confirmation; save the original submitted name as a `known_alias` for that employee. This reuses the single human gate (no free-text confirm-vs-correct parsing of client replies) and keeps the invariant that no auto-routing state change happens without the human. **Out of scope here** ‚Äî record as a Phase 5 deliverable.

### D-21-08: Preserve the Phase 2 CR-01 fix (empty extraction)
Zero employees extracted ‚Üí `request_clarification` (gate reason "no employees could be extracted"). Carries over unchanged.

### D-21-09: DRY seam intact
`reconcile_names` and `decide` remain **pure importable functions** (data in, data out, no DB) ‚Äî the Phase 4 eval imports the SAME functions. The deterministic resolver must stay pure; the alias READ uses the `Roster`/`Employee.known_aliases` passed in, not a DB call inside the stage.

</decisions>
