<objective>
Create the durable authorization record that turns the current epoch preflight into a provider-handoff fence.

Purpose: the old handler can read epoch 0 and then call Resend after a retrigger commits epoch 1. A durable active handoff must make the epoch-generation boundary linearizable without holding a database transaction across Resend.
Output: deployed-schema-safe provider-handoff state, a typed repository contract, and schema/repository regressions before consumers are rewired in Plan 20-22.

Artifacts this phase produces: `outbound_provider_handoffs` with an active-run uniqueness fence and retained owner-lease expiry; `ProviderHandoffAuthorization`, `ProviderHandoffRecordOnly`, and `ProviderHandoffActive`; `authorize_outbound_provider_handoff`, `adopt_outbound_provider_handoff`, `finalize_outbound_provider_handoff`, `release_outbound_provider_handoff_for_retry`, `assert_no_active_outbound_provider_handoff`, and explicit review-only supersession seams; plus SQL-shape regressions for the job → snapshot → run → handoff lock protocol.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|---|---|
| Exact queue lease → provider authorization | A worker with stale or reclaimed transport ownership must not acquire authority for an external email. |
| Immutable snapshot/current run epoch → handoff row | Frozen PII bytes and the current conversation generation together determine whether a provider call is authorized. |
| Operator retrigger → active handoff | A human mutation can intentionally create a new epoch, so it must not cross an unresolved provider handoff silently. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|---|---|---|---|---|---|
| T-20-21-01 | Tampering | provider authorization | critical | mitigate | Exact lease, snapshot, status, record-only flag, epoch, and cutoff checks are locked and persisted before any provider request; only an expired matching predecessor handoff can transfer to the current exact lease, and the authorization returns a non-renewable reservation-derived deadline. |
| T-20-21-02 | Tampering | clear_reply_context | critical | mitigate | Active handoff fence makes the retrigger transaction reject/roll back before it increments reply_epoch. |
| T-20-21-03 | Denial of service | lock protocol | high | mitigate | Worker order is job → snapshot → run → handoff; clear/retrigger locks run then reads only handoff, and no provider call sits in a transaction. |
| T-20-21-04 | Information disclosure | handoff diagnostics | high | mitigate | Handoff rows contain UUIDs, epoch, timestamps, and fixed codes only; snapshot bytes remain behind existing scoped readers. |
| T-20-21-SC | Tampering | uv package installation | high | mitigate | No package installation is planned; use the committed uv environment only. |
</threat_model>
