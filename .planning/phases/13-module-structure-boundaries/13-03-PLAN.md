<objective>
Split `app/main.py` (1,857 lines) into `app/routes/` -- five APIRouter modules by URL-prefix
concern (`webhook.py`, `runs.py`, `dashboard.py`, `demo.py`, `health.py`) plus two shared modules
(`pipeline_glue.py` for the HTTP-to-orchestrator bridge helpers, `templating.py` for the Jinja2
instance + badge filters) -- leaving `main.py` as thin app assembly (~15-30 lines). Every shared
bridge helper (`_run_pipeline`, `_resume_pipeline`, `_route_reply`, `_reply_sender_ok`,
`_row_to_inbound`, `_finish_reply_resume`, `_operator_resume`) is promoted to a public name in
`pipeline_glue.py` and wired via module-object imports.

This plan was revised after a cross-AI (Codex) review found the webhook rewiring was factually
wrong (would have regressed a Phase 9 dedup/race guarantee), the router import sets were
non-executable, several routers lacked logger bindings, and a new cross-module private import would
have failed Plan 04's own guard. Fixes applied here (each independently re-verified against live
source during this revision, per the replan instructions -- and one Codex claim about
`test_hitl.py` was found to reference a DIFFERENT coupling point than literally described, corrected
below):

1. **Webhook flow fidelity (Codex HIGH, CONFIRMED against source):** `inbound()` (main.py ~304-577)
   does NOT call `_route_reply`. Verified directly: the transaction block (lines ~406-536)
   classifies the reply INLINE via `repo.find_awaiting_reply_for_header`/`find_business_by_sender`/
   `create_run` and sets a local `outcome` variable; AFTER the `with` block commits, `if outcome ==
   "reply_candidate":` calls `_finish_reply_resume(reply_run_id, email, cleaned, background_tasks)`

   `simulate_reply` (the demo-only affordance, confirmed via its own docstring: "The real webhook's
   `inbound()` route does NOT call this"). Replacing `inbound()`'s call with `route_reply` would
   re-run the header lookup OUTSIDE the transaction, regressing the Phase 9 dedup/race guarantee
   (Codex HIGH-1 fix, cited in `_route_reply`'s own docstring). This plan's Task 1 wires `inbound()`
   to call `pipeline_glue.finish_reply_resume` (not `route_reply`), preserving the exact pre-split
   control flow.
2. **Non-executable router import sets (Codex HIGH, confirmed via source read):** verified exactly
   which stdlib/fastapi names each router needs by reading the actual route bodies, not assuming a
   generic import list: `dashboard.py` needs `Query` (landing, ~line 1146), `Path`/`json`

   logging.getLogger(...)` binding -- the pre-split file has exactly ONE logger
