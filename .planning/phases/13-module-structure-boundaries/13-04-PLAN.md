<objective>
Add the AST-walking BOUND-01 regression guard (D-15) that RESEARCH.md proved is the only
mechanism catching this codebase's actual violation shapes (ruff's PLC2701 has a same-package
exemption and does not inspect function bodies at all). Validate the guard's detection logic
against synthetic fixtures BEFORE trusting it as a permanent gate, then confirm it passes clean
against the fully-split codebase produced by 13-01/13-02/13-03. This is the final phase-closing
plan: it also re-runs the complete suite one last time as the phase-level STRUCT-04 sign-off.

This plan was revised after a cross-AI (Codex) review found the originally-specified guard would
miss `module._private` attribute access entirely, mis-resolve relative imports and `__init__.py`
module names, and rely on broad text-grep acceptance criteria that are impossible to satisfy as
written. Fixes applied here (each independently re-verified against live source/behavior during
this revision):

1. **`module._private` attribute access (Codex HIGH, confirmed real and load-bearing):** the
   originally-specified guard only inspects `ast.ImportFrom` nodes (`from module import _name`).
   Verified live: `scripts/show_confirmation_subject.py`, `scripts/reset_stuck_runs.py`, and
   `scripts/demo_reset.py` (x3 sites) all do `import app.db.repo as repo` (or `as _repo`) then call
   `repo._conn_ctx(None)` -- a plain `ast.Import` followed by attribute access, invisible to an
   `ImportFrom`-only scanner. This is a REAL, currently-in-use pattern this codebase relies on
   (13-01's facade deliberately re-exports `_conn_ctx` for exactly this reason) -- the guard must
   either (a) also flag `module._private` attribute access as a violation (matching the roadmap's
   literal success criterion: "no function body anywhere imports a _private name from another
   module" -- extended here to attribute access, since an attribute reference is functionally
   identical exposure), or (b) explicitly scope BOUND-01 enforcement to the `ImportFrom` form only,
   with a documented rationale for why `module._private` attribute access is treated differently.
   This plan adopts (a) -- extend the scanner -- per RESEARCH's own recommendation ("Prefer
   extending; the roadmap success criterion says 'no function body anywhere imports a _private name
   from another module'"), while EXPLICITLY exempting the already-approved, deliberate exceptions:
   `repo._conn_ctx`/`repo._scrub`/`repo._TERMINAL_STATUSES`/`repo._ACCENT_CLASS_MAP`/
   `repo._pad_references` (13-01's facade intentionally re-exports these as part of its documented
   compatibility surface, D-01/D-14) are accessed via the `app.db.repo` PACKAGE facade, which is
   itself the declared public boundary for that subsystem -- the guard must not flag a facade's OWN
   re-exported private-looking names as violations of ITSELF. See Task 1 for the precise scoping
   rule that resolves this without hand-picking exceptions.
2. **Relative imports and `__init__.py` module naming (Codex MEDIUM, confirmed):** the originally-
   specified module-name computation ignores `node.level` (relative-import depth) and computes
   `app/routes/__init__.py`'s module name as `app.routes.__init__` instead of `app.routes` --
   both cause false positives/negatives. Fixed in Task 1's implementation.
3. **Impossible grep acceptance criteria (Codex MEDIUM, confirmed):** the originally-specified
   "final sweep" step greps the whole tree for literal strings like `_HOURS_FIELDS`/`_is_paid` and
   expects zero matches outside each name's defining module -- but `app/pipeline/calculate.py`
   legitimately retains its OWN, textually-identical, same-module-only `_HOURS_FIELDS` tuple
   (RESEARCH Pitfall 5, explicitly out of scope for de-duplication), and comments elsewhere may
   mention `_is_paid` descriptively without importing it. A text grep cannot distinguish "same-
   module definition" or "descriptive comment" from "cross-module import." Fixed: acceptance
   criteria in this plan use the SCANNER'S OWN AST-based output (`assert not violations`), never a
   standalone grep, as the source of truth for "zero violations remain."
4. **Synthetic-positive check via `tmp_path`, not a real-repo scratch file (Codex MEDIUM, confirmed
   and adopted):** the originally-specified validation step created and deleted a real file under
   `app/` as a manual, uncommitted sanity check. Fixed: this plan adds a PERMANENT, COMMITTED unit
   test (`test_scanner_detects_synthetic_violation`) that constructs synthetic violating and
   non-violating source trees under pytest's `tmp_path` fixture and asserts the scanner's own
   detection function classifies each correctly -- proving the guard's correctness as part of the
   permanent suite, not a one-time manual step that leaves no lasting evidence it was ever done.

Purpose: BOUND-01 (the enforcement mechanism, not just the one-time fixes) + STRUCT-04 (final
full-suite behavior-neutrality confirmation across the whole phase).
Output: `tests/test_bound01_private_imports.py`, a new permanent test module in the suite

criteria.

**Scanner design (corrected per the Codex findings above):**

`SCAN_ROOTS = ["app", "eval", "scripts"]` -- unchanged from the original design; do NOT scan
`tests/` (D-14: tests intentionally keep same-module-private access for unit-testing internals,
which is same-module by construction and outside this guard's cross-module scope).

For each `.py` file under each scan root:
1. Compute the file's own dotted module name CORRECTLY relative to the scan root's PARENT
   directory, with `__init__.py` normalization: `app/routes/__init__.py` -> module name
   `app.routes` (NOT `app.routes.__init__`); `app/pipeline/orchestrator.py` -> `app.pipeline.orchestrator`.
   Implement this explicitly (e.g. strip a trailing `__init__` path segment before joining with
   dots) and cover it with the synthetic-fixture unit test in step 2.
2. Parse with `ast.parse`, then `ast.walk` the FULL tree (module-level AND function-body nodes --
   `ast.walk` already descends into `FunctionDef`/`AsyncFunctionDef` bodies automatically; do not
   restrict to `tree.body`).
3. For every `ast.ImportFrom` node: if `node.level > 0` (a relative import, e.g. `from . import
   x` or `from .sibling import y`), resolve the effective target module by walking UP the
   importing file's own module path `node.level` times before appending `node.module` (if any) --
   a same-package relative import must never be flagged as a violation just because it happens to
   look like a "different module" under a naive string comparison. If `node.level == 0` and
   `node.module` is not None and differs from the importing file's own module name, flag every
   name in `node.names` whose `alias.name` starts with `_` and does not start with `__` (dunder).
4. For every `ast.Attribute` node whose `.attr` starts with `_` and does not start with `__`: walk
   the attribute's `.value` chain to determine if it resolves to a locally-bound name that was
   imported via a module-level `ast.Import` (`import X` or `import X as Y`) elsewhere in the SAME
   file, where `X` (the imported module) is NOT the file's own module and is NOT a package the
   file's own module belongs to (i.e. genuinely a different top-level component under one of
   `SCAN_ROOTS`). If so, flag it as a `module._private` attribute-access violation. This is the
   pattern that catches `scripts/*.py`'s `repo._conn_ctx(...)` (via `import app.db.repo as repo`)
   while NOT flagging same-module `self._helper()`-style attribute access or unrelated attribute
   chains (e.g. `request.headers._some_internal` on a third-party object) -- scope the attribute
   check specifically to attributes of names bound by a LOCAL `ast.Import` of a first-party module
   under `app`/`eval`/`scripts`, not every `ast.Attribute` node in the file (which would produce
   excessive false positives on arbitrary object attribute access).
5. **Facade-boundary exemption (resolves the `_conn_ctx`/`_scrub`/etc. tension from finding #1
   above):** the `app/db/repo/__init__.py` facade is the DECLARED public boundary for the repo
   subsystem (13-01's D-01) -- it deliberately re-exports `_conn_ctx`, `_scrub`, `_TERMINAL_STATUSES`,
   `_ACCENT_CLASS_MAP`, `_pad_references` as part of its documented compatibility surface. Do NOT
   hand-pick these five names as a hardcoded exemption list (fragile, drifts). Instead, scope the
   attribute-access check to only flag `module._private` where `module` is bound to a SUBMODULE
   import (`import app.db.repo.runs as X`, `import app.pipeline.clarification as X`) -- i.e. an
   import of something that is NOT itself a package `__init__.py`. Accessing `repo._conn_ctx` where
   `repo` is bound via `import app.db.repo as repo` (importing the PACKAGE, whose `__init__.py` IS
   the declared facade) is explicitly OUT of this guard's scope, because the package's own
   `__init__.py` re-exporting a name it deliberately chose to expose is the facade pattern working
   as designed, not a violation of it. Verify this scoping rule against the live `scripts/*.py`
   call sites (`repo._conn_ctx(None)` where `repo`/`_repo` is bound via `import app.db.repo as
   repo`) to confirm they are correctly NOT flagged, and against a synthetic
   `import app.pipeline.some_submodule as X; X._private_helper()` case (which SHOULD be flagged)
   in the unit test from step 2.
