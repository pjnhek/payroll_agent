Create `tests/test_bound01_private_imports.py` using RESEARCH.md's AST-based guard code example as
the direct source (not a codebase analog -- no existing test in this repo does AST-walking).
The guard: `SCAN_ROOTS = ["app", "eval", "scripts"]`; for every `.py` file under each root, parse
it with `ast.parse`, walk the tree with `ast.walk`, and for every `ast.ImportFrom` node whose
`node.module` is not None and differs from the importing file's own dotted module name, flag every
imported name in `node.names` that starts with `_` and does not start with `__` (dunder). This
MUST catch imports nested inside function bodies (ast.walk descends into FunctionDef/AsyncFunctionDef
bodies automatically -- do not restrict the walk to only `tree.body`, which would only see
module-level statements). Do NOT scan `tests/` (D-14: tests intentionally keep same-module-private
access for unit-testing internals like `calculate._money`, which is same-module by construction
and outside this guard's cross-module scope; scanning `tests/` risks false positives from the
suite's own legitimate direct-private-import patterns like `from app.pipeline.reconcile_names
import _norm` fixtures that predate this phase in test-only files not in scope for BOUND-01 per
D-14's explicit "tests may keep importing same-module privates" carve-out -- but since this guard
is cross-module-only by construction anyway, confirm no test file trips it and exclude `tests/`
from `SCAN_ROOTS` explicitly, matching D-15's stated scope of "runtime code only").

Compute each file's own module name correctly relative to its scan root (e.g.
`app/pipeline/orchestrator.py` under root `app` is module `app.pipeline.orchestrator`) so a
same-module `from app.pipeline import orchestrator` inside `orchestrator.py` itself (a degenerate
self-import, unlikely but should not false-flag) and legitimate intra-package `from . import x`
relative imports are not miscounted as cross-module violations.

The test function `test_no_cross_module_private_imports` asserts `not violations`, with a failure
message listing every `{file}:{lineno} imports private '{name}' from '{module}'` violation found,
so a future regression is immediately diagnosable from the pytest failure output alone.
