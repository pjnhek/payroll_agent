1. **Relative-import underscore check not explicitly applied after resolution (Codex Round 2
   MEDIUM, confirmed):** the prior revision's step 3 described resolving `node.level > 0` to an
   absolute module path, but then stated the underscore-name flagging rule ("if `node.level == 0`
   and `node.module` is not None and differs... flag every name...") as conditioned specifically on
   the `node.level == 0` branch, leaving the `node.level > 0` branch's post-resolution flagging
   implicit rather than explicit. A careless implementation could resolve the relative import's
   target module correctly but forget to run the SAME underscore check against it, silently
   creating a blind spot for `from .sibling import _private`. Fixed here: Task 1's action states
   explicitly that BOTH branches (level == 0 absolute, and level > 0 after resolution) converge on
   ONE shared flagging step -- there are two ways to determine the target module (direct vs.
   resolved-relative), but exactly one way to decide whether to flag a name once the target module
   is known, and Task 1 now says so in one place rather than describing it once per branch.
2. **`is_package` not preserved during relative-import resolution inside `__init__.py` (Codex
   Round 2 MEDIUM, confirmed):** normalizing `app/routes/__init__.py`'s OWN module name to
   `app.routes` (not `app.routes.__init__`) is necessary but not sufficient -- when a relative
   import appears INSIDE that `__init__.py` file, resolving `node.level` relative imports must walk
   up from the PACKAGE's own module name (`app.routes`), not from a name that has already dropped
   the `__init__` segment as if it were an ordinary module inside `app`. Concretely: `level=1`
   inside `app/routes/__init__.py` (`from . import webhook`) must resolve to `app.routes.webhook`
   (siblings within the SAME package), not `app.webhook` (one level too shallow, since dropping
   `__init__.py`'s module-name segment already "used up" one level of nesting conceptually). Fixed
   here: Task 1 requires tracking an explicit `is_package: bool` alongside each file's own module
   name (`True` for any `__init__.py`), and the relative-import resolution algorithm's level-walking
   step accounts for it (an `__init__.py`'s OWN module name already points AT the package, so a
   `level=1` relative import resolves relative to that same name, not one level further up).
3. **Facade exemption scope, re-confirmed as already correctly narrow (Codex Round 2 MEDIUM item
   was actually about a DIFFERENT, deeper concern than what was fixed -- re-verified this session
   this part of the prior design is sound and needs no further change):** the prior revision's step
   5 already scopes the facade exemption to "imported target is itself a package `__init__.py`",

   two items above, not a flaw in the exemption's own scoping logic. Task 1's synthetic fixture
   unit test still includes both the exempt case (package import) and the non-exempt case (submodule
   import) to prove this stays correct.

**Scanner design (corrected per the Codex Round 2 findings above):**

`SCAN_ROOTS = ["app", "eval", "scripts"]` -- unchanged from the original design; do NOT scan

For each `.py` file under each scan root:

1. **Compute the file's own dotted module name AND an `is_package` flag.** Module name: relative
   to the scan root's PARENT directory, with `__init__.py` normalization -- strip a trailing
   `__init__` path segment before joining with dots (`app/routes/__init__.py` -> `app.routes`, NOT
   `app.routes.__init__`; `app/pipeline/orchestrator.py` -> `app.pipeline.orchestrator`).
   `is_package = True` when the file IS an `__init__.py`, `False` otherwise. Both values are needed
   by step 3's relative-import resolution below -- compute and carry both together, not just the
   module name string.

2. Parse with `ast.parse`, then `ast.walk` the FULL tree (module-level AND function-body nodes --
   `ast.walk` already descends into `FunctionDef`/`AsyncFunctionDef` bodies automatically; do not
   restrict to `tree.body`).

3. **Resolve every `ast.ImportFrom` node to a single target-module string, THEN apply ONE shared
   underscore-flagging step to whatever that target module resolves to** (this is the fix for Codex
   Round 2 MEDIUM finding #1 -- do not describe the flagging rule twice, once per branch; describe
   resolution twice, then flag once):
   - **If `node.level == 0`** (absolute import, e.g. `from app.pipeline import clarification`):
     the target module is `node.module` directly (may be `None` for a bare `from . import x` --
     that case only occurs with `level > 0`, so `node.module` is guaranteed non-None here).
   - **If `node.level > 0`** (relative import, e.g. `from . import x` or `from .sibling import y`):
     resolve the target module by walking UP from the IMPORTING FILE's own module name `node.level`
     times, honoring `is_package` (fix for Codex Round 2 MEDIUM finding #2): if the importing file
     `is_package` (it IS an `__init__.py`, whose module name already points AT the package, e.g.
     `app.routes`), a `level=1` relative import resolves relative to THAT SAME name (`app.routes`),
     not one level further up (`app`) -- because dropping the `__init__` segment during step 1
     already accounts for one level of nesting. If the importing file is NOT a package (an ordinary
     `.py` module, e.g. `app.routes.webhook`), a `level=1` relative import resolves relative to its
     PARENT package (`app.routes`) exactly as CPython's own relative-import semantics work. Append
     `node.module` (if present) to the resolved base with a dot. Concretely: implement this as
     `base = own_module_parts[:-node.level] if not is_package else own_module_parts[:len(own_module_parts) - (node.level - 1)]` in whatever form is clearest, and cover BOTH the package and
     non-package relative-import cases explicitly in step 5's synthetic fixture.
   - **Regardless of which branch computed it,** once the target module string is known: if it
     differs from the importing file's own module name (i.e. this is genuinely a cross-module
     reference, not a same-module `from . import x` circling back to itself -- which cannot
     actually happen for `ImportFrom` but is a defensive check), flag every name in `node.names`
     whose `alias.name` starts with `_` and does not start with `__` (dunder). This ONE flagging
     step applies uniformly whether the target module was determined via the absolute or the
     resolved-relative path -- there is no separate, weaker check for the relative case.

4. For every `ast.Attribute` node whose `.attr` starts with `_` and does not start with `__`: walk
   the attribute's `.value` chain to determine if it resolves to a locally-bound name that was
   imported via a module-level `ast.Import` (`import X` or `import X as Y`) elsewhere in the SAME
   file, where `X` (the imported module) is NOT the file's own module and is NOT a package the
   file's own module belongs to (i.e. genuinely a different top-level component under one of
   `SCAN_ROOTS`). If so, flag it as a `module._private` attribute-access violation. This is the
   pattern that catches `scripts/*.py`'s `repo._conn_ctx(...)` (via `import app.db.repo as repo`)
   while NOT flagging same-module `self._helper()`-style attribute access or unrelated attribute
   chains (e.g. `request.headers._some_internal` on a third-party object) -- scope the attribute
   check specifically to attributes of names bound by a LOCAL `ast.Import` of a first-party module
   under `app`/`eval`/`scripts`, not every `ast.Attribute` node in the file (which would produce
   excessive false positives on arbitrary object attribute access).

5. **Facade-boundary exemption (resolves the `_conn_ctx`/`_scrub`/etc. tension from finding #1

| Boundary | Description |
|----------|--------------|
| CI-enforced module boundary (app/, eval/, scripts/) | This guard is the automated backstop preventing a future contributor from silently reintroducing a cross-module private-name coupling (via either import form, including a relative import) that could hide a monkeypatch-seam break or an unintended internals dependency |
