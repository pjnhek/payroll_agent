| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-8-08 | Tampering / Injection | `run_detail.html`'s new `{{ run.error_detail }}` render | mitigate | Jinja2's default autoescape stays ON (no `| safe` filter added); `error_detail` is treated as untrusted-origin text even after scrubbing, consistent with how `error_reason` is already rendered on the adjacent line |
| T-8-09 | Tampering (data integrity) | live `payroll_runs.status` CHECK swap | mitigate | Blocking human checkpoint runs the pre-migration `SELECT count(*) WHERE status = 'needs_clarification'` guard and requires a confirmed `0` before the ADD CONSTRAINT step; the DROP+ADD is one `DO $$ ... END $$;` transaction so a failed ADD rolls back the DROP too (worst case: no-op, never half-migrated) |
| T-8-10 | Denial of Service | `get_pool()` double-checked locking (WR-02) | mitigate | `threading.Lock()` around the check-then-create closes the race where FastAPI's threadpool executor could construct two `ConnectionPool` instances (one leaked) under concurrent first-callers, even in Render's single-process deploy |
| T-8-11 | Information Disclosure (silent gap) | `RUN_COLS` missing a newly-added column | mitigate | `RUN_COLS` is a single explicit constant every `load_run` caller shares; this plan adds `error_detail` to it in the SAME task that wires the template, and the new integration test asserts on `fake_conn.all_sql()` that the column is actually present in the SQL text -- not just that template text matches a string |
