Review fixes folded in (codex review — this plan absorbs the HIGH finding and 3 others):

**HIGH #1 — roster-scope gap on the main pipeline path (re-verified against live orchestrator.py this session, confirmed exactly as codex described):** the ORIGINAL plan's wiring assumed `run_pipeline`'s own `except` block could pass `roster=roster` to `record_run_error`. It cannot: `run_pipeline` (orchestrator.py:173-188) wraps `_run(run_id, llm=llm)` in a try/except, but `roster = repo.load_roster_for_business(...)` is bound at line 199 — INSIDE `_run`, a completely different function scope, invisible to `run_pipeline`'s except block. As originally planned, EVERY first-run failure (the most common production failure path — extraction/validation errors, LLM parse failures, etc.) would silently fall back to `roster=None`, meaning only the email regex runs and any roster employee name/alias embedded in `str(exc)` (e.g. inside a Pydantic ValidationError echoing LLM output) leaks into `error_detail` — directly failing OPS2-01's "excludes PII" success criterion on the path that matters most. Fix (this plan): restructure so `_run` owns its OWN try/except internally — `roster = None` is initialized as the first statement inside `_run`, reassigned immediately after the real `load_roster_for_business` call succeeds, and `_run`'s own except block (which now has `roster` in scope, whatever its value) calls `repo.record_run_error(run_id, reason, detail_exc=exc, stage="pipeline", roster=roster)` directly. `run_pipeline` becomes a thin, non-raising delegator that just calls `_run(run_id, llm=llm)` — it no longer needs its own try/except because `_run` never lets an exception escape. This is a strictly-additive internal restructure: `_run`'s public contract (still called only by `run_pipeline`), `run_pipeline`'s own signature/contract (still never raises, per `app/main.py:461-466`'s explicit reliance on that guarantee), and every existing test that calls `run_pipeline(run_id)` and asserts on the resulting `error_reason`/status are unaffected — only the INTERNAL scrub fidelity improves. `resume_pipeline` does NOT have this gap (its own `roster` load already happens inside the same try block its except clause guards) — it only needs the simpler `roster = None` top-of-try initialization already planned, to guard the narrower UnboundLocalError window where the exception fires before the roster load line.

**MEDIUM #3 — deploy-order gap:** code that writes `error_detail` (this plan's Task 1) could reach a LIVE database that has not yet had 08-01's schema applied (Task 2, the blocking checkpoint) — `record_run_error`'s `UPDATE ... SET error_detail = %s ...` would then fail with an undefined-column error INSTEAD of recording the original error, which is the exact failure mode D-8-01b exists to prevent (a diagnostics feature must never break the error path it observes). Fix: Task 2's checkpoint instructions now explicitly state the schema apply is a DEPLOYMENT-ORDER GATE — the live schema MUST be migrated before/simultaneously with deploying Task 1's code to any environment sharing that live database, not after. This plan also documents (in Task 2's own text) that `record_run_error`'s new `error_detail`-writing UPDATE is NOT internally tolerant of a missing column (an undefined-column Postgres error would propagate out of the UPDATE and — because `record_run_error` itself has no additional try/except around the UPDATE beyond what already exists — surface as a genuine, unhandled exception in whatever caller invoked it); the mitigation is procedural (deploy-order sequencing enforced at the checkpoint), not a runtime fallback, because a silent runtime fallback that swallows a real schema-mismatch error would itself hide a real production misconfiguration. See Task 2 for the exact sequencing instruction.

**LOW #7 — `InMemoryRepo.load_all_runs()` staleness:** `tests/conftest.py`'s `InMemoryRepo.load_all_runs()` does not compute the new `summary_gate_reason`/`employee_count` aliases 08-02 added to the REAL `repo.load_all_runs()`, so any route-level test that swaps in `InMemoryRepo` would silently stop exercising the real list-summary contract (the fake would return dicts missing the two new keys, and `runs_list.html`'s Jinja `{% if run.summary_gate_reason %}` would just always fall through — no test failure, but zero coverage of the real behavior through that path). Fix: Task 1 updates `InMemoryRepo.load_all_runs()` to compute the same two aliases from its in-memory run records.

**LOW #9 — no deterministic way to produce a live ERROR row for the dashboard checkpoint verification:** Task 2's `<how-to-verify>` now gives an exact, deterministic SQL snippet to manually set an existing run's `status`/`error_detail` via `UPDATE`, view it, then revert — instead of "wait for a real failure" or "trigger one somehow."
