Implement the OPS2-01 diagnostics core and the OPS2-02 `load_all_runs` projection fix entirely inside `app/db/repo.py` — the single funnel every error boundary and the runs-list route already goes through. Add a centralized, fail-open, scrub-before-truncate PII redaction helper and wire it into `record_run_error`'s existing transaction (no new query, no new transaction — one added `SET` clause on the existing `UPDATE`). Purpose: this is the data-access layer half of Phase 8 — independent of the schema DDL (08-01, parallel) and independent of wiring the 3 call sites / templates (08-03, depends on this). All new logic is provable offline via `FakeConnection`; no live DB needed.

Review fixes folded in (codex review):
- **MEDIUM #2 (JSONB scalar edge case):** `COALESCE(jsonb_array_length(pr.extracted_data->'employees'), 0)` is NULL-safe but still ERRORS if `extracted_data->'employees'` is a JSON scalar or a JSON `null` literal (non-array) rather than SQL NULL — `jsonb_array_length` on a non-array JSON value raises a Postgres error, not a NULL. Fix: `CASE WHEN jsonb_typeof(pr.extracted_data->'employees') = 'array' THEN jsonb_array_length(pr.extracted_data->'employees') ELSE 0 END AS employee_count`. A corrupt/legacy row must not take down the entire runs list.
- **MEDIUM #4 / R2-1 (scrubber case/Unicode sensitivity — offset-safe design, replaces the Round-2-flagged "search normalized, slice original with normalized offsets" approach):** the original design's exact `str.replace` is case-sensitive and Unicode-normalization-sensitive, so a roster name in different case (e.g. email client capitalizing "Maria Gonzalez" vs stored "maria gonzalez") or a different Unicode normalization form (combining vs precomposed accents, e.g. "José" as `\u00e9` vs `e\u0301`) survives unredacted — a real project concern already established by MONEY-02. Codex Round 2 flagged the original "normalize the whole message, search it, then slice the ORIGINAL string using the normalized match's offsets" approach as NOT offset-safe: NFKC/casefold can change string length (e.g. a decomposed 2-codepoint "é" normalizes to a 1-codepoint precomposed form, or a ligature expands), so an offset computed against the normalized string does not point to the same span in the original string — this can leave a stray combining mark next to `[REDACTED]` or redact the wrong adjacent characters. **Fix (offset-preserving, one coherent design covering R2-1 + R2-3 together):** build ONE compiled `re.Pattern` PER CANDIDATE NAME directly from the ORIGINAL (non-normalized) name/alias string — never from a normalized copy of the message — so all matching happens directly against the ORIGINAL message string and every match span the regex engine reports is already a valid original-string offset (no translation step, no drift). Each candidate pattern is built by escaping the name with `re.escape` per character and, for any base Latin letter that has a common precomposed-accented counterpart used by the project's roster (transcribed as an explicit small map, not `unicodedata`-computed at match time), replacing that literal character in the pattern with a character class alternation covering the precomposed form, `base_letter + combining_mark_range` (so a decomposed "e" + U+0301 also matches where the roster stores "é"), AND the bare unaccented base letter itself (checker-fixed: a name with no covering alias, e.g. "Ana Núñez", must still be caught when it arrives fully de-accented, e.g. "ANA NUNEZ" — a real shape LLM extraction and all-caps email rendering produce by stripping diacritics rather than decomposing them); the whole per-name pattern is anchored with mark-aware lookarounds — leading `(?<![\w\u0300-\u036f])` and trailing `(?![\w\u0300-\u036f])` — instead of bare `\b...\b` (R3-1: `\b` alone is not mark-aware for NFD/decomposed text — a candidate ending in an accented character can match its bare-base alternative with `\b` succeeding just before the trailing combining mark, stranding e.g. `[REDACTED]` + a stray U+0301; the lookarounds reject a following combining mark, not just a following word character, closing that gap while remaining strictly stronger than `\b` for plain ASCII, R2-3) and compiled with `re.IGNORECASE`. Candidates are still sorted longest-first (by character count) before compiling/applying, so a short alias contained inside a longer name/alias never independently matches and fragments an already-redacted span, and each pattern's `.sub(_REDACTED, message)` call operates directly on the current (partially redacted) ORIGINAL message — no normalize-search-then-slice-original two-step, eliminating the offset-drift class of bug entirely.
- **LOW #8 (conn positional-compatibility):** keep `conn` positional-compatible in the new `record_run_error` signature — matches every existing call site's convention (`record_run_error(run_id, reason, conn=fake_conn)` used positionally today in some call/test sites) — the 3 NEW params (`detail_exc`, `stage`, `roster`) go keyword-only, placed AFTER `conn` in the signature, not before it.
