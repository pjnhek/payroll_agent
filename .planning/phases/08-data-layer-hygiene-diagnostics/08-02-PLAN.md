<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Exception message -> `error_detail` DB column -> dashboard render | The exception message can originate from LLM output, submitted client data, or raw email content — all untrusted from a PII standpoint by the time it reaches `str(exc)`; this is the boundary the scrubber sits on |
| `load_all_runs` JSONB columns -> dashboard list view | `decision`/`extracted_data` may grow new fields over time; the explicit projection is the boundary preventing silent schema-creep leakage to the list view |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-8-01 | Information Disclosure | `_build_error_detail` / `record_run_error` | mitigate | Deterministic scrub-then-truncate (D-8-01): email regex + in-memory roster full_name/known_aliases substring redaction runs on the FULL message before any truncation, closing the audit HIGH-05 gap; unit tests (D-8-04, D-8-04a) assert both PII exclusion and the boundary-straddle ordering case |
| T-8-02 | Denial of Service (soft — silently-hung run) | `_build_error_detail` internal failure | mitigate | Fail-open design (D-8-01b): any exception inside `_build_error_detail` is caught internally and returns `None`; the existing `error_reason` + `set_status(ERROR)` write path in `record_run_error` proceeds regardless, proven by the D-8-04b unit test |
| T-8-06 | Information Disclosure | `_scrub` querying the DB for roster data during a DB-outage error path | mitigate | `_scrub` takes `roster` as a plain in-memory parameter only — no DB call, no import of `repo.load_roster_for_business` inside `_scrub`/`_build_error_detail`; enforced by code review (no such call exists) and the D-8-04b "no roster" test asserting no new SQL appears |
| T-8-07 | Information Disclosure (schema creep) | `load_all_runs` | mitigate | Explicit scalar column list + 2 computed aliases (D-8-07) replaces `SELECT pr.*`; a new `payroll_runs` column can no longer silently reach the dashboard list view without an explicit, reviewed SQL edit |
</threat_model>
