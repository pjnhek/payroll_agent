<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| schema.sql -> live Postgres (Supabase) | DDL applied by the idempotent bootstrap; this plan only edits the DDL source file — no live apply happens here (deferred to the 08-03 blocking checkpoint) |
| Python enum <-> SQL CHECK | Dual-sourced value set; a drift here (one side updated, the other not) silently permits or rejects the wrong status values in a way CI would not catch without the guard test |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-8-03 | Tampering | schema.sql status CHECK swap (DO-block) | mitigate | Guard is `contype='c' AND conrelid='payroll_runs'::regclass AND conname LIKE '%status%'` (narrowed lookup, mirrors the proven email_messages.purpose pattern) so the DROP cannot accidentally target an unrelated constraint; the DROP+ADD is inside one `DO $$ ... END $$;` block so a failed ADD rolls back the DROP too (no half-migrated table) |
| T-8-04 | Information Disclosure | new indexes on `payroll_runs(status)` / `email_messages(...)` | accept | Indexes on status/direction/send_state columns carry no PII themselves (enum-like values); no new information-disclosure surface vs. the existing table data |
| T-8-05 | Denial of Service (schema drift) | dual-sourced RunStatus enum vs. schema.sql CHECK | mitigate | `tests/test_status_drift.py`'s `TestEnumCheckDrift` parametrized test re-derives the Python value set live via `{member.value for member in RunStatus}` (not hardcoded) and asserts set-equality against the parsed SQL CHECK — a future one-sided edit fails CI immediately |
</threat_model>
