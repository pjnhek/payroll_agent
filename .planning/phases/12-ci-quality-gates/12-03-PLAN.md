<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Any branch push → GitHub Actions runner | Untrusted/arbitrary branch content triggers `ci.yml` on ALL branches per D-07 — a fork PR or a pushed branch runs CI with the default `GITHUB_TOKEN` |
| ci.yml → external actions (`actions/checkout@v4`, `astral-sh/setup-uv@v5`) | Third-party GitHub Action supply chain |
| README badge URL → GitHub's badge.svg endpoint | Public, unauthenticated image fetch by anyone viewing the README |
| Dirty working tree → git commit | Unrelated pre-existing dirty/staged files (e.g. mid-flight `.planning` deletions from another session) could be accidentally swept into this plan's commits if staged carelessly |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-06 | Elevation of Privilege | ci.yml triggers on every branch push with default token permissions | mitigate | Neither `lint` nor `test` job requests `permissions: write` (both omit any `permissions:` block, defaulting to the repo's configured minimal token scope); the workflow never checks out with write access, never pushes, never touches secrets — matching the least-privilege pattern already established by `eval.yml`'s `record` job explicitly scoping `contents: write` only when it needs to push |
| T-12-07 | Information Disclosure | Lint/test step output could leak secrets in logs | accept | Neither job sets any `env:` referencing `secrets.*` (D-11 — bare `uv run pytest -q` with no DB service, no live-LLM keys); there is nothing secret-bearing in this workflow's env to leak |
| T-12-SC | Tampering | Supply-chain: `actions/checkout@v4` and `astral-sh/setup-uv@v5` are pinned to major-version tags, not commit SHAs | accept | Matches the existing house convention across all 4 current workflows (none of them SHA-pin); introducing SHA-pinning here alone would create an inconsistent convention within the same repo. Both actions are widely-used, GitHub-verified (`actions/checkout`) or maintained-by-Astral (`astral-sh/setup-uv`) publishers; no package-manager install is involved (this is GitHub Actions marketplace, not npm/pip/cargo), so the Package Legitimacy Gate protocol does not apply |
| T-12-08 | Tampering | Badge URL could be spoofed/mistyped to point at a different repo | mitigate | Badge URL uses the confirmed `pjnhek/payroll_agent` slug read directly from `git remote -v` during planning, not guessed or templated |
| T-12-13 | Tampering | Dirty working tree causing unrelated files to be swept into this plan's commits | mitigate | Both tasks preflight with `git status --short` before editing, and stage only this task's own explicit paths (`git add .github/workflows/ci.yml`, `git add README.md .github/workflows/ci.yml`) — never `git add -A` / `git commit -a` |
