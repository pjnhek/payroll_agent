## Phase Goal

**As an** operator, **I want to** approve payroll runs through a race-safe gate, **so that** duplicate approvals are structurally impossible and a failed delivery never silently strands a run.

<deviation_note>
## Finding #4 Disposition: Alias Binding — Single-Token-Only Restriction (option a)

Codex finding #4: alias binding at resume is unsolved for multiple unresolved names. The system re-extracts from the REPLY (which carries the CORRECTED name, not the original token), making the original-token→employee mapping unreliable when multiple names were unresolved.

CHOSEN RESOLUTION: Option (a) — single-token-only restriction.

Rule: alias_candidates is written in _clarify ONLY when decision.unresolved_names contains EXACTLY ONE token AND that token passes the non-collision check (_safe_to_learn_alias against the current roster without appending). If unresolved_names has 2+ tokens, alias_candidates is NOT written for this run (no alias learning for this approval; the run still completes normally). If the single token is ambiguous (like "D. Reyes"), it is also excluded.

Rationale: The single-token case has a deterministic binding: one unresolved token → the operator approves the reply → resume resolves to exactly one employee → that employee is unambiguously the target. The "David Reyez" → David Reyes demo works cleanly in this case. Multi-token cases are deferred to a future phase (where a UI-confirmed mapping per token can be built into the approval form). This keeps the demo's third beat credible without introducing a binding algorithm that cannot be made reliably testable.

This restriction is ON THE RECORD here. CONTEXT.md's D-04 understanding is NOT silently reversed: D-04 still mandates capture at clarify-emit time and keying by token. The single-token restriction is a scoping decision within D-04's framework.

## Finding #5 Disposition: Capture-Time Collider Exclusion (per D-04 locked constraint)

D-04 and D-01b both state: ambiguous/colliding tokens must be EXCLUDED AT CAPTURE TIME (not merely filtered at write time). The original Plan 07 captured `{name: None for name in decision.unresolved_names}` unconditionally — including colliding tokens. This violates the D-04 locked constraint.

FIX: At capture time in _clarify, run _safe_to_learn_alias (without appending — just check collision) for the candidate token. Only write alias_candidates if the token is non-colliding AND there is exactly one unresolved name (per finding #4 single-token restriction above). A colliding token is logged and skipped at capture time, not stored with an eligible=false flag.
</deviation_note>
