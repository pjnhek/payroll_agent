## Implementation Decisions

### Area 1 — Alias WRITE-side learning loop (SCOPE: IN)

- **D-01 (IN — operator-gated alias write):** Build the minimal write path per `.planning/backlog.md` → "Learn aliases from confirmed clarifications." When the operator **approves** a run whose original submission contained an unresolved name that the client's reply later resolved to employee X, **persist that original shorthand into `X.known_aliases`** (idempotent — do not double-add). A subsequent run with the same nickname then resolves at the deterministic stored-alias fast-path with **no clarification**. This completes the headline narrative ("clarifies *once, then learns*") and is the demo's third beat.
- **D-01b (write-side collision exclusion — CORRECTNESS-CRITICAL, from review #1):** Idempotency is necessary but NOT sufficient. **Before appending a token to `X.known_aliases`, verify the token would not match MORE THAN ONE roster employee under the SAME collision rule `reconcile_names` uses** (`app/pipeline/reconcile_names.py`). If the token is ambiguous across the roster, **DO NOT learn it** — log it and skip the write (the run still completes normally; it just doesn't earn an alias). **Why this is load-bearing, with the seed's own trap:** David Reyes (e0000003) and Daniel Reyes (e0000007) both already carry `"D. Reyes"`. Run the third beat with that token — original "D. Reyes" → collision → clarify → client says "David" → operator approves → a naive D-01 appends `"D. Reyes"` to *David*. Now the alias READ resolves a future "D. Reyes" to David at the fast-path with **NO clarification and NO collision detected** (only David got the alias, so the read matches exactly one employee) — even when the client meant Daniel. That is a **silent misroute of a money-moving decision** — the exact failure the architecture exists to prevent, and it surfaces on camera if anyone demos with the Reyes pair. Read-side collision safety does not save you because the read only sees the one employee who learned it. The guard MUST be on the write.
- **D-02 (write fires at the operator-approval gate):** The alias persists **only** when the human operator approves the resolved run — the one moment a human has verified the entire resolved run is correct. Reuses the single existing gate; **no auto-write on the clarification reply** (a reply might *correct* the name, not confirm it — auto-learning a wrong alias would silently poison future resolution, violating "never guesses on a money-moving decision"). This is the backlog's "safer, human-in-the-loop" recommendation.
- **D-03 (eval stays seed-bound — no Phase-4 corpus change):** The alias write lands in the **live DB only**; the Phase 4 eval reads static `app/db/seed.py` values, so it is insulated by construction. The **demo proves the loop**; no change to the existing eval corpus. (Proving the loop *in* the eval via a seedable before/after fixture is noted as a deferred "if-time" idea, not in scope.)
- **D-04 (the mapping the approval gate needs — RESEARCH ITEM; storage spot open, but timing + keying are LOCKED CONSTRAINTS, sharpened by review #3):** The approval gate must know **which original shorthand resolved to which employee**. The storage *spot* is the planner's call (candidate: `payroll_runs.reconciliation` JSONB, `app/db/schema.sql:87` — already the home for per-run resolution facts). But two properties are **load-bearing constraints, not open**:
  - **TIMING — the mapping MUST be captured at clarify-EMIT time, not reconstructed at approval time.** The original unresolved token ("David Reyez") lives in the **original** email; `resume_pipeline` re-extracts from the **reply**, which carries the *corrected* name — so by approval time the original token is GONE unless persisted earlier. The ONLY moment the system holds the original token *and* knows it's unresolved is when `_clarify` emits the clarification (`orchestrator.py:200`). Persist the unresolved tokens THERE; bind each to its resolved employee at resume (when the reply resolves it); read at approval. A planner who tries to reconstruct the pairing from resume/approval data will find it **unrecoverable** — flag this explicitly so that dead end isn't attempted.
  - **KEYING — it is a dict `{original_token: resolved_employee_id}`, NOT a single pair.** One email can carry multiple unresolved names resolved in one reply.
  - **INTERACTION with D-01b — a colliding/ambiguous token is NOT a learning candidate.** Don't even store it as one (or store it flagged-ineligible). The write-side collision check (D-01b) is the final gate, but excluding ambiguous tokens at capture time keeps the mapping honest.

- **D-15 (drop-order / sequencing lever — recommended flip, from review sequencing flag):** The phase currently marks **INGEST-05's re-trigger route as the drop-if-tight candidate** while treating the **alias loop (D-01..D-04, D-01b) as the essential third beat**. Review argues — and this discussion accepts — that the priorities may be **inverted for the "visibly works end-to-end" bar (#1):**
  - **Re-trigger is load-bearing** for the "nothing silently hangs" reliability story (D-13b shows the claim helper *introduces* strandable states, so recovery is not optional polish).
  - **The alias loop is the most schedule-risky item** in the phase (unresolved D-04 storage + the D-01b collision-exclusion correctness work) **and the least essential for #1** — beats 1 (clean→approve→deliver) and 2 (unknown shorthand→clarify with suggestion) already prove the thesis on camera. The alias loop is the *impressive* beat, not the *necessary* one.
  - **Therefore, if something must drop under time pressure, prefer dropping/deferring the alias WRITE loop (D-01..D-04) over the re-trigger route.** This is a **recommendation to the planner**, NOT a reversal of the user's explicit "alias loop is IN" decision — the user chose it IN with full knowledge it's the headline beat. Surface this trade-off at planning time if the schedule tightens; do not silently re-order.

### Area 2 — Over-40-no-OT validation rule (SCOPE: folded into Phase 5)

- **D-05 (fold in as a named deliverable with its own success criterion + demo beat):** Add the per-**workweek** silent-underpay guard to `app/pipeline/validate.py` (pure function, calc untouched — Phase 3 D-03 "trust the submitted split, never auto-derive" is unchanged; the catch is purely upstream in validation, emitting a `ValidationIssue` into the **already-built** clarification gate). Spec (from `.planning/backlog.md` → "Over-40-no-OT validation rule"):
  - **Weekly** (`pay_periods_per_year=52`, 1 workweek): `hours_regular > 40` with no `hours_overtime` field → emit `ValidationIssue` → `decide` gates to `request_clarification`. **This is COMPLETE detection** — the period *is* one workweek, so >40 catches every over-40-no-OT case. This is the demo beat and the only frequency with full coverage.
  - **Biweekly** (`=26`, 2 workweeks): `hours_regular > 80` with no OT → clarify. **PARTIAL detection, honestly labeled (review #5).** `>80` is a *sufficient* trigger (pigeonhole: >80 over 2 weeks guarantees one week passed 40) but NOT *complete*: an unbalanced split like 50+30=80 or 45+20=65 has a week over 40 yet a period total that never trips the threshold — it **silently underpays and does not flag**. Because the period total cannot reveal per-week distribution, full biweekly detection would require a per-workweek hour breakdown (out of scope). So biweekly is documented as **partial detection**, NOT parity with weekly. Claiming biweekly is "caught" while these holes exist is exactly the overclaim a "never silently underpay" thesis must avoid.
  - **Semi-monthly** (`=24`) / **Monthly** (`=12`): period boundaries cut across workweeks → period total cannot reveal OT → **NO flag, documented README limitation** (this is where the "client must state OT explicitly" line is correct — the undetectable slice).
  - **EDGE — explicit `hours_overtime = 0` with `hours_regular > 40`:** the rule keys on the OT field being **absent**, so an explicit `0` (e.g. weekly 45 reg + 0 OT) currently slips through under Phase 3 D-03's "trust the submitted split" — even though it's the most facially-suspicious case. **Make this an explicit, documented decision, not an accidental gap:** either treat explicit-`0`-with-reg-over-threshold the same as absent (recommended — flag it; the client almost certainly didn't mean "45 straight, zero premium"), or document it as a deliberate "we trust an explicit zero" choice. The planner picks, but it must be on the record.
  - Testable against already-seeded employees (the seed covers weekly + biweekly). Demo beat (weekly, the complete-coverage frequency): "Bob worked 45 hours" with no OT field → "is that 40 regular + 5 overtime, or 45 straight?" — puts the never-wrong-on-money thesis on camera.

### Area 3 — Operator UI: shape & polish

- **D-06 (interaction mechanism — plain server-rendered forms + POST-redirect-GET):** Approve / reject / send-test / re-trigger are `<form method=post>`; the handler does the work and **303-redirects** back to the run detail or runs list. `python-multipart` is in deps exactly for this. No JS state machine, no partial-render edge cases — lowest risk, demos reliably on a cold dyno. Full-page reloads are imperceptible at ~4 pages with one operator action per run. (The existing JSON `/runs/{id}/approve|reject` endpoints may remain as an API, but the UI does not depend on hand-written fetch glue.)
- **D-06b (sync-vs-background pipeline execution — DECIDE EXPLICITLY for demo latency, from review #4):** The plain-form handlers run their work **before** the 303 fires, and several handlers trigger **live LLM calls**: approve → `compose_confirmation` (draft tier) + PDF gen + send; "Send test email" / webhook → `extract` (extraction tier). On a cold Render free dyno these can be multi-second-to-10s+ hangs **on camera, inside a 60–90s budget**. This is the single most likely thing to look bad on video (priorities #1/#2). **The planner MUST verify how `run_pipeline` is invoked today** (it is a FastAPI `BackgroundTask` from the webhook — `main.py:103` — so the inbound webhook is already backgrounded; the **approve route `_operator_transition` is fully synchronous in the request** and is where the new delivery LLM work lands) **and choose deliberately per action:**
  - **Approve/delivery:** keep synchronous BUT bound the wall-clock (see D-10's hard draft timeout). A bounded approve→redirect is fine; an unbounded live draft call is the cliff.
  - **Send-test/webhook extraction:** either background it (and let the no-JS UI show states advance on manual refresh — a *decent, deliberate* demo beat: "watch it move through extracting → computed → awaiting_approval") or accept a bounded synchronous wait. Do NOT leave this to silent inheritance from `main.py`.
- **D-07 (three-column run-detail rendering — literal 3-col CSS grid):** raw cleaned email as a **monospace `<pre>`** (shows exactly what the client sent, cruft and all — the point of the honest gate) | `extracted_data` as a **readable per-employee table** (employee → the 5 hours fields + any 401k override) | computed paystubs as a **table** (gross / FICA / federal / net per employee). The decision object's reasons (`gate_reasons` / `unresolved_names` / `missing_fields`, or "process") render in a **prominent banner** so *why this run is in this state* is unmissable. **NO responsive** (single-screen demo; PROJECT.md makes responsive a non-goal). Optimize for the operator scanning the raw email column against the extracted column.
- **D-08 (runs list — DASH-01):** reverse-chronological **table**: created-at, business, status badge, a one-line summary (e.g. # employees or the gate reason), action link to detail. **Badge colored by status class:** pending-action (`awaiting_approval`) **emphasized**, in-flight (extracting/computing/awaiting_reply) neutral, terminal-good (sent/reconciled) green, terminal-bad (rejected/error) red. The runs list is the operator's triage queue — "what needs me" at a glance.
- **D-09a (eval view — DASH-04 — embed the committed SVG, render the rest from `summary.json`):** Serve Phase 4's committed `eval/chart.svg` **as-is** (it is *the* recruiter-visible proof artifact — do NOT rebuild a second chart renderer in the dashboard; that would violate the DRY seam). Read `eval/summary.json` for the **headline metrics** (false-process count + the three core metrics) and the **per-fixture drill-in** (each fixture's raw email beside expected vs actual extraction/decision — `summary.json` already carries per-fixture details, stored in Phase 4 D-08 precisely so the dashboard could read them). Hermetic: no live eval, no DB on this path. The drill-in table is the only genuinely new work.

### Area 4 — Delivery: confirmation email + PDF + concurrency

- **D-10 (confirmation email — `compose_confirmation` mirroring `compose_clarification`):** New composer in the same DRY shape as `app/pipeline/compose_email.py:88` — cheap `draft` tier for warmth + a **deterministic template floor** (run total, per-employee net, pay date) so a draft-tier failure **never strands an already-approved send** (even more critical post-approval than on the clarify path). The PDFs carry the authoritative numbers; the email body is the cover note. Satisfies HITL-02's "LLM-drafted" while keeping reliability via the floor; keeps both composers structurally identical.
- **D-10b (hard draft timeout — bound the approve-click wall-clock, from review #4):** The deterministic floor (D-10) protects the *content* of a draft failure but NOT the *latency* — a slow-but-successful draft still blocks the 303. Give the confirmation draft call a **hard short timeout (~2–3s)** that falls through to the template floor on expiry, so worst-case approve→redirect is bounded regardless of dyno state. The floor is already built; this just caps wall-clock. (Same reasoning applies to the clarify draft, but approve is the on-camera climax, so it's the priority.)
- **D-11 (paystub PDF — one PDF per employee, on-demand):** reportlab + BytesIO, in-memory (HITL-03 locked). **One PDF per employee** (how real paystubs are issued — the builder's accounting-days authenticity), each containing the `PaystubLineItem` fields Phase 3 computes: gross (with OT / leave breakdown), pre-tax 401k, FICA (SS + Medicare), federal withholding, **net**. Attached to the confirmation email **and** downloadable per-line-item from the run-detail page. **The PDF generator is a PURE function of `PaystubLineItem` + employee + run metadata** (mirrors the pure-stage seam) so it is testable without a DB and reused by both the email-attach path and the on-demand download route.
- **D-12 (atomic status-claim helper — FOUND-04 — `claim_status()` via conditional `UPDATE … RETURNING`):** Add `claim_status(run_id, expected, new) -> bool` to `app/db/repo.py`: `UPDATE payroll_runs SET status=%s, updated_at=now() WHERE id=%s AND status=%s RETURNING id`. A claim succeeds **only if the row was still in `expected`**; a losing concurrent caller gets no row → logs a late/duplicate and **drops cleanly** (does not re-run). This becomes the **second sanctioned status writer**: `set_status` stays for *unguarded* forward transitions inside a run that already holds the path (e.g. computing → computed → awaiting_approval); `claim_status` is the **atomic guarded** transition for every contended gate — **(a) approve/reject** (claim `awaiting_approval → approved/rejected`), **(b) resume** (claim `awaiting_reply → extracting` — closes the `resume_pipeline` non-atomic load-then-set documented at `orchestrator.py:115-116`), **(c) re-trigger** (claim `error → received`/extracting — AND `approved → received` per D-13b, so a delivery that died after the approve claim is recoverable), and **(d) the initial run claim**. **MUST update the "ONE AND ONLY writer of payroll_runs.status" invariant doc** (`repo.py` set_status docstring) to "two writers: `set_status` (unguarded) + `claim_status` (atomic, guarded)" so the codebase's own rule stays honest. A single-row `UPDATE … WHERE status=? RETURNING` is atomic in Postgres without an explicit transaction and composes with the existing optional-`conn=` pattern. **FOUND-04's literal `SELECT … FOR UPDATE` wording is read as satisfied-in-spirit** by the atomic conditional UPDATE (both structurally prevent double-approval) — see Specific Ideas.
- **D-13 (idempotent send + error path / re-trigger — CLAR-04 + INGEST-05, BUILD both):** Idempotency is two layers: (1) the **`claim_status` claim** means only one caller wins `approved → sent`, so the send fires once; (2) **before sending, check for an existing outbound confirmation row** for the run (reuse the existing `repo.get_outbound_message_id` pattern, `repo.py:467`) — if one exists, **skip the send and just advance**. Re-trigger re-runs from the **start of the run**; the claim + already-sent check make a re-trigger structurally unable to duplicate a confirmation. **Error path:** the orchestrator already persists `ERROR` + `error_reason` on any stage failure (the D-A1-03 error-wrap in `run_pipeline`/`resume_pipeline`) — the dashboard surfaces it (red badge + the reason) with a **re-trigger button**. "Nothing silently hangs" is a real recruiter-facing reliability beat, nearly free once the claim helper exists. **If the phase runs long, INGEST-05's re-trigger route is the clean drop** (per its drop-if-tight flag), keeping the CLAR-04 send guard — but see the sequencing note: review #2 argues re-trigger is load-bearing for the reliability story, so weigh the drop against the alias loop (D-15).
- **D-13b (claimed-but-not-terminal states must NEVER strand — CORRECTNESS, from review #2):** The atomic claim closes double-action but opens a NEW hole: **after a winning claim, if the work dies before reaching a terminal/pause state, the run sits in an in-between status with NO recovery action.** The existing `run_pipeline`/`resume_pipeline` paths are wrapped in the D-A1-03 error-wrap → `record_run_error` → `ERROR`, but the **post-approval delivery path is NEW orchestrator work that currently has NO such wrap** (the approve handler lives in `main.py:_operator_transition`, a synchronous route outside any orchestrator error boundary). Concretely: approve claims `awaiting_approval → approved`, then compose→PDF→send runs; if it throws or the dyno dies before `sent`, the run is stuck in `approved` — the approve button is gone (status ≠ `awaiting_approval`) and the re-trigger button is absent (only shows on `ERROR`). **No operator action recovers it. MANDATES:**
  1. **Wrap the new delivery path in the same ERROR-converting boundary** — any failure between the approve claim and `sent` → `approved → error` + `error_reason` (mirror the D-A1-03 pattern; do not leave the delivery work bare in the route).
  2. **Re-trigger must be able to claim from `approved` (and `error`), not just `error`** — re-running from start is safe because of the already-sent check (D-13/D-13c). Every claimed non-terminal state (`approved`, `extracting` from a dead resume, `received` from a dead re-trigger) must have either a forward driver or a recovery button. "Nothing silently hangs" is only TRUE if that invariant holds across the new states the claim helper introduces.
- **D-13c (intent-row-BEFORE-send for crash-safe idempotency — from review #2 rider):** The already-sent guard reuses `get_outbound_message_id`, which reads a row the stub `send_outbound` writes **after** minting the Message-ID (`gateway.py:58→59`). That's **record-after-send ordering**. It's harmless today (the Phase-5 gateway is a stub — mint+record are atomic, no real network), but it **bakes in the ordering that becomes a real double-send the moment Phase 6 wires a live provider**: crash between the provider call and the row insert → re-trigger finds no row → sends again. **Fix it now while it's cheap:** write a sentinel/intent row (or set an "attempting-send" marker) **before** the provider call, so the already-sent guard is crash-safe by construction. Note this explicitly as a Phase-6-forward-looking invariant the Phase-5 stub should already honor, so the live wiring is a no-op swap, not a retrofit.

## Specific Ideas

- **The alias-write loop is the demo's *third beat* — "clarifies once, then learns."** Beat 1: clean run → approve → confirmation+PDFs. Beat 2: unknown shorthand "David Reyez" → gate clarifies (names the suggested employee). **Beat 3: operator approves the resolved run → the shorthand is learned → a re-run with the same shorthand resolves with NO clarification.** This is the narrative payoff the deterministic-resolution + single-human-gate architecture was built to enable. **CAVEAT — only ever demo/learn an UNAMBIGUOUS token (D-01b):** the seed's "D. Reyes" is shared by David AND Daniel Reyes, so learning it would silently misroute future runs — the demo's learnable token (e.g. "David Reyez" → David) must be one that resolves to exactly one employee. The write-side collision check makes this safe by construction; the demo script must respect it.
- **FOUND-04 literally says `SELECT … FOR UPDATE`; D-12 uses a conditional `UPDATE … WHERE status=? RETURNING` instead — and the conditional UPDATE is the SUPERIOR primitive, not merely an equivalent (review-confirmed).** A single-statement conditional UPDATE is an **atomic compare-and-swap**; `SELECT … FOR UPDATE` requires lock→read→decide→write and only adds value when the transition branches on **non-status** columns — which **none of the four claim sites do** (they all gate purely on the current status). So state it affirmatively in the plan: the conditional UPDATE is the better fit, atomic without an explicit transaction block, composes with the optional-`conn=` pattern. Treat FOUND-04's `FOR UPDATE` wording as **satisfied-in-spirit**; flag it for reviewers so a "missing FOR UPDATE" isn't raised. (If a reviewer still insists on the literal `FOR UPDATE`, it's a drop-in swap inside the same helper.)
- **Update the "sole status writer" invariant doc when D-12 lands** — the current `set_status` docstring claims it is "the ONE AND ONLY writer." Leaving that text while adding `claim_status` makes the codebase's own rule a lie. The honest statement: "two writers — `set_status` (unguarded forward transitions inside an owned path) and `claim_status` (atomic guarded claim at every contended gate)."
- **The over-40-no-OT rule is a DEMO beat, not just a guard** — weekly "Bob worked 45 hours" with no OT field on camera, producing "is that 40 regular + 5 overtime, or 45 straight?", visibly catches a money-affecting ambiguity instead of silently under-paying. Under-paying is worse than over-paying for this project's thesis.
- **PDFs per-employee, downloadable per line item, generated in memory** — clicking a line item on the run detail streams that employee's real paystub PDF (reportlab → BytesIO → StreamingResponse). Authenticity beat for anyone who knows payroll; reportlab was chosen for exactly this tabular layout.
- **Reliability over polish on every UI choice** — plain forms + 303 redirects, no JS state, embed the already-committed SVG, no responsive. The dashboard IS the demo surface on a cold Render dyno; nothing should have a JS failure mode on camera.
