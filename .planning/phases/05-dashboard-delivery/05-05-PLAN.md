## Phase Goal

**As an** operator, **I want to** approve payroll runs through a race-safe gate, **so that** duplicate approvals are structurally impossible and a failed delivery never silently strands a run.

<deviation_note>
## D-13c sharpening: real send-state in email_messages, not overloaded direction

D-13c (CONTEXT.md) specifies writing an intent row BEFORE the provider call for crash-safe idempotency. The original plan expressed this intent via a temporary direction='outbound-pending' value. The verified schema fact (schema.sql:132 CHECK direction IN ('inbound','outbound')) and Codex finding #3 confirm that 'outbound-pending' would violate the CHECK constraint at runtime.

REFINEMENT in this plan: gateway.send_outbound writes `purpose` and `send_state='sent'` (the real mechanism added by Plan 03's DDL). The Phase 5 stub is still synchronous (no live network), so writing send_state='sent' inline is correct — in Phase 5 there is no crash window between the send and the insert. Phase 6 live-provider wiring writes send_state='reserved' BEFORE the provider call and flips to 'sent'/'failed' after — exactly the crash-safe ordering D-13c mandates — and it is a no-op swap on this plan's code.

No 'outbound-pending' value is used anywhere in gateway.py or tests. The gateway.py must_have artifact changed from "contains: outbound-pending" to "contains: send_state".
</deviation_note>

<deviation_note>
## Finding #6 disposition: stale in-flight state recovery (D-13b invariant extension)

D-13b (CONTEXT.md) mandates that no claimed non-terminal state strands without a recovery path. The existing retrigger route claims only from ERROR and APPROVED. Codex finding #6 identifies that a dead worker after a `resume` claim leaves `extracting` with no recovery UI.

RESOLUTION in this plan: the retrigger route is extended to also claim from `received`, `extracting`, `computed`, and `sent` — BUT only when the run's `updated_at` is older than a staleness threshold (5 minutes default, configurable). This avoids force-restarting a freshly-started in-flight run. The staleness check is in the route handler logic, not in claim_status itself (claim_status remains a pure CAS with no staleness logic). [There is no `computing` status — the post-calculation in-flight state is `computed`.]

For `sent` semantics (finding #6 second part): `sent` is treated as a transitional state (not terminal-success); `reconciled` is the only terminal-success. The run advances from `sent` to `reconciled` immediately in `_deliver` (steps 9-10 are sequential in the same synchronous call). A crash between `sent` and `reconciled` would strand in `sent` — to handle this, the retrigger route also attempts to claim from `sent` with a staleness check, advancing back to `received` for a full re-run (safe because the already-sent confirmation guard in `_deliver` prevents duplicate send). This decision is ON THE RECORD here.
</deviation_note>
