## Phase Goal

**As an** operator, **I want to** approve payroll runs through a race-safe gate, **so that** duplicate approvals are structurally impossible and a failed delivery never silently strands a run.

<objective>
The Jinja2 dashboard UI slice. This plan delivers the four HTML pages that make Beat 1 visible to the operator.

This plan incorporates two MEDIUM review findings:
1. DASH-04 drill-in: the per-fixture table in eval.html must include a "Raw Input" column showing the fixture's inbound email body text beside Expected and Actual columns. IMPORTANT: eval/summary.json per_fixture records do NOT store body_text (verified: keys are fixture_id, fixture_path, fixture_category, extraction, reconciliation, decision). The eval view shows the chart plus per-fixture drill-in with the raw input body. The "Send Test Email" button is safe to click repeatedly without a unique constraint error.

Purpose: The dashboard IS the demo surface. This plan makes the pipeline visible.
Output: 4 templates, 1 CSS file, 6 new routes in main.py.
