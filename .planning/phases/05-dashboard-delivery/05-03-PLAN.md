## Phase Goal

**As an** operator, **I want to** approve payroll runs through a race-safe gate, **so that** duplicate approvals are structurally impossible and a failed delivery never silently strands a run.

<deviation_note>
## Sharpening D-13c: Real send-state mechanism instead of overloading `direction`

D-13c (CONTEXT.md) specifies writing an intent row BEFORE the provider call for crash-safe idempotency. The original plan expressed this as writing direction='outbound-pending'. The Codex review (finding #3) and the verified schema fact (app/db/schema.sql:132 CHECK direction IN ('inbound','outbound')) confirm that 'outbound-pending' is SCHEMA-INVALID — it violates the CHECK constraint and would raise at runtime.

REFINEMENT (preserves the D-13c intent, changes only the encoding):
- Add a `send_state TEXT CHECK (send_state IN ('reserved','sent','failed'))` NULLABLE column to email_messages. Nullable (not NOT NULL DEFAULT 'sent') because inbound rows have no send lifecycle — giving them send_state='sent' by default would be semantically wrong and would weaken audit queries (R2-MEDIUM finding: inbound rows get 'sent' by default). Inbound rows keep send_state=NULL; outbound rows are written with send_state='sent' by gateway.send_outbound in Phase 5's stub (synchronous, no crash window). The Phase 6 live-provider swap writes send_state='reserved' BEFORE the provider call, then flips to 'sent' on success or 'failed' on error — exactly the crash-safe ordering D-13c mandates.
- The 'outbound-pending' fake direction value is NOT used anywhere. This is a sharpening of D-13c's encoding, not a reversal of its intent.
- R2-HIGH fix: get_outbound_message_id now filters send_state='sent' in addition to purpose — a row with send_state='reserved' (intent-only, pre-send crash) or 'failed' does NOT count as proof-of-send and will not cause the guard to skip a required send.

Also sharpening D-13 (finding #1): Add `purpose TEXT CHECK (purpose IN ('clarification','confirmation'))` to email_messages and a UNIQUE constraint on (run_id, purpose). The existing get_outbound_message_id(run_id) is updated to get_outbound_message_id(run_id, purpose) so a prior clarification row cannot match a confirmation guard query — the core correctness hole Codex identified.

## NEW-1 (Round-3): uq_email_run_purpose + sent-only guard INTERACTION requires UPSERT on outbound write

The R2-HIGH fix (send_state='sent' filter in get_outbound_message_id) and the uq_email_run_purpose UNIQUE constraint now interact: when a `reserved` or `failed` row exists for (run_id, purpose), the read guard correctly does NOT skip the send — but the write path (gateway.send_outbound → insert_email_message) then tries to INSERT a new (run_id, purpose) row and hits the unique constraint, crashing the retry.

FURTHER SHARPENING of D-13c (preserves intent, hardens the write path):
- `insert_email_message` must UPSERT on (run_id, purpose) when both values are non-NULL:
  `INSERT INTO email_messages (...) VALUES (...) ON CONFLICT (run_id, purpose) DO UPDATE SET send_state=EXCLUDED.send_state, message_id=EXCLUDED.message_id, subject=EXCLUDED.subject, body_text=EXCLUDED.body_text, created_at=now()`
- This turns a retry over a prior `reserved` or `failed` row into an advance to `sent` rather than a constraint crash.
- NULL-purpose rows (inbound) are unaffected: Postgres treats NULLs as distinct in unique constraints, so inbound rows (purpose=NULL) never conflict and the existing ON CONFLICT (message_id) DO NOTHING path on the inbound INSERT is unchanged.
- The inbound INSERT helper (`insert_inbound_email`) is a SEPARATE function and must NOT be modified.
- `gateway.send_outbound` is the only caller that passes non-NULL purpose to `insert_email_message`; the upsert clause is therefore only ever triggered on the outbound path.
- This is a sharpening of D-13c: the send-state lifecycle now explicitly supports reserve→sent advancement via upsert, not just the initial write.
</deviation_note>
