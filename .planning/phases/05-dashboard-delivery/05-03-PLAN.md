## Phase Goal

**As an** operator, **I want to** approve payroll runs through a race-safe gate, **so that** duplicate approvals are structurally impossible and a failed delivery never silently strands a run.

<objective>
The atomic-status-claim slice. This plan delivers the foundational correctness primitives that ALL four claim call sites (approve, reject, resume, retrigger) depend on. Without this plan's deliverables, the delivery path cannot be made race-safe and the error recovery cannot work.

Three concrete deliverables:
1. `claim_status` CAS helper + `_TERMINAL_STATUSES` fix + invariant doc update + `alias_candidates` DDL (repo.py + schema.sql)
2. `resume_pipeline` hardened to use `claim_status` (closes CR-02 non-atomic seam)
3. Over-40-no-OT validation rule in `validate.py` (D-05 guard that makes the second demo beat work)

After this plan: the D. Reyes money-misroute guard and the double-approval race are structurally closed in the DB layer, and the weekly "Bob worked 45 hours with no OT" case flows through the clarification gate.

This plan is NOT autonomous: it includes a blocking checkpoint to apply the live-DB DDL ALTER and confirm the alias_candidates column is present on the running local database. Without this checkpoint, Plan 05-07 fails at runtime.

Purpose: Makes approve race-safe and dead delivery recoverable — the prerequisite for Beat 1.
Output: repo.py (4 changes), schema.sql (1 DDL column), validate.py (1 helper + 1 loop), orchestrator.py (1 claim replacement), plus a human-verified live-DB DDL apply.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-06 | Repudiation | claim_status — double-approval race | mitigate | Conditional `UPDATE … WHERE status=%s RETURNING id` is atomic in Postgres; exactly one concurrent caller gets a row back; the other drops cleanly (D-12, FOUND-04). `approved` removed from _TERMINAL_STATUSES so delivery failure can route to ERROR (D-13b). |
| T-05-07 | Integrity | validate.py — OT silent underpay | mitigate | D-05 rule emits ValidationIssue when weekly employee has >40 regular hours with no OT → gates to clarification; explicit-zero treated same as absent (never silently underpays) |

| T-05-09 | Availability | resume_pipeline — late duplicate reply processing | mitigate | claim_status(AWAITING_REPLY → EXTRACTING) — losing caller returns cleanly, no double-pipeline-run; CR-02 residual closed |
| T-05-SC | Tampering | npm/pip/cargo installs | accept | No new packages; uv.lock unchanged |
