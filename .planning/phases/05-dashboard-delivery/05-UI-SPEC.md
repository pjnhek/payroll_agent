# Phase 5 — UI Design Contract

> Visual and interaction contract for the operator-facing demo dashboard.
> Generated by gsd-ui-researcher; verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (server-rendered Jinja2 3.1.6) |
| Preset | not applicable |
| Component library | none |
| Icon library | inline SVG snippets only (colored status circles, no external icon library) |
| Font | system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", Helvetica Neue, Arial, sans-serif (body); ui-monospace, "SF Mono", Consolas, "Liberation Mono", Menlo, monospace (raw-email `<pre>`) |

**Rationale:** The stack is locked (CONTEXT.md D-06, CLAUDE.md §7). No SPA, no React, no shadcn, no build step. System fonts load before the first paint with zero network dependency — critical for a cold Render free dyno on camera. Reliability beats polish on every choice here.

## Spacing Scale

Declared values (all multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge inner padding, table cell gap |
| sm | 8px | Table cell padding (td/th), compact button padding |
| md | 16px | Card/section padding, form element spacing, nav height padding |
| lg | 24px | Section gap, `<pre>` block padding |
| xl | 32px | Column gutter in the 3-column run-detail grid |
| 2xl | 48px | Page top/bottom margin |
| 3xl | 64px | Major section break (page wrapper max-width padding) |

Exceptions:
- Touch targets: not applicable — single-screen desktop demo only (D-07, PROJECT.md). No mobile breakpoints. No 44px touch-target requirement.
- Monospace `<pre>` block: uses `lg` (24px) padding on all sides so the raw email content has breathing room without wasting screen width in the leftmost column.
- Status badge: 4px vertical + 8px horizontal inner padding (`xs` / `sm`), matching the compact table-row rhythm.

## Typography

| Role | Size | Weight | Line Height | Font Stack |
|------|------|--------|-------------|------------|
| Body | 14px | 400 (regular) | 1.5 | system-ui stack |
| Label / table header | 12px | 600 (semibold) | 1.3 | system-ui stack |
| Heading (page title, section title) | 20px | 600 (semibold) | 1.2 | system-ui stack |
| Raw-email `<pre>` | 12px | 400 (regular) | 1.6 | ui-monospace stack |

Two weights only: 400 (regular) and 600 (semibold). No italic variants needed.

**Rationale for sizes:**
- 14px body: dense, scannable operator table — the same size payroll software uses for data grids. Readable at screen-recording resolution.
- 12px label/table-header: tight enough to label data columns without dominating them; semibold compensates for the small size.
- 20px heading: unambiguous hierarchy without consuming vertical space on a data-heavy page.
- 12px monospace pre: matches the label size so the three columns feel vertically aligned at the top; monospace stack preserves any spacing/indentation in the raw email exactly as sent.

**Contrast:** All badge text/background pairs meet WCAG AA at 14px (verified by ratio):
- `#1D4ED8` on `#DBEAFE`: ~5.2:1 (pass)
- `#15803D` on `#DCFCE7`: ~5.8:1 (pass)
- `#B91C1C` on `#FEE2E2`: ~5.1:1 (pass)
- `#6B7280` on `#F3F4F6`: ~4.6:1 (pass at 12px bold per WCAG)
- `#DC2626` background with white `#FFFFFF` text: ~5.5:1 (pass)
- `#2563EB` background with white `#FFFFFF` text: ~5.9:1 (pass)

Readable at screen-recording resolution — the camera and monitor pipeline is the hard constraint, not a color-blind audit (no auth, no real users).

## Layout Contracts

### base.html

- `<body>` background: `#F9FAFB` (dominant).
- Top nav bar: `#FFFFFF` background, 1px bottom border `#E5E7EB`. Height 48px (3× `md`). Contains: app name "Payroll Agent" in 20px semibold text-primary, navigation links "Runs" and "Eval" in 14px regular text-primary with underline on `:hover`.
- Page content wrapper: `max-width: 1280px`, centered, horizontal padding `3xl` (64px) on each side, vertical padding `2xl` (48px) top and bottom.
- Page `<h1>`: 20px semibold, `#111827`, margin-bottom `md` (16px).

### runs_list.html (DASH-01, from D-08)

**Layout:** Single-column full-width `<table>` under the `<h1>`.

**Column order (left-to-right):**
1. Created at — `#6B7280` 12px, format `YYYY-MM-DD HH:MM`
2. Business name — 14px text-primary
3. Status badge — colored by class (see Badge Contract below)
4. Summary — one-line: either "N employees" or the first `gate_reason` string, 14px `#6B7280`
5. Action — "View" link to `/runs/{run_id}`, 14px text-primary underline

**Table styling:**
- `<thead>` background: `#FFFFFF` (secondary). `<th>` text: 12px semibold `#111827`. Padding: `sm` vertical, `sm` horizontal (8px each).
- `<tbody>` background: `#F9FAFB`. `<td>` text: 14px regular `#111827`. Padding: `sm` vertical, `sm` horizontal.
- Row border-bottom: 1px `#E5E7EB`.
- `awaiting_approval` rows get no special row highlight — the badge emphasis is sufficient triage signal.
- **Primary visual focal point:** the `badge-pending` (`awaiting_approval`) badge is the page's focal point — the operator's eye should land there first. It is the only badge using the accent-blue family; all other badges are muted by comparison so "what needs me" reads at a glance.
- In-table action links ("View"): `color: #111827` (text primary) with `text-decoration: underline` on `:hover` — matching the `base.html` nav link pattern. Not the accent color.
- Rows are in reverse-chronological order (newest first).

**Empty state (no runs yet):**
- Heading: "No payroll runs yet"
- Body: "Use the Send test email button on the eval page to fire a demo fixture through the pipeline."

### run_detail.html (DASH-02, DASH-03, from D-07)

**Decision reason banner:**
Rendered full-width ABOVE the 3-column grid. Height auto (wraps content). Border-radius 4px. Padding `md` (16px). Font 14px regular. Bold prefix label followed by the reason text.

| Run State | Banner Style | Label | Content |
|-----------|-------------|-------|---------|
| `final_action = process` | Green (`#F0FDF4` bg, `#166534` text, `#86EFAC` border) | "Decision: Process" | "All names resolved. No validation issues." |
| `final_action = request_clarification` | Amber (`#FFFBEB` bg, `#92400E` text, `#FCD34D` border) | "Decision: Clarification requested" | Bulleted list of `gate_reasons` + `unresolved_names` + `missing_fields`. One bullet per reason. |
| `status = error` | Red (`#FFF1F2` bg, `#991B1B` text, `#FECDD3` border) | "Error" | `error_reason` string. "Use the Re-trigger button below to restart this run from the beginning." |
| `awaiting_reply` | Amber (same as clarify) | "Awaiting client reply" | "A clarification email has been sent. Waiting for the client to respond." |

**3-column CSS grid (from D-07):**

```
grid-template-columns: 1fr 1fr 1fr
column-gap: xl (32px)
```

Column labels (12px semibold `#111827`, margin-bottom `sm` 8px):
- Col 1: "Raw Email (as received)"
- Col 2: "Extracted Data"
- Col 3: "Computed Paystubs"

**Column 1 — Raw email `<pre>`:**
- `background: #FFFFFF`, padding `lg` (24px), border `1px solid #E5E7EB`, border-radius 4px.
- Font: 12px monospace, line-height 1.6.
- `overflow-x: auto` so long lines scroll horizontally within the column rather than breaking the grid.
- `white-space: pre` — preserves the exact client-sent formatting.
- `max-height: 600px`, `overflow-y: auto` — prevents a very long email from dominating the screen.
- This column contains `email_messages.body_text` (the cleaned body from `repo.load_inbound_email`).

**Column 2 — Extracted data table:**
- A `<table>` per employee (one table per extracted employee, stacked vertically with `md` (16px) gap between them).
- Each table header row: employee submitted name in 14px semibold `#111827`, spanning all columns.
- Table rows: field label (12px semibold `#6B7280`) | value (14px regular `#111827`).
- Fields rendered: regular hours, OT hours, vacation hours, sick hours, holiday hours, 401k override (if non-null).
- If `extracted_data` is null (run still in `received` / `extracting`): display "Extraction in progress..."
- Resolution status per employee: small badge after the submitted name showing the `NameMatchResult.source` value — "exact", "stored-alias", or "unresolved" — using the in-flight/good/bad badge colors respectively.

**Column 3 — Computed paystubs table:**
- One `<table>` per `PaystubLineItem`, stacked with `md` (16px) gap.
- Table header: employee full name (resolved) in 14px semibold.
- Rows: Gross Pay | Pre-tax 401k | Social Security (6.2%) | Medicare (1.45%) | Federal Withholding | Net Pay
- Values: right-aligned, formatted as `$X,XXX.XX` (2 decimal places).
- If `state_withholding` is null: omit the row (do not show "$0.00 State").
- If `additional_medicare_not_modeled = true`: show a 12px `#6B7280` footnote below the table: "Additional Medicare (0.9% over $200k) not modeled."
- A "Download PDF" link below each employee's table: `GET /runs/{run_id}/pdf/{employee_id}`. 14px `#111827` (text primary) with `text-decoration: underline` on `:hover`, matching the `base.html` nav link pattern (not the accent color).
- If paystubs are not yet computed: display "Calculation in progress..."

**Operator controls (DASH-03) — only rendered when `status = awaiting_approval`:**

Rendered as a `<div>` below the 3-column grid, with `md` (16px) top margin.
Contains two `<form>` elements side by side (inline-flex, gap `sm` 8px):

Form 1 — Approve:
```html
<form method="post" action="/runs/{run_id}/approve">
  <button type="submit" class="btn-approve">Approve &amp; Send</button>
</form>
```

Form 2 — Reject (with inline confirmation via `<details>` / confirm dialog):
```html
<form method="post" action="/runs/{run_id}/reject"
      onsubmit="return confirm('Reject this payroll run? This cannot be undone.')">
  <button type="submit" class="btn-reject">Reject</button>
</form>
```

Note on JS confirmation: the `confirm()` dialog is the one JS call permitted. It is a browser native dialog — zero failure mode, no framework dependency, and it degrades to "submits anyway" if somehow skipped. The form still does the right thing server-side.

Re-trigger button (INGEST-05) — only rendered when `status = error` OR `status = approved` (delivery failed):
```html
<form method="post" action="/runs/{run_id}/retrigger">
  <button type="submit" class="btn-retrigger">Re-trigger from Start</button>
</form>
```

**Button styles (no framework classes — raw CSS):**
- `.btn-approve`: background `#2563EB`, color `#FFFFFF`, padding `8px 20px`, border-radius 4px, font 14px semibold, border none.
- `.btn-reject`: background `#DC2626`, color `#FFFFFF`, same geometry.
- `.btn-retrigger`: background `#6B7280`, color `#FFFFFF`, same geometry.
- All buttons: `cursor: pointer`, no `:hover` transform — keep it static and reliable.

### eval.html (DASH-04, from D-09a)

**Layout:** Two sections stacked vertically, `xl` (32px) gap between them.

**Section 1 — Headline metrics:**
A `<dl>` (definition list) rendered as a 3-column horizontal strip. Each `<dt>` is 12px semibold `#6B7280` label, `<dd>` is 28px semibold `#111827` value. Three metrics from `eval/summary.json`:
- "Extraction F1": `extraction_overall_f1` formatted as a percentage (e.g. "94%")
- "Decision Accuracy": computed from `confusion_matrix.true_process + confusion_matrix.true_clarify / total` as a percentage
- "False Process Rate": `false_process_rate` formatted as a percentage (e.g. "0%")

Below the metrics strip: a 14px `#6B7280` line showing "Eval run: {generated_at} — Models: {extraction_model_id}"

**Section 2 — Chart:**
```html
<img src="/eval/chart.svg" alt="Per-category eval chart" style="max-width: 100%; height: auto;">
```
Served via `GET /eval/chart.svg` as `image/svg+xml`. Embedded as-is — never re-rendered.

**Section 3 — Per-fixture drill-in:**
A `<table>` with columns: Fixture | Category | Extraction F1 | Reconciliation | Decision | Status
- Each row links to a fixture detail panel (below the table, toggled by `id` anchor or a second GET route `GET /eval/fixtures/{fixture_id}`).
- Fixture detail shows: raw email body (read from `eval/fixtures/{fixture_path}` on disk) in a monospace `<pre>`, plus expected vs actual extraction/decision side-by-side.
- If `eval/summary.json` does not exist: display "No eval results available. Run the eval locally: `uv run python eval/run_eval.py`"

**"Send test email" button (DASH-05):**
Placed at the bottom of `eval.html` (and also in the nav as a standalone element — or at the bottom of `runs_list.html`). A `<form>` that POSTs to `/demo/send-test`:
```html
<form method="post" action="/demo/send-test">
  <button type="submit" class="btn-approve">Send Test Email</button>
</form>
```
Uses the accent (blue) button style — this is one of the three reserved accent usages. After the POST, the handler 303-redirects to `/runs` so the operator sees the new run appear in the list and watches it advance through statuses on manual refresh.

## Interaction Contracts

All interactions sourced from D-06 (POST-redirect-GET, locked).

| Action | Trigger | Handler | Response | Side Effects |
|--------|---------|---------|----------|-------------|
| View runs | `GET /runs` | Jinja2 route | Renders `runs_list.html` | None |
| View run detail | `GET /runs/{id}` | Jinja2 route | Renders `run_detail.html` | None |
| Approve | Form POST `/runs/{id}/approve` | `_operator_transition` | 303 → `GET /runs/{id}` | `claim_status(awaiting_approval→approved)` + delivery + advance to sent/reconciled (or error) |
| Reject | Form POST `/runs/{id}/reject` | `_operator_transition` | 303 → `GET /runs/{id}` | `claim_status(awaiting_approval→rejected)` |
| Re-trigger | Form POST `/runs/{id}/retrigger` | new route | 303 → `GET /runs/{id}` | `claim_status(error→received)` or `claim_status(approved→received)` + background re-run |
| Send test email | Form POST `/demo/send-test` | new route | 303 → `GET /runs` | POSTs clean demo fixture to `/webhook/inbound` (backgrounded); new run appears in list |
| Download PDF | `GET /runs/{id}/pdf/{emp_id}` | new route | `StreamingResponse` PDF | reportlab BytesIO, no disk write |
| View eval | `GET /eval` | Jinja2 route | Renders `eval.html` | None (hermetic disk read of summary.json + chart.svg) |

**No JS state machine anywhere.** The one permitted JS call is `confirm()` on the Reject button form submit. Full-page reloads on every action. Stateless GET routes — no session, no auth.

**Sync vs background (D-06b):**
- Approve: synchronous bounded by D-10b hard ~3s draft timeout on `compose_confirmation`. Total approve→303 is bounded to ~4–5s worst case on a cold dyno.
- Reject: fully synchronous (~50ms).
- Re-trigger: synchronous `claim_status` call; pipeline re-run is a `BackgroundTask` (mirrors original webhook flow). 303 fires immediately.
- Send test email: the POST goes to `/webhook/inbound` which already backgrounds the pipeline. 303 fires immediately. Operator manually refreshes the runs list to watch states advance — this is a legible demo beat.
