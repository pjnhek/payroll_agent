<objective>
Build D-06's anti-rot guard: a registry of every falsifying mutation this phase executed, whose
targets are resolved as **real AST nodes** and whose entries each name the **specific assertion**
expected to fail.

Purpose: the evidence document plan 21-11 publishes is a static artifact, and static artifacts rot.
A full automated mutation harness was considered and rejected as over-engineering for four proofs —
the patches themselves rot, and a no-longer-applying patch becomes another gate to get right (D-05).
D-06 does the anti-rot job instead, with two independent halves:

- **AST resolution** catches a **mis-targeted** mutation. A grep-based check can be satisfied by a
  docstring or a comment — and that is not hypothetical here: `claim_job`'s own docstring contains
  prose copies of both of the SQL fragments this phase mutates, and this repo has already had a
  real blind spot look like a passing proof because a mutation landed on a docstring copy of the
  string rather than the live one. `git grep -E` also silently ignores word-boundary escapes, which
  has produced a false-clean verification in this repo before.
- **The named assertion** catches a **right-target/wrong-red**. A mutation that reddens something
  unrelated has not falsified anything.

Both halves are required; either alone leaves a hole.

Output: an AST-resolving guard with its own synthetic red-proofs, and a populated registry covering
all four proofs.

**Corrections applied from cross-AI review (Codex, HIGH ×2 — both independently verified against
live source). Both are the same irony: a guard written to catch blind guards, itself blind.**

1. **The resolver could not resolve one of its own four targets.** An earlier draft searched only
   non-docstring **string constants** inside the named function. But PROOF-02's mutation target is
   `external_event_id = request.headers["svix-id"]` at `app/routes/webhook.py:143` — an executable
   `ast.Assign` whose value is a `Subscript`, not a string literal anywhere. The guard would have
   been structurally incapable of resolving it, and the registry entry would either have been
   dropped or fudged into some nearby string. Targets are therefore modelled below as **AST
   predicates** — enclosing function + node type + a normalized value path — with string-fragment
   matching available as one predicate kind among several, not as the only mechanism.
2. **"The named assertion exists" was a text search, which a comment can satisfy.** Searching the
   proof's whole source file for the assertion text recreates *exactly* the docstring/comment false
   positive D-06 was written to prevent, one level up: a `# we assert final_row["attempts"] == 2`
   comment would satisfy it while the assertion itself had been deleted. The assertion check below
   walks to the target test function by AST and requires the text inside an `ast.Assert` node's
   condition or message.
</objective>

    **`sql_fragment` predicate kind (targets living inside SQL string literals):**
    - A fragment appearing in a non-docstring string literal inside the named function resolves.
    - The same fragment appearing ONLY in that function's docstring does NOT resolve.
    - The same fragment appearing ONLY in a comment does NOT resolve.
    - The same fragment appearing only inside a DIFFERENT function does NOT resolve.
    - A named function that does not exist in the source does NOT resolve, and says so distinctly

    **`assignment` predicate kind (executable targets — the shape PROOF-02 needs):**
    - An `ast.Assign` to the named target whose value matches the declared value path resolves —

      `request.headers["svix-id"]`.
    - The same assignment with a DIFFERENT value (say `external_event_id = uuid.uuid4().hex`) does
      NOT resolve — the predicate is about the derivation, not merely the variable name.
    - The same assignment appearing only in a docstring or a comment does NOT resolve.
    - An assignment to a different name does NOT resolve.

    **assertion resolution (D-06's second half):**
    - Assertion text appearing inside an `ast.Assert` condition within the named test function
      resolves.
    - The identical text appearing only in a **comment** in that file does NOT resolve.
    - The identical text appearing only in the test's **docstring** does NOT resolve.
    - The identical text appearing inside an `ast.Assert` in a DIFFERENT function does NOT resolve.

    Create `tests/test_proof_mutation_targets.py` with a module docstring stating what the guard
    establishes and, explicitly, what it does not: it proves a published mutation still names a real
    location in live source, and it does not re-execute the mutation.

    Implement a pure resolver — no I/O in the function itself — that takes a **target predicate**
    rather than a bare string, along the lines of
    `resolve_target_in_source(source: str, *, function_name: str, predicate: TargetPredicate) -> bool`.
    A predicate is a small structured description with a `kind` discriminator; implement the two
    kinds this phase's four mutations actually need, and shape the dispatch so a third kind can be
    added later without rewriting the walker.

    Common steps for every kind:

    1. `ast.parse` the source.
    2. Walk for a `FunctionDef` or `AsyncFunctionDef` whose `name` equals `function_name`. If none
       exists, that is a distinct failure from an unsatisfied predicate — return or raise in a way
       the caller can distinguish, and give the tests distinct messages for the two cases.
    3. Restrict all further inspection to **that function's subtree**. Comments never appear in an
       AST at all, so they are excluded for free — state that in a code comment so the next reader
       knows the exclusion is deliberate and complete.

    **Kind `sql_fragment`** — for targets that live inside a SQL string literal (PROOF-01's
    attempts increment and PROOF-04's expired-lease disjunct, both inside `claim_job`'s SQL, and
    PROOF-03's target if it lands on a SQL literal). Collect every `ast.Constant` whose value is a
    string, **excluding the function's own docstring node** (the first statement's value when it is
    a string constant), and return whether the fragment appears in any of them.

    Normalise whitespace before comparing — the SQL in this repo is indented inside triple-quoted
    strings and a fragment copied from a docstring uses different spacing than the live SQL. Collapse
    runs of whitespace on both sides before the containment check, so the guard resolves the
    *content* rather than the formatting. Say why in a comment: without normalisation the guard
    would red on a harmless reindent, which is how a guard gets deleted for being annoying.

    **Kind `assignment`** — for **executable** targets, which a string-constant scan cannot see at
    all. This kind exists because PROOF-02's mutation target is
    `external_event_id = request.headers["svix-id"]` (`app/routes/webhook.py:143`): an `ast.Assign`
    whose value is a `Subscript`. Read that line in live source before implementing, so the
    predicate is written against the real node shape rather than an imagined one.

    The predicate carries the assignment target name, the expected value **node type**, and a
    normalized **value path** (a canonical rendering of the value expression, e.g. the dotted
    receiver plus the subscript key). Walk the function subtree for `ast.Assign`/`ast.AnnAssign`
    nodes whose target matches the name, render each one's value to the same normalized form, and
    return whether any matches. Do the rendering structurally — from node attributes — and not by
    slicing the original source text: a source slice reintroduces the text-matching weakness this
    whole module exists to avoid. Where the standard library offers a faithful structural rendering
    of a node, prefer it over a hand-rolled walker, and normalize whitespace consistently on both
    sides before comparing.

    Also implement the **assertion resolver**, D-06's second half, as its own pure function — for
    example `assertion_is_asserted_in(source: str, *, function_name: str, assertion_text: str) ->
    bool`. It must NOT search the file. Walk to the named test function by AST, then collect only
    `ast.Assert` nodes within that function's subtree, and check the normalized assertion text
    against each node's **condition** rendering and its **message** rendering. A match anywhere else
    — a comment, the test docstring, a neighbouring function — must not satisfy it. Write in a code
    comment exactly why: a text search over the file can be satisfied by a comment, which is
    precisely the docstring/comment false positive D-06 exists to prevent, and letting it back in
    one level up would make this guard the very thing it guards against.

    Write one test per behavior above, each building a small synthetic source string inline and
    asserting the resolver's answer. The docstring-copy and comment-copy cases matter most — they
    are the exact trap D-06 exists to close — so give each its own test with a failure message
    naming the trap. Include the `assignment`-kind case whose value differs (same variable, wrong
    derivation) and the assertion-in-a-comment case; those two are the review's findings made
    executable, and without them the corrections are only claimed.
