<objective>
Land the proof-identity substrate D-02 requires: a registered `proof` pytest marker carrying a
stable id, and a selection-layer completeness checker whose *decision logic is a pure function*
so it can be red-proofed hermetically before it is ever wired to CI.

Purpose: PROOF-05's real substance (per CONTEXT.md's Requirement corrections — the
"hard-coded file list" premise is stale) is closing the gap `concurrency-proof.yml` documents
in its own comment: a typo'd marker on one newly-added test while the other queueproof tests
still pass, so the log says "N passed" and the new proof silently never ran. That gap lives at
the *selection* layer, which no execution-layer log guard can see.

Output: a registered marker, a pure `evaluate_inventory()` decision function with its subprocess
collector, and hermetic tests proving the checker reds on each failure shape it claims to catch.

**This plan deliberately does NOT edit `.github/workflows/concurrency-proof.yml`.** The gate
asserts all four PROOF ids exist; wiring it before the proofs are tagged would red CI for the
rest of the phase. Wiring + the live-repo assertion land in plan 21-09 (wave 4).

  </read_first>
  <action>
    Append a fourth entry to `pyproject.toml`'s `markers` list registering a marker named `proof`.
    Its description must state that the marker identifies a durability proof by id, that the id is
    passed as the **keyword argument `id`**, and that the keyword spelling is load-bearing.

    The keyword spelling is not stylistic and must be recorded as the reason: pytest's `-m`
    marker-expression syntax supports **keyword** marker arguments only. Verified against the
    pinned pytest (9.1.1) in this repo: `-m "proof(id='PROOF-01')"` collects exactly the tests
    carrying that id, while a marker applied with a positional argument cannot be selected by any
    `-m` expression at all. D-02's completeness property ("each of PROOF-01..04 appears exactly
    once") is only checkable at the selection layer if the id is selectable — so the canonical
    application form for this phase is `@pytest.mark.proof(id="PROOF-0N")`.

    Then add a `TestProofMarkerRegistered` class to `tests/test_queue_config.py`, sibling to the
    existing `TestQueueproofMarkerRegistered`, following its exact shape. It must assert two
    things against `_PYPROJECT_TOML.read_text()`: that a marker entry beginning `proof:` is
    registered, and that the registered description records the keyword-`id` requirement (assert
    on a substring of the description you wrote, so a future edit that drops the rationale reds).
    Do not modify `TestQueueproofMarkerRegistered`, `TestD14NoWideningGuard`, or any other
    existing class in that file.
  </action>
  <verify>
    <automated>uv run pytest tests/test_queue_config.py -v</automated>
  </verify>

<task type="auto" tdd="true">
  <name>Task 2: Build the inventory checker as a pure decision function plus a thin collector</name>
  <files>scripts/check_proof_inventory.py, tests/test_proof_inventory.py</files>
  <behavior>
    - `evaluate_inventory` returns `[]` when every expected id maps to exactly one node id and no
      extra `proof`-marked node id exists (the no-false-positive half).
    - A missing id (empty node-id list) yields a violation naming that id.
    - A duplicate id (two node ids) yields a violation naming that id and both node ids.
    - A stray/typo'd id — a node id carrying the `proof` marker that appears in the all-marked set
      but under no expected id — yields a violation naming that node id.
    - All three failure shapes can be produced simultaneously and all three are reported.
  </behavior>
  <read_first>
    - `scripts/check_operator_resolution_inventory.py` — the in-repo precedent for a
      `scripts/`-hosted checker: module docstring style, `main() -> int` convention, and how it
      reports violations to stderr. Mirror its shape; do not invent a new script idiom.
    - `tests/test_bound01_private_imports.py` — specifically the pairing of
      `scan_tree_for_violations` (a pure function over parsed input) with
      `test_scanner_detects_synthetic_violation` (feeds a synthetic bad case through that same
      pure function). This pairing is the required shape for this task.
    - `.github/workflows/concurrency-proof.yml` — read the second step's comment block in full,
      especially the paragraph headed with the stated gap this guard does not close. That
      paragraph is the exact specification this checker implements.
    - `tests/test_bound01_private_imports.py` is scanned by BOUND-01's own guard; `scripts/` is in
      its `SCAN_ROOTS`, so the new script must not reference a private name from another module.
  </read_first>
  <action>
    Create `scripts/check_proof_inventory.py` with a module docstring explaining that it closes
    the selection-layer completeness gap named in `concurrency-proof.yml`'s own comment.

    It exposes exactly four public names:

    - `EXPECTED_PROOF_IDS` — a tuple of the four canonical ids in order: `PROOF-01`, `PROOF-02`,
      `PROOF-03`, `PROOF-04`.
    - `evaluate_inventory(per_id, all_marked, expected_ids)` — the **pure** decision function.
      `per_id` is a mapping of proof id to the sequence of node ids collected for that id;
      `all_marked` is the sequence of node ids collected for the bare `proof` marker;
      `expected_ids` is the expected tuple. Returns a list of human-readable violation strings,
      empty when conforming. It must detect three independent shapes: an expected id whose
      sequence is empty; an expected id whose sequence has length greater than one (report every
      node id); and any node id present in `all_marked` that appears under no expected id (the
      stray/typo'd-id shape — this is what catches `id="PROOF-3"` or `id="PROOF-O1"`). Each
      violation string must name the offending id and/or node ids so a CI log is actionable.
      This function performs no I/O and starts no subprocess.
    - `collect_inventory(expected_ids)` — the thin impure collector. For each expected id it runs
      `sys.executable -m pytest tests/ -m "proof(id='<ID>')" --collect-only -q` via `subprocess`
      from the repository root, and once more with `-m proof` for the all-marked set. Parse the
      node ids as the output lines containing `::` and starting with `tests/`; ignore pytest's
      trailing summary line and any warning block. Treat pytest's no-tests-collected exit code
      (5) as a legitimate empty result, not a crash — an empty result is precisely the missing-id
      violation `evaluate_inventory` must report, and swallowing it as an error would lose the
      message. Any other non-zero, non-5 exit code must raise, because a broken collection is not
      the same fact as an absent proof. Return `(per_id, all_marked)`.
    - `main()` — calls `collect_inventory(EXPECTED_PROOF_IDS)`, passes the result to
      `evaluate_inventory`, prints each violation to stderr, and returns `1` when any violation
      exists and `0` otherwise. Guard it with the standard `if __name__ == "__main__":
      raise SystemExit(main())`.

    Create `tests/test_proof_inventory.py` exercising `evaluate_inventory` only — no subprocess,
    no live repo scan (the live-repo assertion belongs to plan 21-09, after the proofs are
    tagged). Write one test per behavior listed above: the conforming case returning an empty
    list; a missing id; a duplicate id; a stray id; and the all-three-at-once case. Build the
    inputs as literal dicts and lists of fabricated node-id strings. Each failure test must assert
    on the *content* of the violation string (that it names the offending id), not merely that the
    list is non-empty — a violation list that reds for the wrong reason is the same class of
    defect as a proof that reds for the wrong reason.
  </action>
  <verify>

  </verify>
