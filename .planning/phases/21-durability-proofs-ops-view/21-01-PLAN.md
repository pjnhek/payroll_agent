<objective>
Land the proof-identity substrate D-02 requires: a registered `proof` pytest marker carrying a
stable id, and a selection-layer completeness checker whose *decision logic is a pure function*
so it can be red-proofed hermetically before it is ever wired to CI.

Purpose: PROOF-05's real substance (per CONTEXT.md's Requirement corrections — the
"hard-coded file list" premise is stale) is closing the gap `concurrency-proof.yml` documents
in its own comment: a typo'd marker on one newly-added test while the other queueproof tests
still pass, so the log says "N passed" and the new proof silently never ran. That gap lives at
the *selection* layer, which no execution-layer log guard can see.

Output: a registered marker, a pure `evaluate_inventory()` decision function with its subprocess
collector, and hermetic tests proving the checker reds on each failure shape it claims to catch.

**Correction applied from cross-AI review (Codex, HIGH — independently verified against live
source).** An earlier draft of this plan had the checker collect `-m proof` alone. That is blind to
CI's actual selection: `proof` and `queueproof` are two **independent** registered markers, and
`.github/workflows/concurrency-proof.yml` (the queue-durability step) executes `-m queueproof`. A
test carrying a correct `proof(id="PROOF-0N")` but a missing or misspelled `queueproof` marker would
therefore pass the completeness gate and **never run in CI** — reintroducing, one level up, the
precise vacuity D-02 exists to prevent. The gate below checks the **intersection** that CI runs.

**This plan deliberately does NOT edit `.github/workflows/concurrency-proof.yml`.** The gate
asserts all four PROOF ids exist; wiring it before the proofs are tagged would red CI for the
rest of the phase. Wiring + the live-repo assertion land in plan 21-09 (wave 4).
</objective>

    once") is only checkable at the selection layer if the id is selectable — so the canonical
    application form for this phase is `@pytest.mark.proof(id="PROOF-0N")`.

    existing `TestQueueproofMarkerRegistered`, following its exact shape. It must assert two

    registered, and that the registered description records the keyword-`id` requirement (assert
    on a substring of the description you wrote, so a future edit that drops the rationale reds).
    Do not modify `TestQueueproofMarkerRegistered`, `TestD14NoWideningGuard`, or any other
    existing class in that file.

  <files>scripts/check_proof_inventory.py, tests/test_proof_inventory.py</files>

    - `evaluate_inventory` returns `[]` when every expected id maps to exactly one node id under the
      **CI-executed intersection selection**, no extra `proof`-marked node id exists, and the bare
      `proof` and bare `queueproof` selections agree on those node ids (the no-false-positive half).
    - A missing id (empty node-id list) yields a violation naming that id.
    - A duplicate id (two node ids) yields a violation naming that id and both node ids.
    - A stray/typo'd id — a node id carrying the `proof` marker that appears in the all-marked set
      but under no expected id — yields a violation naming that node id.
    - **A node id carrying `proof(id=...)` that is ABSENT from the `queueproof` selection** — i.e. a
      proof that would pass an id-only inventory but never execute in CI — yields a violation naming
      that node id and stating that CI selects by `queueproof`. This is its own failure shape,
      independent of the three above.
    - All four failure shapes can be produced simultaneously and all four are reported.
  </behavior>
  <read_first>
    - `scripts/check_operator_resolution_inventory.py` — the in-repo precedent for a
      `scripts/`-hosted checker: module docstring style, `main() -> int` convention, and how it
      reports violations to stderr. Mirror its shape; do not invent a new script idiom.
    - `tests/test_bound01_private_imports.py` — specifically the pairing of
      `scan_tree_for_violations` (a pure function over parsed input) with
      `test_scanner_detects_synthetic_violation` (feeds a synthetic bad case through that same

    - `.github/workflows/concurrency-proof.yml` — read the second step's comment block in full,
      especially the paragraph headed with the stated gap this guard does not close. That

    - `tests/test_bound01_private_imports.py` is scanned by BOUND-01's own guard; `scripts/` is in
      its `SCAN_ROOTS`, so the new script must not reference a private name from another module.
  </read_first>
  <action>
    Create `scripts/check_proof_inventory.py` with a module docstring explaining that it closes
    the selection-layer completeness gap named in `concurrency-proof.yml`'s own comment.

    It exposes exactly four public names:

    - `EXPECTED_PROOF_IDS` — a tuple of the four canonical ids in order: `PROOF-01`, `PROOF-02`,
      `PROOF-03`, `PROOF-04`.
    - `evaluate_inventory(per_id, all_marked, queueproof_marked, expected_ids)` — the **pure**
      decision function. `per_id` is a mapping of proof id to the sequence of node ids collected
      for that id **under the CI-executed intersection selection**; `all_marked` is the sequence of
      node ids collected for the bare `proof` marker; `queueproof_marked` is the sequence of node
      ids collected for the bare `queueproof` marker; `expected_ids` is the expected tuple. Returns
      a list of human-readable violation strings, empty when conforming. It must detect four
      independent shapes:
        1. an expected id whose sequence is empty (missing);
        2. an expected id whose sequence has length greater than one (report every node id);
        3. any node id present in `all_marked` that appears under no expected id — the
           stray/typo'd-id shape, which catches `id="PROOF-3"` or `id="PROOF-O1"`;
        4. any node id present in `all_marked` but **absent from `queueproof_marked`** — a proof
           that carries a valid id yet is not selected by the marker CI actually runs. Its
           violation string must say so explicitly, naming `queueproof` and the workflow step, so
           the CI log tells the reader what to add rather than only that something is wrong.
      Each violation string must name the offending id and/or node ids so a CI log is actionable.
      This function performs no I/O and starts no subprocess.
    - `collect_inventory(expected_ids)` — the thin impure collector. For each expected id it runs
      `sys.executable -m pytest tests/ -m "queueproof and proof(id='<ID>')" --collect-only -q` via
      `subprocess` from the repository root — the **intersection**, which is the selection
      `.github/workflows/concurrency-proof.yml` executes. It then runs the collection twice more:
      once with `-m proof` for `all_marked`, and once with `-m queueproof` for `queueproof_marked`.
      Treat pytest's no-tests-collected exit code (5) as a legitimate empty result, not a crash —
      an empty result is precisely the missing-id violation `evaluate_inventory` must report, and
      swallowing it as an error would lose the message. Any other non-zero, non-5 exit code must
      raise, because a broken collection is not the same fact as an absent proof. Return
      `(per_id, all_marked, queueproof_marked)`.

      **Node-id parsing (review MEDIUM — brittleness).** Do not accept "any output line beginning
      with `tests/`": that is brittle under parametrization and under a pytest collection-rendering
      change, and a parsing miss silently understates the inventory. Instead define a module-level
      compiled regex pinning the node-id grammar this repo uses — a path under `tests/` ending in
      `.py`, followed by `::`, followed by a test name, optionally followed by a `[...]`
      parametrization suffix — and accept only fully-matching lines. Anchor the pattern at both
      ends. Any collected line that is neither a full node-id match nor a recognisably ignorable
      line (pytest's trailing count summary, a blank line, or a warnings-block line) must **raise**
      rather than be silently dropped: an unparseable collection output is a broken gate, not an
      empty inventory. Export the compiled pattern as a public name so a test can pin it.
    - `main()` — calls `collect_inventory(EXPECTED_PROOF_IDS)`, passes the result to
      `evaluate_inventory`, prints each violation to stderr, and returns `1` when any violation
      exists and `0` otherwise. Guard it with the standard `if __name__ == "__main__":
      raise SystemExit(main())`.

    Create `tests/test_proof_inventory.py` exercising `evaluate_inventory` and the node-id pattern
    only — no subprocess, no live repo scan (the live-repo assertion belongs to plan 21-09, after
    the proofs are tagged). Write one test per behavior listed above: the conforming case returning
    an empty list; a missing id; a duplicate id; a stray id; **a node id present in `all_marked` but
    absent from `queueproof_marked`**; and the all-four-at-once case. Build the inputs as literal
    dicts and lists of fabricated node-id strings. Each failure test must assert on the *content* of
    the violation string (that it names the offending id, and for the fourth shape that it names
    `queueproof`), not merely that the list is non-empty — a violation list that reds for the wrong
    reason is the same class of defect as a proof that reds for the wrong reason.

    Add two tests pinning the node-id pattern: one asserting it matches a plain node id and a
    parametrized node id (`tests/x.py::test_y[case-1]`), and one asserting it rejects pytest's
    trailing summary line and a bare directory path. The parsing contract is part of the gate's
    correctness, so it gets the same both-halves treatment as the decision logic.
  </action>
