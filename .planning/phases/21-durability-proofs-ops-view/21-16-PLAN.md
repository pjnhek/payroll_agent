<objective>
Fix a live deploy-blocking defect in the production migration path, and remove the test-side masking
that let this phase ship green over it.

**The defect.** `app/db/schema.sql:342-349` re-adds the obsolete 3-column
`uq_email_run_purpose_round`, guarded only on its own absence and never on the presence of its
4-column successor `uq_email_run_purpose_round_epoch`. The block at `:358-376` immediately drops it
again. That is harmless churn until the database holds two `email_messages` rows sharing
`(run_id, purpose, round)` and differing only in `epoch` — at which point the ADD is unsatisfiable,
raises `UniqueViolation`, and aborts the **entire** schema apply. Because the fatal block sits at
`:348` and the `failure_category` CHECK repair sits at `:508` in the same file, the repair never
runs either.

That row shape is not exotic. `app/db/schema.sql:265-273` documents it as the whole reason `epoch`
exists: a retrigger resets `clarification_round` to 0, so the retriggered send carries the same
`(run_id, purpose, round)` tuple as the stale pre-retrigger row.

**Confirmed reproducible with no test framework involved:**
```
$ python -m app.db.bootstrap          # the exact command deploy-migrate.yml:53 runs
psycopg.errors.UniqueViolation: could not create unique index "uq_email_run_purpose_round"
DETAIL:  Key (run_id, purpose, round)=(2222…, confirmation, 1) is duplicated.
CONTEXT:  ALTER TABLE email_messages
          ADD CONSTRAINT uq_email_run_purpose_round UNIQUE (run_id, purpose, round)
```

**Severity: latent but armed.** `deploy-migrate` history is currently all-green, so production has
not yet recorded a retriggered run that creates the duplicate tuple. The first one arms it, and
every deploy migration after that aborts. Retrigger is a first-class operator affordance — the
documented fallback for a stuck run.

**The masking.** Plan 21-13 (wave 0 of this phase) was told to wrap the schema mutation in
`try/finally` and was explicitly forbidden from touching `app/db/schema.sql`. The executor complied
exactly. Seven tests went green over a live deploy-blocker. That instruction was wrong, and this
plan corrects both halves.

Output: the obsolete block removed, non-reset bootstrap covered by a real regression test for the
first time, and the schema-repair test restored to asserting that the repair actually happened.
</objective>

<critical_context>
**Deleting the obsolete ADD is the fix — verify that claim before relying on it, then rely on it.**
Nothing in application code depends on the 3-column constraint: `insert_email_message`'s `ON CONFLICT`
arbiter is the 4-column `(run_id, purpose, round, epoch)` (`app/db/repo/emails.py:84,246`), and the
only references to the 3-column name anywhere in `app/` are the block that adds it and the block that
drops it. Re-confirm with `grep -rn "uq_email_run_purpose_round" app/ | grep -v _epoch` before editing.

**The migration ladder must still work — this is the one real risk in the edit.** The two blocks form
a chain: `uq_email_run_purpose` (2-col) → `uq_email_run_purpose_round` (3-col) → `uq_email_run_purpose_round_epoch` (4-col).
Deleting the middle ADD must not strand a legacy database. Reason it through and then *prove it*: on a
DB carrying only the 2-column constraint, the first block drops it and adds nothing, the second block
finds no 3-column constraint to drop and adds the 4-column one. Truth #2 requires a test that
actually walks that path, not an argument that it should work.

**Keep the `try/finally` from 21-13.** It is correct as defence-in-depth — a test that mutates schema
should restore it on every path. What was wrong was treating it as the fix. Do not remove it; remove
the *masking*, which is the test tolerating an aborted bootstrap and still passing.

**Do not weaken any assertion to make something pass.** If removing the masking reds a test, that red
is information — report it rather than re-accommodating it.
</critical_context>
