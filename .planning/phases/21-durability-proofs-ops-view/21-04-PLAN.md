<objective>
Promote the existing same-Svix-redelivery proof in place (D-01), give it its PROOF-02 identity
(D-02), close REQUIREMENTS' named vacuity condition as a checked fact rather than a claim, and
execute the falsifying mutation live (D-05).

Purpose: ROADMAP criterion 2 requires exactly one `jobs` row, one run and one email against a
real Postgres, **and** that the same test would fail if dedup were keyed on the RFC `Message-ID`
alone — a value unavailable until after the provider fetch. RESEARCH.md Finding 2 verifies the
incumbent already proves the positive half and that the key (`svix-id`) is available pre-fetch.
The remaining work is identity, making the pre-fetch property an assertion instead of an
argument, and executed falsification.

Output: the PROOF-02 marker on the audited incumbent, a source-level pre-fetch assertion, and
publishable mutation evidence.

**Corrections applied from cross-AI review (Codex, HIGH).** Two, both about honesty of evidence:

1. **The predicted red was wrong.** Replacing the event identity with a per-delivery value makes the
   two concurrent deliveries *both* insert, so both responses come back `"accepted"` — and the test
   reds first at the response-status-set assertion
   (`tests/test_webhook_dedup_race.py:256-260`, `== {"accepted", "duplicate"}`), well before the
   `inbound_events` count assertion an earlier draft named. The named expected red below is
   corrected accordingly.
2. **The mutation is a stand-in, and the plan must say so where it matters.** A per-delivery
   identity proves the dedup key is *unstable*; it does not prove the criterion's specific premise —
   that dedup is not keyed on the RFC `Message-ID`, a value unavailable until after the provider
   fetch. That premise is structural (the handler cannot read the message at that point), so it is
   pinned **structurally**, by an AST/dataflow guard, and the behavioural mutation is kept as a
   separate stability falsification. Two different claims, two different instruments.
</objective>
