<objective>
Repair a regression introduced by this phase's own wave 0, and close the test-quality gap it exposed.

Plan 21-12 added an autouse fixture stubbing `DATABASE_URL` with a sentinel DSN whenever none is
set, which correctly fixed 16 `ValidationError` failures. But 19 other tests — which had been
passing and fast — began attempting real pool connections against that sentinel. Each attempt waits
out `ConnectionPool(timeout=5)` (`app/db/supabase.py:76`), so the hermetic suite went from **62s to
182s**. Measured A/B on one test: `test_run_detail_inflight_poll_reloads_on_settle` went
**0.03s → 10.08s**.

The latency is the symptom. The finding underneath it is the reason this plan exists:

**These 19 tests never had a working database in the hermetic suite, and they pass anyway.** Before
the stub, `get_settings()` raised instantly, the route fell to its error path, and the assertions
held. After the stub, the route waits 5s per attempt, falls to the *same* error path, and the same
assertions hold. A test named `test_run_detail_inflight_run_renders_200_not_500` that returns 200
because the DB was unreachable is not proving what its name claims.

That is precisely the failure mode this phase exists to eliminate — a green result that proves
nothing — so it gets fixed here rather than logged.

Output: connection attempts that fail fast and loudly instead of slowly and silently; 19 tests
asserting against real faked data; and a guard so this cannot recur unobserved.
</objective>

<critical_context>
**Fix the guard before the tests.** Task 1 (fail-fast) restores CI time on its own and converts
every remaining instance of this problem from "slow and silent" into "loud". Do it first so task 2
has a detector: any test you have not yet hardened will fail immediately and tell you so, instead of
quietly passing. If task 2 turns out larger than expected, task 1 alone still leaves the tree better
than it found it.

**Do not fix the latency by raising or removing the sentinel.** Deleting the stub reintroduces
21-12's 16 `ValidationError` failures. Pointing it at a real database makes the hermetic suite
non-hermetic. The stub is correct in kind; what is wrong is that reaching the pool at all is silent.

**Do not fix a slow test by asserting less.** Several of these assert bounded/absent content
(`test_run_detail_never_renders_raw_error_detail`, `test_retry_exhausted_diagnostics_are_bounded_across_html_and_polling`).
Those pass trivially on an error page that renders none of the content in question. Making them fast
while leaving them on the error path would preserve the exact defect this plan is closing. Truth #2
requires each hardened test to demonstrably red on wrong data.

**The fake-repo registration trap is live in this plan.** `tests/test_threading.py` is one of the
three files holding hard-coded monkeypatch name tuples. If you add or rebind any `repo.*` name, its
NAME STRING must appear in all three inventories (`conftest.py` ~:1015, `test_threading.py` ~:346
and ~:427). A miss fails **silently** — conftest's `hasattr` guard lets the real function run
against a `FakeCursor` and the write disappears. Gate any such change with
`grep -c` proving exactly two occurrences in `test_threading.py`.
</critical_context>

     A short timeout still burns time per attempt and still yields a degraded pass; an immediate

     addition to the loud failure, not instead of it.
  3. Do not change `app/db/supabase.py`'s production default. Any seam you add must be inert when a
     real `DATABASE_URL` is set — the live-DB suite still opens real pooled connections.
  4. Update 21-12's now-inaccurate comment at `tests/conftest.py` (~:160) claiming the sentinel
     "fails loudly (connection refused/DNS error) if anything ever actually tries to connect through
     it". That was measured false; make the comment describe what the code now actually does.
  </steps>
