<objective>
Build the read-only data layer `/ops` and the alarm endpoint both consume: four queue-metric
queries on `app/db/repo/jobs.py` (D-12) and the D-13 swallowing-bug predicate on
`app/db/repo/job_settlement.py`.

Purpose: OPS-01's literal REQUIREMENTS predicate ("job success ≈100% while
`payroll_runs.status='error'` count is nonzero") is a **false-positive generator** after Phase
18's D-16 made "job `done` + run `error`" the normal, correct shape of a legitimately-classified
terminal failure. CONTEXT.md D-13 supersedes it: fire on runs in `error` with **no corresponding
terminal/dead job settlement** — an error state no job ever claimed responsibility for. This
plan designs that predicate explicitly rather than leaving money-adjacent SQL to improvisation,
and proves it against a real Postgres on both sides: silent on every legitimate settlement,
firing on every unaccounted one.

Output: five new repository functions, their facade exports, their fake-repo pairing, hermetic
SQL-shape tests, and a live-Postgres behavioural proof of the alarm predicate.

**Correction applied from cross-AI review (Codex, HIGH — a genuine design tension, resolved
deliberately).** An earlier draft correlated the settlement with the error using
`jobs.updated_at >= payroll_runs.updated_at`. The `>=` was chosen to absorb same-transaction
settlements, and it does — but it also creates a **false negative** on a real, reachable sequence:
`record_run_error()` (`app/db/repo/runs.py`) can drive a run to `error` on its own; a previously
leased pipeline job that then loses its forward status CAS returns `OK`
(`app/queue/handlers/pipeline.py`) and is subsequently settled `done` (`app/db/repo/job_settlement.py`)
with a **strictly later** `updated_at` — which under `>=` silently vouches for the earlier,
genuinely unaccounted error.

**For an alarm, a false negative is the worse failure.** A swallowed error that stays swallowed is
exactly the pathology D-13 exists to surface; a false positive merely makes an operator look. The
predicate below therefore correlates by **equality**, which is both stricter and more truthful:
a legitimate settlement writes the run's error and the job's terminal state in ONE transaction, and
Postgres evaluates `now()` at transaction start, so those two timestamps are *identical* — not
merely ordered. Equality asserts "same event"; `>=` only asserted "not earlier".
</objective>

    The docstring must carry the design argument, because the correctness of this query is not
    self-evident from its shape:

    - **Why it is silent on a legitimate terminal failure.** Every legitimate settlement writes
      the run's `error` status and the settling job's terminal state inside ONE transaction.
      Postgres evaluates `now()` as the transaction start time, so both rows receive the identical
      timestamp and the equality holds exactly. Phase 18's D-16 shape therefore never fires — which
      is precisely the false-positive REQUIREMENTS' literal ratio predicate would have generated on
      every correctly-handled terminal failure.
    - **Why the comparison is temporal and not merely existential.** A run typically already has
      an older completed job (the pipeline job that carried it to `approved`). A bare
      `NOT EXISTS (job in done/dead)` would let that stale success vouch for a later, unrelated
      error and the alarm would never fire at all. Correlating the settlement with the error
      transition is the whole discriminating power of the query.
    - **Why equality rather than `>=`, in both directions.** An *earlier* terminal job is a stale
      success and must not vouch — both forms handle that. A *strictly later* terminal job is the
      case that separates them, and it is a real reachable sequence, not a hypothetical: an
      independently-errored run followed by a CAS-losing job settled `done`. `>=` suppresses the
      alarm there; equality still fires. Note plainly that this trades a possible false positive
      for the elimination of a false negative, and that for an alarm this is the correct direction:
      a false positive costs an operator a look, a false negative is the swallowing bug persisting
      undetected, which is the entire thing D-13 exists to surface.
    - **`settle_background_terminal()`.** It writes a run error with **no job at all** and has no
      production caller today. Classify it explicitly: a run errored through it is reported, which
      is correct under D-13's definition — no job took responsibility for that error. State that
      this is a deliberate classification of a currently-dormant path, so a future caller inherits
      a decided semantics rather than an accident.

      route's delivery error boundary sets a run to `error` through `record_run_error` while its

      settles, so the run is reported. This is a **true positive**, not a bug in the predicate: it
      is by definition an error the transport never recorded, which is exactly D-13's target. Do
      not widen the predicate to silence it.
