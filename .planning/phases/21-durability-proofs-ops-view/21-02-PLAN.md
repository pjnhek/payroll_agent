<objective>
Build the read-only data layer `/ops` and the alarm endpoint both consume: four queue-metric
queries on `app/db/repo/jobs.py` (D-12) and the D-13 swallowing-bug predicate on
`app/db/repo/job_settlement.py`.

Purpose: OPS-01's literal REQUIREMENTS predicate ("job success ≈100% while
`payroll_runs.status='error'` count is nonzero") is a **false-positive generator** after Phase
18's D-16 made "job `done` + run `error`" the normal, correct shape of a legitimately-classified
terminal failure. CONTEXT.md D-13 supersedes it: fire on runs in `error` with **no corresponding
terminal/dead job settlement** — an error state no job ever claimed responsibility for. This
plan designs that predicate explicitly rather than leaving money-adjacent SQL to improvisation,
and proves it against a real Postgres on both sides: silent on every legitimate settlement,
firing on every unaccounted one.

Output: five new repository functions, their facade exports, their fake-repo pairing, hermetic
SQL-shape tests, and a live-Postgres behavioural proof of the alarm predicate.
</objective>

    Add `list_unaccounted_error_runs(limit: int = 50, conn=None) -> list[dict[str, Any]]` to

    The predicate is an anti-join, not a ratio. Select from `payroll_runs` where `status = 'error'`
    and `NOT EXISTS` a `jobs` row with the same `run_id`, whose `state` is in `('done','dead')`,
    **and whose `updated_at` is greater than or equal to the run's `updated_at`**. Order by the
    run's `updated_at` descending and bound by `limit`. Project only `id`, `error_reason`, and
    `updated_at` — `error_reason` is a bounded code, and `error_detail` is deliberately excluded
    from this projection because the alarm needs to make a problem findable, not reproduce a
    diagnostic (the operator follows the run-detail link for that, per D-10).

    The docstring must carry the design argument, because the correctness of this query is not
    self-evident from its shape:

    - **Why it is silent on a legitimate terminal failure.** Every legitimate settlement writes
      the run's `error` status and the settling job's terminal state inside ONE transaction.
      Postgres evaluates `now()` as the transaction start time, so both rows receive the identical
      timestamp and the `>=` comparison holds by equality. Phase 18's D-16 shape therefore never
      fires — which is precisely the false-positive REQUIREMENTS' literal ratio predicate would
      have generated on every correctly-handled terminal failure.
    - **Why the comparison is temporal and not merely existential.** A run typically already has
      an older completed job (the pipeline job that carried it to `approved`). A bare
      `NOT EXISTS (job in done/dead)` would let that stale success vouch for a later, unrelated
      error and the alarm would never fire at all. Requiring the settlement to be at-or-after the
      error transition is the whole discriminating power of the query.
    - **A known, reachable production path this predicate correctly fires on.** The approve
      route's delivery error boundary sets a run to `error` through `record_run_error` while its
      enclosing transaction — including any send job it had enqueued — has rolled back. No job
      settles, so the run is reported. This is a **true positive**, not a bug in the predicate: it
      is by definition an error the transport never recorded, which is exactly D-13's target. Do
      not widen the predicate to silence it.
    - **No mute, no acknowledge, no time-boxed auto-clear** (D-16). The result is purely derived;
      when the run is retriggered or settled the query returns empty and the alarm goes quiet on
      its own. Deliberately record that a lookback window was considered and rejected, because a
      window is a time-boxed auto-clear by another name.

    Create `tests/test_ops_alarm_predicate.py` as a live-Postgres module. Mark it
    `pytest.mark.integration` **and** `pytest.mark.queueproof` (module-level `pytestmark`, mirroring
