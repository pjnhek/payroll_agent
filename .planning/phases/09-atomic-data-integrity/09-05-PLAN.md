**Disposition (per this replan's scope analysis, consistent with
`planner_authority_limits` and `scope_reduction_prohibition`):** the finding is
real and verified against live source (see below), but it is a MONEY-class
correctness gap in the resume/clarification-reply extraction path — the same
family as Phase 7.5's field-regression work (CR-01/R2 findings) — NOT an
atomicity/concurrency/recovery gap. Phase 9's goal (per ROADMAP.md) is "the data
layer becomes correct under concurrency and crashes"; this finding is about
extraction-context completeness across MULTIPLE clarification rounds, which is
a distinct correctness dimension requiring its own disposition decision (accumulate
reply bodies vs. diff-against-last-persisted-extraction vs. defer) that the
`09-CONTEXT.md`/`09-RESEARCH.md` locked-decision set never scoped or budgeted for.
Per the review's own instruction ("Consider scope carefully... option (a)/(b) may
belong in a follow-up phase, but the FIXTURE (c) is cheap and fits Phase 9's
test-heavy shape"), this plan implements disposition (c): an explicit deferred
known-edge, recorded in 09-CONTEXT.md's Deferred Ideas, PLUS a cheap integration
fixture that proves the CURRENT behavior — landing in Phase 9's test set as
directed, so a future MONEY-class phase has a concrete, already-written
regression target instead of re-discovering and re-tracing the bug from scratch.

**Verified against live source (this session), the exact chain 09-REVIEWS.md's
Claude in-session review traced:**
  1. `clean_body` strips quoted reply history (">"-lines, "On … wrote:" blocks) at
     ingest, before persisting (`app/email/clean.py:35-60`) — so thread quoting
     cannot preserve intermediate replies once the NEXT reply arrives.
  2. `repo.load_source_email` (`app/db/repo.py:279-296`) returns ONLY the
     ingest-time ORIGINAL cleaned body — never updated with any reply's content.
  3. `_combined_context_email` (`app/pipeline/orchestrator.py:772-785`) builds the
     resume extraction context as ORIGINAL body + the CURRENT/LATEST reply's
     `body_text` only — verified: `combined_body = "ORIGINAL PAYROLL
     EMAIL:\n{original_body}\n\nCLARIFICATION REPLY FROM CLIENT:\n{reply.body_text}"`
     — an intermediate reply from an EARLIER round never accumulates into this
     context on a LATER round.
  4. `detect_field_regression` (`app/pipeline/validate.py`) only fires on a
     paid→unpaid (dropped) transition — a paid→paid VALUE CHANGE (e.g. 40→30) is
     invisible to it by design (it detects field DROPS, not value corrections).
  5. The Round-2 classify-first logic (`resume_pipeline`, `orchestrator.py`) only
     reclassifies fields marked `asked`; a field the client corrected in Round-1
     but that was never the SUBJECT of a Round-2 clarification question is not
     re-examined.

  Concrete failure the fixture reproduces: Round-1 reply says "Maria worked 30,
  not 40" (extraction persists 30 as the paid value for `hours_regular`). A
  field-regression clarification fires Round 2 for an UNRELATED field (e.g. a
  dropped `hours_overtime`); the Round-2 reply answers ONLY that unrelated
  question. Round-2's combined extraction re-reads the ORIGINAL body (via
  `_combined_context_email`) → `hours_regular=40` again (the original value,
  Round-1's correction is not in this context) → 40 is paid (not None) → no
  backfill triggers, `hours_regular` was never `asked` in Round 2 → no override
  fires, 40 vs 40 (both extractions of the SAME original text) → no regression
  detected. The final paystub pays 40; the client said 30 in Round 1. Silent
  overpay class, no gate, no clarification, no operator visibility.

Output: `tests/test_resume_pipeline.py` gains a dedicated fixture test proving
this IS the current, documented behavior (a red-flag regression target, not a
"this is fine" green check); `09-CONTEXT.md`'s Deferred Ideas section gains an
explicit entry recording the finding, its verified chain, and the three
candidate fix dispositions ((a) accumulate reply bodies, (b) diff against
last-persisted extraction, (c) this fixture) for whichever future phase picks it
up.
