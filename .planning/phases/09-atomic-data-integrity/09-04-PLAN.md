(defaults to `None`, meaning "no client-side timeout override," i.e. the raw
10-minute library default with no app-level retry wrapping it at all) — this gap
was NOT closed by tightening `call_structured` alone, since `compose_clarification`
never goes through `call_structured`; (3) `app/pipeline/orchestrator.py:377,380`
(verified this session): `resume_pipeline`'s Round-2 branch calls `extract(inbound,
...)` then, two lines later, `extract(combined_email, ...)` — TWO separate
`call_structured`-backed extraction calls back-to-back, BEFORE the next DB write
— so the real worst-case gap on THAT specific path is `_STRUCTURED_TIMEOUT_S × 2
app-attempts × 2 extractions` = 4× the single-call ceiling, not 2×. This plan
corrects all three: tightens `call_structured` (closing both the extraction AND
suggestion gaps at once), adds an explicit `timeout_s=` to
`compose_clarification`'s `call_text` invocation (closing the previously-wholly-
unbounded clarification-drafting gap), and re-derives the D-9-13 threshold against
the ACTUAL longest real sequence (Round-2's double extraction), not a
single-call assumption.

**Revision note 2 (09-REVIEWS.md Codex Round-2 — STILL-OPEN HIGH, the sole NO-GO
blocker):** re-verified against live source this session, `app/llm/client.py`'s
`call_text` (lines 182-187) builds its `OpenAI(...)` client with ONLY a
conditional `timeout=` kwarg (`client_kwargs["timeout"] = timeout_s` when
`timeout_s is not None`) — there is NO `max_retries` kwarg anywhere in `call_text`,
unlike this plan's fix to `call_structured`. The `openai` library (2.43.0, pinned)

`for attempt in (1, 2):`), the library's own retry layer is the ONLY retry layer
on this call path — so even after this plan's Revision-1 fix (adding
`timeout_s=` to `compose_clarification`'s call), the true worst case remains
`timeout_s × 3` (library attempts), not `timeout_s × 1` as Revision 1 assumed.
This is a DIFFERENT compounding shape than `call_structured`'s (which stacks
library-retries × app-retries); for `call_text` the fix is simply to ALSO pass
`max_retries=0` on `call_text`'s own client construction — for symmetry with
`call_structured` and because it is the simpler, exact fix (Codex's own stated
preference over re-deriving the threshold as `timeout_s × 3`). This plan now
adds `max_retries=0` unconditionally to `call_text`'s `OpenAI(...)` construction
(not just when `timeout_s` is provided — the two kwargs are independent: a caller
may want retry-suppression even without a custom timeout) and updates the
threshold derivation, the acceptance criteria, and the test asserting
`call_text`'s client-construction kwargs to cover BOTH call paths
(`call_structured` AND `call_text`).

retry layer (`max_retries=0`), tightens `call_text`'s client construction the SAME
