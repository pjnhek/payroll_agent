<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Resend (external) → `POST /webhook/inbound` | Untrusted network input; the dedup/ingest transaction is the mechanism that guarantees exactly-once run-creation even under adversarial/duplicate/racing delivery. |
| Unauthenticated `GET /runs` → sweep UPDATE | The sweep runs on every dashboard load with no additional auth check (matches the existing route's auth posture, unchanged by this plan) — its blast radius is bounded entirely by the hardcoded scope list built in 09-01. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-09 | Tampering (duplicate financial run from retried delivery) | `inbound()` dedup+create_run transaction | mitigate | Postgres UNIQUE constraint (`uq_message_id`) + `ON CONFLICT DO NOTHING` inside ONE transaction (D-9-09) — verified Postgres MVCC blocking behavior (RESEARCH.md), not an app-level lock; proven by Task 2's real-thread concurrency test. |
| T-09-10 | Denial of Service (orphaned email row with no run, un-repairable) | `inbound()` ingest transaction | mitigate | The whole dedup+routing+create_run sequence is now one transaction — a crash mid-ingest rolls back the email row itself, so Resend's retry starts clean and creates the run on the next attempt. No repair/adoption path needed (D-9-09 dissolves rather than patches the orphan window). |
| T-09-11 | Repudiation (stuck run invisible to the operator) | `runs_list()` sweep hook | mitigate | Every dashboard load sweeps stranded runs to a visible, diagnosable ERROR state with a distinguishing `error_reason` sentinel (D-9-10/11), closing MED-05's "no visible signal" gap. |
| T-09-12 | Denial of Service (sweep threshold too aggressive, live task swept mid-flight) | Shared `STALE_THRESHOLD_SECONDS` constant | accept (documented tension) | D-9-13's known accepted tension: a pathologically-slow-but-alive task could be swept to ERROR and later have its unguarded `set_status` silently un-sweep it. This plan documents the current true worst case (≈30 min, RESEARCH.md Pitfall 1) in the code comment rather than picking a value that pretends the risk doesn't exist; Phase 10's concurrency proof is the designated venue for any further guard-hardening evidence. |
</threat_model>
