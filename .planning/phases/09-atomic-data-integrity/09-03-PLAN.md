**Revision note (09-REVIEWS.md Codex HIGH-1 — reply-routing break):** the prior
draft of this plan wrapped `insert_inbound_email + find_business_by_sender +
create_run` in one transaction while leaving `_route_reply` (the header-chain
reply classifier, which calls `find_awaiting_reply_for_header` /
`find_any_run_for_header`) to run AFTER that transaction had already decided to
create a run. Verified against live `app/main.py`: `_route_reply` currently runs
at line 336-339, BETWEEN the dedup insert (line 311-320, today an independent
auto-commit) and the sender-check + `create_run` (line 342-350). If the new
transaction spans dedup-insert-through-create_run as a single unconditional
block, a header-bearing clarification reply would never get a chance to be
classified as a reply BEFORE `create_run` runs inside that same block — it would
silently create a second, spurious run for what should have been a resume. This
plan corrects the design: the reply-classification READS
(`find_awaiting_reply_for_header`, `find_any_run_for_header`) move INSIDE the
ingest transaction, executed IMMEDIATELY after a successful dedup insert and
BEFORE any code path that could reach `create_run`. The transaction commits one
of exactly four outcomes — duplicate / reply-resume / reply-late (no run) /
unknown-sender (no run) / new-run — and only AFTER that commit does the route
schedule the corresponding background task (`_resume_pipeline` OR `_run_pipeline`,
never both, never inside the `with` block).

DATA-02: wrap the webhook's dedup-insert + reply-classification + sender-routing +
`create_run` sequence in one transaction, committed BEFORE
`background_tasks.add_task` is ever scheduled — closing the orphan window where a
crash mid-ingest could leave an email row with no run, and making the two-concurrent-
duplicates race resolve to exactly one run via Postgres's own MVCC blocking behavior
on the `uq_message_id` unique index (no new locking primitive). DATA-03: derive the
corrected threshold (informed by RESEARCH.md's Pitfall 1 finding that the naive
90s-3min estimate under-counts the real LLM worst-case) and wire
`repo.sweep_stranded_runs` (built in 09-01) into the `GET /runs` dashboard route —
the one HTTP entry point Render's free tier guarantees will be hit periodically.

Purpose: close audit finding HIGH-04 (webhook dedup race) and MED-05 (stuck-run
recovery, 5-minute threshold too long with no visible signal) — the two remaining
DATA requirements not covered by 09-02's orchestrator work — WITHOUT introducing
the reply-routing regression Codex's review caught.

Output: `app/main.py`'s `inbound()` route (restructured around a transactional
ingest-decision block that classifies duplicate/reply/unknown-sender/new-run
INSIDE the transaction) and `runs_list()` route wired per D-9-09/D-9-11; a shared
`STALE_THRESHOLD_SECONDS` constant sized against the corrected LLM worst-case;
`tests/test_webhook_dedup_race.py` proving SC2 via real concurrent threads;
`tests/test_webhook.py` gains a regression test proving a clarification reply
never creates a second run under the new transaction.
