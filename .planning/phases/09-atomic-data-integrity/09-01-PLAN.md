<objective>
Land the two new repo-layer primitives Phase 9 needs (`sweep_stranded_runs` for
DATA-03, `find_run_by_source_email` for DATA-02's dedup-loser lookup) and — the
prerequisite every other Phase 9 plan depends on — make `app.db.repo.get_connection`
mockable inside the existing `fake_repo` test fixture. Per D-9-03, the mechanism
is exactly the existing `conn=` seam (~30 repo helpers already accept it via
`_conn_ctx`) — no new abstraction, no ORM, no unit-of-work class. Every subsequent
plan in this phase wraps orchestrator/webhook code in `with repo.get_connection()
as conn: with conn.transaction(): ...`; today NOTHING in the offline test suite exercises
`get_connection`, so without this plan the very first `with repo.get_connection()`
added anywhere would make every `fake_repo`-driven test in the suite try to open a
real Supabase pool and fail/hang.

Purpose: give Wave 2 (orchestrator DATA-01 wiring) and Wave 2 (main.py DATA-02/03
wiring) a shared, already-tested foundation so neither plan has to invent its own
mocking strategy for the connection seam, and so DATA-03's sweep function exists
and is unit-proven before main.py's dashboard hook (09-03) wires it in.

Output: `repo.sweep_stranded_runs`, `repo.find_run_by_source_email`, an updated
`fake_repo` fixture that patches `app.db.repo.get_connection` to return a
`FakeConnection` (so orchestrator/main.py code that does
`with repo.get_connection() as conn: with conn.transaction(): ...` runs against the
existing offline double, exactly like every other repo call in the mocked suite),
and the SQL-shape/scope unit tests for the sweep (per D-9-12, "this list must be
pinned by a test").
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Dashboard route → sweep UPDATE | The sweep runs on every unauthenticated `GET /runs` load (wired in 09-03); this plan only builds the function, but its blast radius (a single UPDATE with a hardcoded scope list) is assessed here since the function's own correctness IS the control. |
| Test double → app.db.repo.get_connection | The new `fake_repo` monkeypatch changes what "a live connection" means inside the offline suite — a leaky abstraction here (e.g. `FakeConnection.transaction()` silently succeeding on a real rollback path) could mask a real production bug in a LATER plan's test. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | Tampering (TOCTOU) | `sweep_stranded_runs` | mitigate | Single CAS-shaped `UPDATE ... WHERE status = ANY(%s) AND updated_at < ... RETURNING id` — no read-then-write; identical idiom to the already-audited `claim_status`/`record_run_error` WR-03 fix (repo.py:565-586). Pinned by Task 2's SQL-shape test. |
| T-09-02 | Denial of Service (over-broad sweep) | `sweep_stranded_runs` scope parameter | mitigate | D-9-12 scope list is hardcoded to exactly `{received, extracting, computed}` inside the function body (not caller-supplied) and pinned by an explicit unit test asserting the three-element list and the ABSENCE of the three parked statuses — a future edit that widens the scope (e.g. adds `approved`) fails this test immediately. |
| T-09-03 | Repudiation (untraceable sweep) | `sweep_stranded_runs` error_detail | accept | The sentinel `error_reason="StrandedRunSwept"` + a fixed, non-PII `error_detail` string makes every swept run distinguishable from a real exception-driven ERROR in the dashboard/DB — no roster/email content is ever included (unlike `record_run_error`'s `detail_exc` path), so there is no PII-leak surface to mitigate here. |
| T-09-04 | Tampering (test-double drift) | `tests/conftest.py::fake_repo` `get_connection` patch | accept | `FakeConnection.transaction()` is already documented (RESEARCH.md Pitfall 3, existing code comment) as a no-op that cannot prove real rollback semantics — this plan does not change that fact, it only makes the seam reachable offline. SC1/SC2's genuine atomicity proof is explicitly deferred to `@pytest.mark.integration` tests against a real/local Postgres in 09-04, per the existing project-wide convention. |
</threat_model>
