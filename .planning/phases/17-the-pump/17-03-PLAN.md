<objective>
Fold keepalive.yml into a new `pump.yml` (D-06/07/08, criterion #4) and document the cadence math in the
README (criterion #3, PUMP-02). `pump.yml` is a single job on a 30-minute cron (+ workflow_dispatch for
GitHub's 60-day auto-disable) with three `curl -f` steps: (1) the authenticated `/internal/pump` drain,
(2) `/health/ready` (wake + Supabase touch), (3) `/health/schema` (drift → RED). keepalive.yml is deleted;
its RENDER_URL secret is reused; a new PUMP_TOKEN GitHub secret + render.yaml `sync:false` entry are added.

The criterion #4 trap: carrying only the wake ping would silently drop the `/health/schema` drift monitor
— the only thing that catches a manual Supabase edit bypassing deploy-migrate.yml. Both keepalive jobs
carry over, verbatim including their `--max-time 90`. Only the NEW pump step gets a larger `--max-time 420`
(RESEARCH Pitfall 2), documented HONESTLY (review finding #1 — the derivation was wrong twice, so the METHOD
changed): the CORRECTNESS guarantee is lease-reclaim, NOT the curl budget. `lease_seconds = 900`
(app/config.py:117) means any job a pump request does not finish — because it hit `--max-time`, Render's
undocumented server-side ceiling, or cold-start ate the budget — is held by its lease and idempotently
re-run by the next 30-min cadence (SKIP LOCKED), so the exact `--max-time` is decoupled from data safety.
The 420 accounting names its terms: Render cold-start ≤60s (the curl spans it BEFORE the request reaches the
app — keepalive.yml:44), + 17-04's between-jobs cap 120s (checked BETWEEN drain_once() calls, so a request
can begin ONE final job just under it), + a worst-case single job's ≈240s EXTERNAL-CALL ALLOWANCE (the SUM
of the explicit provider timeouts — extraction/suggestion ≤90s each, clarification draft ≤30s, Resend send
≤30s), with deterministic compute/DB/network overhead ON TOP. So 60 + 120 + 240 already = 420 before
overhead: the earlier "~60s headroom" claim is WITHDRAWN as false, 420 is a NOMINAL operating budget (NOT a
proven containment bound), and a rare worst-case-clarification request can overrun it → a RED that is
acceptable-and-documented (the job is lease-reclaimed next cadence). The 420 is PROVISIONAL until the manual
live-smoke confirms Render's undocumented server-side ceiling (RESEARCH Open Question #2 / A1). A 90s timeout
would false-RED even a routine pump. It is NOT sized from the 210s inter-write STALL gap (a stall threshold,
not a job runtime; review finding #2). The workflow also gets a
workflow-level `concurrency` group so a scheduled and a manual pump run cannot overlap (a long drain must not
race a second invocation) — SKIP LOCKED already protects correctness; serialization protects capacity.

Purpose: turn durable storage into durable execution via an external cron, and document why 30 minutes.
Output: pump.yml (+ concurrency), keepalive.yml deleted, render.yaml PUMP_TOKEN, README cadence block,
tests/test_pump_workflow.py static guard.
</objective>
