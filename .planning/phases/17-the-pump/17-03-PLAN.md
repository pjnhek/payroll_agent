<objective>
Fold keepalive.yml into a new `pump.yml` (D-06/07/08, criterion #4) and document the cadence math in the
README (criterion #3, PUMP-02). `pump.yml` is a single job on a 30-minute cron (+ workflow_dispatch for
GitHub's 60-day auto-disable) with three `curl -f` steps: (1) the authenticated `/internal/pump` drain,
(2) `/health/ready` (wake + Supabase touch), (3) `/health/schema` (drift → RED). keepalive.yml is deleted;
its RENDER_URL secret is reused; a new PUMP_TOKEN GitHub secret + render.yaml `sync:false` entry are added.

The criterion #4 trap: carrying only the wake ping would silently drop the `/health/schema` drift monitor
— the only thing that catches a manual Supabase edit bypassing deploy-migrate.yml. Both keepalive jobs
carry over, verbatim including their `--max-time 90`. Only the NEW pump step gets a larger `--max-time 360`
(RESEARCH Pitfall 2): worst-case single-job runtime is 210s and the pump's wall-clock cap (120s, checked
between jobs) means a pump request can legitimately run ~330s; a 90s timeout would false-RED a working pump.

Purpose: turn durable storage into durable execution via an external cron, and document why 30 minutes.
Output: pump.yml, keepalive.yml deleted, render.yaml PUMP_TOKEN, README cadence block.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions cron → Render service | The cron carries the PUMP_TOKEN Bearer secret over the network; the secret must never live in the repo, the URL, or step output. |
| repo file → committed secret | A `value:` (not `sync:false`) entry, or a hardcoded token in pump.yml, would ship a live secret to a public portfolio repo. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-17-08 | Information Disclosure | PUMP_TOKEN provisioning | high | mitigate | PUMP_TOKEN is a render.yaml `sync:false` entry (never committed) and a GitHub Actions repo secret referenced as `${{ secrets.PUMP_TOKEN }}`; it appears in the URL header, not the query string, so it stays out of access logs and Actions step output. Never a `value:` entry. |
| T-17-09 | Repudiation / Monitoring-loss | keepalive fold-in | high | mitigate | Deleting keepalive.yml while dropping the /health/schema curl would silently remove the only live drift monitor. pump.yml carries the /health/schema step forward, verified by the `grep -q '/health/schema'` gate and the 3-curl count. |
| T-17-10 | Denial of Service (false RED) | pump step curl timeout | medium | mitigate | The pump step's `--max-time 360` is sized above the ~330s worst case; copying the health steps' 90s would false-RED a still-working pump and get misread as data loss (RESEARCH Pitfall 2). |
| T-17-SC | Tampering | package installs | low | accept | Zero new packages this phase; the only dev-time addition is `yaml.safe_load` in a verify step, which uses already-installed PyYAML. |
