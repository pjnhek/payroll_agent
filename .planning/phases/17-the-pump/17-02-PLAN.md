<objective>
Add the point-in-time queue-depth read the pump reports as `queue_depth` (D-04, criterion #1's fifth
count). `count_open_jobs()` is a plain `SELECT count(*) FROM jobs WHERE state IN ('pending','leased')`
following jobs.py's uniform `_conn_ctx` read convention. It is deliberately backlog-scoped (total
outstanding), not "claimable right now" — the useful ops signal is outstanding depth. Two easy-to-miss
companions land in the same commit: the module docstring's "Six functions…" line becomes false (update to
seven + enumerate the new name), and the facade re-export in `app/db/repo/__init__.py`.

Review corrections (17-02 MEDIUM): adding a seventh function silently invalidates THREE
`test_repo_jobs_sql.py` tests that still assert a six-function public surface (`:193` the f-string SQL
guard, `:228` the facade-exports check, `:242` the conn/`_conn_ctx` inventory) — they would stay green while
falsely claiming to cover the whole surface. All three move to seven and include `count_open_jobs` in this
plan. And the hermetic count test is scoped HONESTLY: `test_repo_jobs_sql.py` is a `FakeConnection`
SQL-recording suite, so it can prove the return→`int` mapping and the exact `state IN ('pending', 'leased')`
WHERE text, but NOT that Postgres actually counts pending/leased vs done/dead. The behavioral mixed-state
proof is live-Postgres and rides in 17-05.

Schema-gate note: this project has no ORM; the schema is hand-maintained `app/db/schema.sql` applied via
the existing deploy-migrate path. This plan adds only a read — NO schema-push task is needed or created.

Purpose: give the pump a truthful backlog number without reusing a composition snapshot as per-run counts.
Output: `count_open_jobs`, re-exported and hermetically tested.
</objective>
