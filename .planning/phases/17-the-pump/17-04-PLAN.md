<objective>
Add the authenticated pump endpoint (PUMP-01, criterion #1). `GET /internal/pump` is a thin, sync `def`
route modeled on app/routes/health.py: it authenticates with a constant-time Bearer compare
(hmac.compare_digest, D-01) that fails closed on an unset/empty PUMP_TOKEN (D-03) and returns 401 on any
bad/missing credential (D-02, deliberately not 404). It then loops the SAME `drain_once()` the worker
threads call (PUMP-01 — never a fork), bounded by a max-jobs cap AND a wall-clock cap checked between jobs
(D-05), aggregating each DrainOutcome into real counts, and returns
`{claimed, done, retried, dead, fenced, queue_depth}` (queue_depth via repo.count_open_jobs). Business
outcomes (dead-letter, backoff) return 200 + counts (D-09). Any exception mid-drain surfaces as 503 (D-10):
a DB outage on claim/count, OR — critically for D-10 — a propagated `drain_once()` exception (17-01's
double-failure re-raise: `fail_job`'s own write failed during an outage). The catch-all is honest, not
narrow: an unexpected programming error would also 503; in normal operation only infra failures reach it.

Discretion calls (documented at the call site): method is GET (simplest for a curl cron; the drain is
idempotent — SKIP LOCKED makes repeat/concurrent hits safe); env var is PUMP_TOKEN / setting pump_token;
caps are `_MAX_JOBS_PER_PUMP = 20` and `_MAX_WALL_CLOCK_SECONDS = 120` (the between-jobs cap). HONEST timeout
METHOD (review finding #1 — the derivation was wrong twice, so the METHOD changed, not just the number): the
CORRECTNESS guarantee is lease-reclaim, NOT the curl budget. `lease_seconds = 900` (app/config.py:117) means
any job a pump request does not finish — because it hit `--max-time`, Render's undocumented server-side
ceiling, or cold-start ate the budget — is NOT lost: its lease holds it and the next 30-min cadence reclaims
and idempotently re-runs it (SKIP LOCKED claim + at-most-once status CAS + send_guard). This DECOUPLES the
exact `--max-time` from data safety. The numbers are named terms, not a point estimate: Render cold-start
≤60s (curl spans it BEFORE the request reaches the app, keepalive.yml:44); the between-jobs cap is 120s
(checked BETWEEN drain_once() calls, so a request can begin ONE final job just under it); one worst-case
single job's EXTERNAL-CALL ALLOWANCE ≈240s — the SUM of the explicit provider wall-clock timeouts on the
longest clarification path: extraction ≤90s (`_STRUCTURED_TIMEOUT_S` 45 × 2 app-attempts, app/llm/client.py:47-71),
suggestion ≤90s (same call surface), clarification draft ≤30s (`_CLARIFICATION_TIMEOUT_S`, compose_email.py),
Resend send ≤30s (resend client default) — an external-call ALLOWANCE, explicitly NOT total job runtime,
with deterministic compute + DB + SDK/network overhead ON TOP. So a single request's worst case
≈ 60 + 120 + 240 + overhead, which CAN exceed `--max-time 420`: the earlier "~360s ceiling + ~60s headroom"
claim is WITHDRAWN as false (60 + 120 + 240 already = 420, before overhead). `--max-time 420` is a NOMINAL
operating budget (a deliberately tighter-than-full-worst-case ceiling), PROVISIONAL until the manual
live-smoke confirms Render's undocumented server-side request-duration ceiling (RESEARCH Open Question #2 /
A1); a rare worst-case-clarification request that overruns it goes RED, but that RED is acceptable-and-documented
because the job is lease-reclaimed next cadence — it signals "a slow run, auto-retried", not "lost work". It
is NOT derived from the 210s inter-write stall gap (that is a stall threshold, review finding #2). queue_depth
is a separate cheap SELECT, not part of the claim transaction.

Purpose: the pump is the PRIMARY execution trigger — it must drain with zero live worker threads.
Output: /internal/pump route, pump_token setting, router wiring, hermetic auth/bounded/infra tests.
</objective>
