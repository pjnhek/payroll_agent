<objective>
Build the phase's single most important test (criterion #2, PUMP-01) — the anti-vacuous-proof anchor. It
constructs exactly the state the roadmap names: a job scheduled for later (future `available_at`), on an
instance with NO live worker threads, and proves the pump — hit via the HTTP endpoint cron actually calls —
executes it. This project's history is dominated by vacuous proofs (a "concurrency proof" that passed while
proving nothing); this test is deliberately engineered to be non-vacuous and is demonstrated able to fail.

The suite already pins `WORKER_COUNT=0` (tests/conftest.py), so every TestClient runs cold-started by
construction — the state criterion #2 wants is the default, not elaborate staging. The design (RESEARCH.md
§ "Designing the anti-vacuous-proof anchor", reused verbatim) makes the three vacuity traps impossible:
(a) not requesting `live_worker` (no worker races ahead), (b) asserting the job is genuinely not-claimable
while future-dated (the claim query isn't trivially satisfied), (c) asserting claimed==1 + the specific
outcome + the row reaching state='done' by its own id (not merely a 200).

Purpose: prove the pump, not the workers, is the durable-execution guarantee — and prove the proof can fail.
Output: one @pytest.mark.queueproof test in tests/test_queue_durability.py + the falsifying-mutation record.
</objective>
