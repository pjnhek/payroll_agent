<objective>
Wave 3 — Add the interactive self-serve demo as the hero surface, with two architectural fixes that make Path-1 (in-app composer) genuinely no-SMTP and decouple operator binding from the canonical seed contacts (D-06 / CHANGE-5 through CHANGE-11 + review fixes HIGH-1, HIGH-2, and three MEDIUMs).

HIGH-1 FIX (record-only delivery sink): Add a `record_only` boolean column to `payroll_runs`. Compose-created runs set this flag. The orchestrator checks it at each `gateway.send_outbound` call site (_clarify and _deliver): if True, skip the Resend provider call and write the outbound `email_messages` row directly via `repo.insert_email_message` (synthetic Message-ID, send_state='sent'). The outbound row still exists — so the thread view and simulate-reply seam work identically to Path-2 runs. Path-2 runs (record_only=False, the default) keep using the real gateway.send_outbound call without change.

HIGH-2 FIX (decouple operator binding from businesses.contact_email): Two independent sub-fixes:
(a) POST /demo/compose routes by allowlisted business_id directly (from the _SEED_BUSINESS_IDS constant, keyed by validated business_name). No find_business_by_sender call — compose never touches businesses.contact_email or the demo binding. Two concurrent recruiter composes for different businesses route independently with no shared mutable state.
(b) POST /demo/bind writes to a new demo_sender_bindings table (operator_email TEXT PK, business_id UUID FK, bound_at TIMESTAMPTZ) via an UPSERT. businesses.contact_email is NEVER mutated by /demo/bind. find_business_by_sender gains a second additive lookup: if contact_email match returns None, check demo_sender_bindings for operator_email → business_id. Path-2 real-email inbound routes via this additive lookup. Seed .example contacts are permanently stable.

Consequence for simulate-reply: Because seed contacts are never mutated, load_inbound_email(run_id) returns the seed .example from_addr for compose runs. find_business_by_sender still resolves it correctly (seed contact row is untouched). The FIX-5 spoof guard in _route_reply passes for simulate-reply without any special-casing.

Also delivers: landing page (GET /) with recruiter composer and no bind form; /demo/bind as unlinked operator route; extended thread view including the source inbound; alias-rationale notes; in-page reply seam; body/subject length caps on compose.

Purpose: A recruiter can complete the full payroll demo entirely in-app (Path 1), without real email or SMTP. The operator does one real round-trip (Path 2) via direct URL /demo/bind + their Gmail, proven once for the recruiter video. Both paths share the identical thread view and run detail UI.

Output: schema.sql (demo_sender_bindings + payroll_runs.record_only), repo.py (new helpers, patched find_business_by_sender), orchestrator.py (record-only branch at each send_outbound call site), main.py (3 new routes, extended run_detail), index.html (new), run_detail.html (extended), tests/test_demo_landing.py.
