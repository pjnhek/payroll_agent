Purpose: Every security/correctness requirement (D-13 dedup, D-17 signature verify, D-01a two-step, D-14 threading, D-13c reserved-before-send) converges here. This plan turns the RED/XFAIL test stubs from 06-01 GREEN (and REMOVES the xfail markers) while keeping the existing 422-test mocked suite green (the no-op-swap invariant).

HIGH-2 fix (Codex review — most important correctness fix): parse_inbound and the route's verify step must PRESERVE the fixture/dev path. The current dev path POSTs a canonical InboundEmail JSON dict directly to /webhook/inbound — this is used by the existing ~422 tests AND the DASH-05 demo button. If every POST triggers verify() + Resend-only two-step parse, the fixture path breaks. Fix: shape-detect in the route — a payload with `data.email_id` in its Resend envelope AND svix-* headers present is a real Resend webhook and goes through verify() → two-step parse. A payload that matches the canonical InboundEmail shape (no svix-* headers, or `ALLOW_UNSIGNED_FIXTURES=true` dev setting) skips verify() and goes through the canonical passthrough. D-17 is NOT weakened: a Resend-envelope payload with bad/absent sig still returns 400.

HIGH-3 fix (Codex review): send_outbound wraps resend.Emails.send in try/except. On exception, it flips the reserved row to send_state='failed' and re-raises into the existing ERROR/retrigger path — mirroring the Phase-5 D-13b delivery error boundary. Runs must reach ERROR, not strand in 'reserved'.

HIGH-4 fix (Codex review): The dedup short-circuit is made explicit. After parse_inbound yields the RFC message_id, the route calls repo.insert_inbound_email (ON CONFLICT (message_id) DO NOTHING RETURNING id). Background pipeline task is enqueued ONLY if a row was inserted (returned id). A duplicate (no returned id) returns 200 and does NOT enqueue/re-run decide.py. "Rest of route unchanged" is not enough — this must be spelled out in the action.

MEDIUM-5 fix (from 06-01): Remove the @pytest.mark.xfail(strict=True) markers from all gateway tests as they go GREEN in this plan. Tests that XPASS with strict=True cause the suite to exit non-zero; removing the markers as they pass is the required step.
