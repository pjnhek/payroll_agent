<objective>
Wave 3: Wire the real Resend provider behind the gateway seam. This is the single file (gateway.py) the project's provider-agnostic design always intended — only gateway.py changes for provider-specific logic (D-18). The /webhook/inbound route in main.py gains structural changes: raw Request → shape detection → conditional verify → parse → explicit dedup short-circuit → existing flow.

Purpose: Every security/correctness requirement (D-13 dedup, D-17 signature verify, D-01a two-step, D-14 threading, D-13c reserved-before-send) converges here. This plan turns the RED/XFAIL test stubs from 06-01 GREEN (and REMOVES the xfail markers in Task 3) while keeping the existing 422-test mocked suite green (the no-op-swap invariant).

HIGH-1 decision (Round-3 HIGH — RESOLVED here, WAIVE provider_message_id): The email_messages table in app/db/schema.sql has ONLY these columns: id, run_id, direction, message_id, in_reply_to, references_header, subject, from_addr, to_addr, body_text, purpose, send_state, created_at. There is NO provider_message_id column and NO updated_at column on email_messages. The prior plan text that wrote `updated_at=now()` and `provider_message_id` in UPDATE helpers would fail at runtime against the live schema. COMMITTED RESOLUTION: WAIVE provider_message_id persistence entirely. The synthetic message_id is already the sole routing anchor and works without provider_message_id. repo.update_email_message_sent SQL: `UPDATE email_messages SET send_state='sent' WHERE message_id=%s`. repo.update_email_message_state SQL: `UPDATE email_messages SET send_state=%s WHERE message_id=%s`. Resend's returned provider id is logged only (not persisted). No schema change needed — schema.sql is in files_modified only to make the confirmation grep executable in the done criteria.

HIGH-1-AUTH fix (Round-5 HIGH — live-demo-critical): resend.api_key must be set before EVERY resend SDK call — both inbound and outbound. The /demo/send-test path invokes send_outbound WITHOUT first running parse_inbound (no inbound webhook fires during a demo button press). If resend.api_key is only set inside the Resend parse_inbound path, any outbound send that did not go through inbound parse (e.g., the demo button, a clarification send on an already-parsed run) will call resend.Emails.send with an unset key and fail live. Fix: set `resend.api_key = get_settings().resend_api_key` as the FIRST line of send_outbound — before minting the synthetic id, before any DB write, before resend.Emails.send. This is idempotent and cheap. The parse_inbound Resend path also sets it (defensive), but send_outbound MUST set it independently. The module-init approach (set at import time) was considered but rejected: Settings reads env vars at instantiation and the lru_cache on get_settings() means a test that monkeypatches env vars AFTER import would not see the updated key — per-function setting is safer for testing.

HIGH-2 fix (Codex review — most important correctness fix): parse_inbound and the route's verify step must PRESERVE the fixture/dev path. The current dev path POSTs a canonical InboundEmail JSON dict directly to /webhook/inbound — this is used by the existing ~422 tests AND the DASH-05 demo button. If every POST triggers verify() + Resend-only two-step parse, the fixture path breaks. Fix: shape-detect in the route — a payload with `data.email_id` in its Resend envelope AND svix-* headers present is a real Resend webhook and goes through verify() → two-step parse. A payload that matches the canonical InboundEmail shape (no svix-* headers, or `ALLOW_UNSIGNED_FIXTURES=true` dev setting) skips verify() and goes through the canonical passthrough. D-17 is NOT weakened: a Resend-envelope payload with bad/absent sig still returns 400.

HIGH-4 fix: In prod (ALLOW_UNSIGNED_FIXTURES=False, default), ALL unsigned POSTs to /webhook/inbound return 400, including canonical-shape POSTs. Only when ALLOW_UNSIGNED_FIXTURES=True (dev/test only, never in render.yaml) do unsigned POSTs succeed. This closes the canonical-bypass hole. The canonical fixture path is intact in dev mode; it is not exempt from auth in prod.

MEDIUM-4 fix: This plan contains NO statements that "canonical POSTs work unsigned" in production. The ONLY consistent statement: unsigned → 400 in prod (ALLOW_UNSIGNED_FIXTURES=False); unsigned → 200 in dev mode (ALLOW_UNSIGNED_FIXTURES=True). Every task, success criterion, and threat-model entry is consistent with this.

MEDIUM-5 fix: Task 0 (Wave 0, pre-implementation) adds a tiny SDK smoke-check test that asserts resend.Webhooks.verify, resend.EmailsReceiving.get, and resend.Emails.send (accepting headers= and attachments=) exist on the installed resend 2.32.2 module. This catches a Python SDK naming mismatch at Wave 0, not at the live human gate.

HIGH-3 fix: (a) send_outbound wraps resend.Emails.send in try/except. On exception, it flips the reserved row to send_state='failed' via repo.update_email_message_state and re-raises. (b) send_outbound maps `attachments: list[tuple[str, bytes]] | None` into resend.Emails.send as `[{"filename": name, "content": base64.b64encode(pdf_bytes).decode()}]`.

LOW-7 note: Resend outbound idempotency key is a v2 hardening (not built now). Adding `idempotency_key` to the resend.Emails.send call would reduce duplicate-send risk after "provider sent, DB update failed" crashes. Documented here for the record; not implemented in this phase.

Output: A fully wired Resend gateway that handles both the real two-step inbound shape AND the canonical fixture shape; verifies signatures only for Resend-envelope payloads; dedups explicitly before enqueuing; rebuilds the full References chain from persisted DB state before every send; sends outbound with D-13c crash-safe ordering including the failed-state on exception. The existing pipeline (extract, reconcile, decide, deliver) is entirely unchanged.
