<objective>
Wave 3: Wire the real Resend provider behind the gateway seam. This is the single file (gateway.py) the project's provider-agnostic design always intended — only gateway.py changes for provider-specific logic (D-18). The /webhook/inbound route in main.py gains structural changes: raw Request → shape detection → conditional verify → parse → explicit dedup short-circuit → existing flow.

Purpose: Every security/correctness requirement (D-13 dedup, D-17 signature verify, D-01a two-step, D-14 threading, D-13c reserved-before-send) converges here. This plan turns the RED/XFAIL test stubs from 06-01 GREEN (and REMOVES the xfail markers in Task 3) while keeping the existing 422-test mocked suite green (the no-op-swap invariant).

HIGH-2 fix (Codex review — most important correctness fix): parse_inbound and the route's verify step must PRESERVE the fixture/dev path. The current dev path POSTs a canonical InboundEmail JSON dict directly to /webhook/inbound — this is used by the existing ~422 tests AND the DASH-05 demo button. If every POST triggers verify() + Resend-only two-step parse, the fixture path breaks. Fix: shape-detect in the route — a payload with `data.email_id` in its Resend envelope AND svix-* headers present is a real Resend webhook and goes through verify() → two-step parse. A payload that matches the canonical InboundEmail shape (no svix-* headers, or `ALLOW_UNSIGNED_FIXTURES=true` dev setting) skips verify() and goes through the canonical passthrough. D-17 is NOT weakened: a Resend-envelope payload with bad/absent sig still returns 400.

HIGH-1 fix (Round-2 Codex review — message_id identity): The "set-our-own-Message-ID" design is the committed approach. send_outbound mints a synthetic RFC Message-ID (`<uuid@_OUTBOUND_DOMAIN>`), passes it to `resend.Emails.send` as `headers={"Message-ID": synthetic_id}` so the real outbound email carries the app-minted ID, stores that same synthetic id in `email_messages.message_id` (the column reply routing matches on), and stores Resend's returned `response["id"]` separately as a provider audit value. A5 (RESEARCH.md assumption: "Resend lets you set the Message-ID header") is promoted from "discover at 06-05" to a DESIGN ASSUMPTION this plan commits to. 06-05 verifies the assumption survived: the operator's reply In-Reply-To header must equal the synthetic message_id stored in the DB (not Resend's internal id). If A5 is wrong (Resend overwrites the header), the fallback is to store Resend's returned RFC Message-ID in `email_messages.message_id` and update reply routing accordingly — but that fallback is a 06-05 observation, not a code path in this plan.

HIGH-3 fix (Codex review — two parts): (a) send_outbound wraps resend.Emails.send in try/except. On exception, it flips the reserved row to send_state='failed' and re-raises into the existing ERROR/retrigger path — mirroring the Phase-5 D-13b delivery error boundary. Runs must reach ERROR, not strand in 'reserved'. (b) send_outbound maps the `attachments: list[tuple[str, bytes]] | None` parameter into `resend.Emails.send` as `[{"filename": name, "content": base64.b64encode(pdf_bytes).decode()}]` — paystub PDFs that `_deliver` passes through the gateway interface must not be silently dropped. A test `test_send_outbound_forwards_attachments` asserts the attachments kwarg passed to the mocked `resend.Emails.send` contains the PDF filename and correct base64 content.

HIGH-4 fix (Codex review — extended): The canonical InboundEmail fixture shape is NOT exempt from authentication in production. In prod (ALLOW_UNSIGNED_FIXTURES=False, the default), ANY unsigned POST to /webhook/inbound returns 400 — including canonical-shape POSTs. Only when ALLOW_UNSIGNED_FIXTURES=True (explicitly set in dev/test env, never in render.yaml) do unsigned fixture POSTs succeed. Tests that use canonical fixture POSTs must monkeypatch `get_settings().allow_unsigned_fixtures = True` or set the env var in the test scope. The `/demo/send-test` DASH-05 button path must also enable this flag in the dev/staging environment (or be converted to sign its own payloads — dev flag is simpler).

HIGH-4b fix (Codex review): The dedup short-circuit is made explicit. After parse_inbound yields the RFC message_id, the route calls repo.insert_inbound_email (ON CONFLICT (message_id) DO NOTHING RETURNING id).
