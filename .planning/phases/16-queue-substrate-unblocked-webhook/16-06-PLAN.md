<objective>
Build `app/queue/` — the execution layer between the durable rows (16-04) and the worker threads
(16-07): the in-process wake signal (D-09), the kind-to-handler dispatch table, the `run_pipeline`
handler carrying **D-01's reclaim rewind** and **INVARIANT J-1's CAS**, and `drain_once()` — the one
function the worker threads and the Phase-17 pump will both call.

**The hardest thing here is D-01, and it is money-adjacent.** The trap: an operator retriggers, the
CAS lands the run at `RECEIVED`, worker W1 claims the job and drives the run to `EXTRACTING`/
`COMPUTED` — and then W1's process dies. The lease expires, W2 reclaims the job and calls the handler
against a run that is no longer at `RECEIVED`. Under a naive reading of J-1 ("a failed CAS is a DONE
job") the job goes `done` and **the run is stranded forever** — precisely what ROADMAP criterion #2
forbids. The discriminator is `attempts`, which QUEUE-02 already increments AT CLAIM, so the signal is
free: `attempts > 1` means "the previous me died halfway" (rewind, re-run); `attempts == 1` means
"someone else legitimately advanced this" (drop cleanly — strict J-1, preserved where it applies).

Purpose: turn a durable row into durable work, without ever letting the transport tier decide what
payroll status comes next.
Output: the `app/queue/` package and its hermetic proofs.

## INVARIANT J-1 — the PRECISE restatement (resolves a blocking review finding)

> **⚠ The old wording was self-contradictory and the cross-AI review was right to block on it.**
> It said the handler's "first durable action" must be `claim_status(RECEIVED → EXTRACTING)` **and**
> that "only the CAS may advance business state" — while ALSO requiring `rewind_for_reclaim()` to run
> **first** when `attempts > 1`. But `rewind_for_reclaim` writes `payroll_runs.status`. So it is a
> second business-state writer, and it runs *before* the "first" durable action. As written, 16-06's
> own "first durable action" test and its "only the CAS advances status" test **could not both pass.**

**J-1, restated. This is the version that goes in the code, in `handle_run_pipeline`'s docstring, and
in the guard test's docstring:**

> `payroll_runs.status` has exactly **TWO** permitted writers reachable from the queue tier, and both
> are **compare-and-swap** writes — a conditional `UPDATE ... WHERE status IN (...) RETURNING id`
> whose `False`/`None` return means *"someone else owns this; drop cleanly."* There is no third
> writer, and there is no unconditional `set_status` from `app/queue/`.
>
> 1. **`claim_status(expected → next)` — the sole FORWARD transition.** A lost forward CAS is a
>    **DONE job, not a retry.** The transport tier never decides what payroll status comes next; it
>    only asks, and accepts *no* for an answer.
>
> 2. **`rewind_for_reclaim(run_id)` — the sole, explicitly-named RECOVERY transition.** It is
>    **not a forward transition**: it moves *backward* (`extracting|computed|sent → received`), it is
>    itself a CAS (so it is exclusive and cannot clobber a run outside its three-status scope), it is
>    permitted **only when `job.attempts > 1`** (i.e. only on a genuine crash-reclaim), and it never
>    bumps `reply_epoch` (D-02).

**Therefore "first durable action" is a *conditional*, and both tests are satisfiable:**
- `attempts == 1` → first durable action is `claim_status`. (Strict J-1, unchanged.)
- `attempts > 1` → first durable action is `rewind_for_reclaim`, which is itself a CAS; `claim_status`
  follows immediately.

Both branches obey the single invariant that actually matters: **every business-status write from the
transport tier is a CAS, and the tier drops cleanly whenever it loses one.** Task 3 turns that into a
static guard (`test_queue_tier_status_writers_are_cas_only`) whose permitted-writer set is exactly
`{claim_status, rewind_for_reclaim}` — so a future third writer, or any `repo.set_status` call from
`app/queue/`, fails CI **by name**.

`drain_once() -> bool` — returns whether a job was claimed:
1. `job = repo.claim_job()` (lease_seconds from settings). `None` -> return False.
2. Register `job.lease_token` in a module-level, `threading.Lock`-guarded held-token set.
3. `try: dispatch.handle(job)` then `repo.complete_job(job.id, job.lease_token)`.
   `except Exception:` -> `repo.fail_job(job.id, job.lease_token, error=exc, backoff_seconds=_backoff_seconds(job.attempts))`.
4. `finally:` discard the token from the held set.
5. Return True.

Both `complete_job` and `fail_job` are FENCED. A `False`/`None` return means the lease was stolen and
this worker is a zombie: **log and drop.** Do not retry, do not error the run, do not re-enqueue.

(16-07) calls `repo.release_leases(...)` with it, which is what makes a redeploy hand work back in
