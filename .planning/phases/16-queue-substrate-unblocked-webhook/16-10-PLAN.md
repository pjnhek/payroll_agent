## The defect (traced against live source, CONFIRMED — do not re-litigate, fix it)

`gateway.send_outbound` is crash-safe in *intent*, but its guards are not:

1. `gateway.py:271-289` — reserve an outbound row with a **freshly-minted synthetic `message_id`**,
   `send_state='reserved'`. This INSERT owns its own transaction, so it **commits** here.
2. `gateway.py:339-345` — call the provider.
3. `gateway.py:355-359` — **only now** flip the row to `'sent'`.

Every existing duplicate guard counts **only `send_state='sent'`**:
- `repo.get_outbound_message_id` (`emails.py:140-171`) — delivery's confirmation guard.
- `repo.get_outbound_for_round` (`emails.py:174-218`) — clarify's round guard.

**A worker killed between step 2 and step 3 leaves NO `'sent'` row while the client already has the
email.** The lease expires, another worker reclaims the job, `rewind_for_reclaim` rewinds the run
(D-01), the pipeline re-runs, the guard sees nothing — and the client is emailed a **second time**,
under a **brand-new `message_id`**. Because that synthetic id is the *sole* reply-routing anchor,
this does not merely double-email: it **orphans the client's reply into a phantom run**.

`'failed'` is the same hazard wearing a different label. `send_outbound` flips to `'failed'` on **any**
Resend exception — including a **timeout after the mail was accepted**. REQUIREMENTS.md **SEND-01**
says so in as many words: *"`failed` … is not proof of non-delivery."*

**Why it must land HERE.** Phase 16 ships live daemon workers (16-07) AND lease-expiry reclaim
(ROADMAP criterion #3; 16-09 is literally *"a retrigger survives a worker death and completes on the
next drain"*). **The reclaim path is live in THIS phase.** Deferring the fix to Phase 20 means wave 4
is the commit that opens the window.

## The rule: FAIL-CLOSED

> An **unconfirmed** outbound row — `send_state IN ('reserved','failed')` — for this run's
> `(run_id, purpose, round, epoch)` send slot means **the message may already have reached the
> provider.** The pipeline **does not send again.** It **escalates to the operator.**

`(run_id, purpose, round, epoch)` is not an arbitrary key — it is **exactly** the `ON CONFLICT`
arbiter of `insert_email_message` (`emails.py:84`) and exactly the columns of
`uq_email_run_purpose_round_epoch` (`schema.sql:279`). It IS the identity of "the send slot this
message would occupy." Use it; do not invent a different one.

## Why ERROR is the escalation state (and why that is not a cop-out)

The guard **raises**. The pipeline's existing catch-all boundary (`orchestrator._run`, and `approve()`
for the delivery path) persists the exception TYPE as `error_reason` and lands the run in **`ERROR`**.
That gives, for free, four properties a bespoke mechanism would have had to build:

1. **Operator-visible today** — `ERROR` already renders on the run page with `error_reason` /
   `error_detail`. No UI work, which D-10 forbids in this phase anyway.
2. **The machine can never auto-retry it.** `ERROR` is deliberately **outside**
   `rewind_for_reclaim`'s three-status scope (`extracting`/`computed`/`sent`, see 16-04). A reclaim
   hits the rewind, gets `False`, loses the forward CAS, and marks the job `done`. **The escalated run
   stays put.** That is the whole point.
3. **Only a HUMAN can move it** — via Retrigger, whose `ERROR → RECEIVED` CAS **bumps `reply_epoch`**
   (`clear_reply_context`). A new epoch is a new send slot, so this epoch-scoped guard permits the
   send. **That is precisely REQUIREMENTS.md's "Accepted residual risk"**: *an operator retrigger can
   legitimately send a second email.* A human may take that risk. The reclaim loop may not.
4. **Zero new vocabulary** — no new `RunStatus` member, so no SQL CHECK change, no drift-guard churn,
   no `jobs.kind` collision surface (criterion #5).
