Public surface (exactly these six; no more):

- `enqueue_job(*, kind: JobKind, dedup_key: str, run_id=None, email_id=None, business_id=None, max_attempts: int | None = None, conn=None) -> uuid.UUID | None`
  INSERT with `ON CONFLICT (dedup_key) DO NOTHING RETURNING id`. **A `None` return means "already
  enqueued" ‚Äî return it, do not raise.** `max_attempts` defaults to `get_settings().max_attempts`
  when None, so the `MAX_ATTEMPTS` env knob (D-08) is actually live rather than being shadowed by the
  column default. Do NOT set `event_id` ‚Äî that column does not exist until Phase 19.

- `claim_job(*, lease_seconds: int | None = None, conn=None) -> Job | None`
  The canonical single-statement claim, transcribed verbatim. Three properties you must not lose:
  (a) `FOR UPDATE SKIP LOCKED` lives in the **subquery** and the outer UPDATE re-targets by `id` ‚Äî a
  bare `UPDATE ... LIMIT 1` is not valid Postgres and `FOR UPDATE` on the outer statement gives no
  row-skipping; (b) the WHERE includes **`OR (c.state = 'leased' AND c.leased_until < now())`** ‚Äî
  ARCHITECTURE ¬ß4 calls this "the single most important line in this document", and without it a job
  whose worker died holding the lease stays `leased` forever, which is the exact failure this
  milestone exists to eliminate; (c) `attempts = j.attempts + 1` increments **at claim, not at
  failure** ‚Äî that is what bounds a crash loop where the worker is SIGKILLed before it can report
  anything, and it means the completion path must never also increment.
  **`RETURNING` yields EXACTLY SIX columns ‚Äî `j.id, j.kind, j.run_id, j.attempts, j.max_attempts,
  j.lease_token` ‚Äî and nothing else.** No `email_id`, no `event_id`. This matches the `Job` dataclass
  16-03 declares, field-for-field, in order.
  > **‚öÝ This was the review's #1 blocking finding: an earlier draft had `Job` carrying SEVEN fields
  > (including `email_id`) while telling you to transcribe a SIX-column `RETURNING` verbatim. Those
  > are mutually exclusive. SIX wins** ‚Äî see 16-03 Task 1 for the full rationale. The `jobs.email_id`
  > COLUMN still exists on the table (reserved for Phase 19); it is simply not selected here and not
  > carried on the transport record. Phase 19 adds the column to `RETURNING` and the field to `Job`
  > **together**. Task 1's `tests/test_repo_jobs_sql.py` asserts that bijection so it cannot drift.
  `lease_seconds` defaults to `get_settings().lease_seconds`.
  One statement => one implicit transaction, committing before any real work ‚Äî no session state, no
  `LISTEN/NOTIFY`, no session advisory locks (all fail SILENTLY under Supavisor transaction-mode
