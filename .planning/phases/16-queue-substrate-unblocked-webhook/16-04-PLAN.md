Create `app/db/repo/jobs.py` as a new aggregate module in the existing per-aggregate repo package.
**Every function takes `conn: psycopg.Connection | None = None`** and opens with the package's
`_conn_ctx(conn)` / `_nulltx()` convention. This is the single most load-bearing pattern in the
phase: it is what lets `enqueue_job(..., conn=conn)` drop straight into the retrigger route's
transaction (plan 16-08) with zero new plumbing.

Public surface (exactly these six; no more):

- `enqueue_job(*, kind: JobKind, dedup_key: str, run_id=None, email_id=None, business_id=None, max_attempts: int | None = None, conn=None) -> uuid.UUID | None`
  INSERT with `ON CONFLICT (dedup_key) DO NOTHING RETURNING id`. **A `None` return means "already
  enqueued" ā€” return it, do not raise.** `max_attempts` defaults to `get_settings().max_attempts`
  when None, so the `MAX_ATTEMPTS` env knob (D-08) is actually live rather than being shadowed by the
  column default. Do NOT set `event_id` ā€” that column does not exist until Phase 19.

- `claim_job(*, lease_seconds: int | None = None, conn=None) -> Job | None`
  The canonical single-statement claim, transcribed verbatim. Three properties you must not lose:
  (a) `FOR UPDATE SKIP LOCKED` lives in the **subquery** and the outer UPDATE re-targets by `id` ā€” a
  bare `UPDATE ... LIMIT 1` is not valid Postgres and `FOR UPDATE` on the outer statement gives no
  row-skipping; (b) the WHERE includes **`OR (c.state = 'leased' AND c.leased_until < now())`** ā€”
  ARCHITECTURE Ā§4 calls this "the single most important line in this document", and without it a job
  whose worker died holding the lease stays `leased` forever, which is the exact failure this
  milestone exists to eliminate; (c) `attempts = j.attempts + 1` increments **at claim, not at
  failure** ā€” that is what bounds a crash loop where the worker is SIGKILLed before it can report
  anything, and it means the completion path must never also increment.
  `RETURNING` must yield exactly the `Job` dataclass's fields (no `event_id`).
  `lease_seconds` defaults to `get_settings().lease_seconds`.
  One statement => one implicit transaction, committing before any real work ā€” no session state, no
  `LISTEN/NOTIFY`, no session advisory locks (all fail SILENTLY under Supavisor transaction-mode
  pooling; see `app/db/supabase.py:1-18`).

- `complete_job(job_id: uuid.UUID, lease_token: uuid.UUID, conn=None) -> bool`
  Fenced: `WHERE id = %s AND state = 'leased' AND lease_token = %s RETURNING id`. A `None` row means
  **the lease was stolen and this worker is a zombie** ā€” return False. The caller must not retry, not
  error the run, not re-enqueue: it logs and drops. This is the identical contract to `claim_status`
  returning False; write that comparison into the docstring.

- `fail_job(job_id, lease_token, *, error: BaseException | str, backoff_seconds: float, conn=None) -> JobState | None`

  `SET state = CASE WHEN attempts >= max_attempts THEN 'dead' ELSE 'pending' END`, push

  parsed `JobState`, or `None` when fenced out.

   ('extracting','computed','sent') RETURNING id`. Return `row is not None`.

   **D-02 ā€” the reason this function exists at all, and it must be the first line of its docstring:
   it MUST NOT bump `reply_epoch`.** `clear_reply_context`'s bump is a licence to send the client a
   second email. Granting an *operator* that licence is a reviewed decision. Granting *the machine*
   that licence ā€” every time a lease expires ā€” would mean the milestone's headline claim ("at most one
   confirmation per approved run, **per epoch**") is no longer enforceable by the DB constraint that
   backs it. Because the epoch stays put, delivery's already-sent guard
   (`get_outbound_message_id(purpose='confirmation')`) still matches and genuinely suppresses the
   second confirmation on a rewound-and-re-run run. That is the whole safety argument. Write it down.

   **The status scope is exactly three values ā€” `extracting`, `computed`, `sent` ā€” and the choice is
   load-bearing.** Those three mean "a worker was mid-pipeline and never reached a resting point."
   Document why every other status is EXCLUDED:
   - `received` ā€” nothing to rewind; the handler's own `RECEIVED -> EXTRACTING` CAS will simply succeed.
   - `awaiting_reply` / `needs_operator` / `awaiting_approval` ā€” the pipeline reached a LEGITIMATE
     pause. Rewinding one would destroy a live conversation or a pending human decision.
   - `reconciled` / `rejected` / `approved` ā€” terminal or operator-owned. Not ours to rewind.
   - `error` ā€” the pipeline genuinely ran and failed. Auto-re-running it here would be a silent
     automatic retry, which is FAIL-02 (Phase 18) and explicitly out of scope. The run stays visibly
     in ERROR for the operator; the job goes `done`.
   `sent` IS included, for the same reason `retrigger`'s own stale scope includes it: a run stranded
   between `SENT` and `RECONCILED` can be safely re-run because delivery's already-sent guard
   suppresses a second confirmation ā€” and it actually fires here precisely BECAUSE the epoch is
   unchanged.

   A `False` return (run was outside the rewind scope) is not an error ā€” the handler's subsequent
   `RECEIVED -> EXTRACTING` CAS will then fail, and INVARIANT J-1 says a failed CAS is a DONE job, not
   a retry.
