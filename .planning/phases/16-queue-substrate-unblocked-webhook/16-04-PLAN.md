### D-19 — THE DELETE MUST NEVER LAND UNDER A LIVE WORKER. (Cross-AI review round 3, HIGH, CONFIRMED.)

> **⚠ This is the fix-broke-something case, and it is exactly the class the fixture above exists to
> close — so getting it wrong here would be ironic and fatal.** The round-2 fix made `_isolated_jobs`
> autouse and function-scoped. Round 3 found what that collides with: **16-07's live worker proofs run
> REAL daemon threads, and Proof 4 deliberately constructs a handler blocked past a timed-out join** —
> a straggler that OUTLIVES the test body on purpose. That is not an accident in Proof 4; it is Proof
> 4's entire subject.
>
> Now let an assertion fail — or a hard timeout fire — before the test releases and joins that worker.
> Pytest unwinds the body. `_isolated_jobs`' teardown runs `DELETE FROM jobs` **while the straggler is
> still inside its handler holding a leased row.** Three things follow, all bad:
>
> 1. The worker's `complete_job(id, token)` now fences against **a row that does not exist**. It
>    returns `False` for the wrong reason. **A proof that asserts "the zombie was fenced out" would
>    still PASS** — against a mechanism that never ran. That is a lying proof, and it is worse than a
>    failing one.
> 2. On a test that failed, the forensic state is destroyed before anyone can read it — a real
>    diagnosis becomes noise.
> 3. **Worst case, and the one round 3 did not name:** if the abort happened *before* `worker.stop()`,
>    the straggler's generation stop event was **never set**. Its blocked handler eventually times out,
>    the loop continues, and it calls `claim_job()` again — a rogue, un-stopped daemon worker, now
>    eating the jobs of every subsequent test in the module. The isolation fixture would be deleting
>    rows on one side while a forgotten worker claims them on the other.
>
> **The rule: teardown may destroy state only when the process is QUIESCENT, and it must FAIL LOUDLY
> rather than delete under a live worker.** A silent delete beneath a running worker is precisely the
> class of thing that makes a proof lie, and this repo has shipped that class before.

Two mechanisms, and they are **not** redundant — one prevents, one recovers:

**(1) `_require_quiesced_workers()` — the GATE. Prevents the corrupt delete.**
A module-level plain function (NOT inlined into the fixture, so it can be driven directly by a test):
if `live_queue_worker_threads()` (conftest, Task 2 edit (e)) is non-empty, `pytest.fail(...)` naming
every surviving thread — **and DO NOT DELETE.** `_isolated_jobs` calls it immediately before EACH of
its two `DELETE`s. Its failure message must say, in terms: *"a live queue-worker thread survived this
test; deleting `jobs` beneath it would corrupt the proof rather than isolate it. A test that starts a
real worker MUST use the `live_worker` fixture, whose teardown quiesces the pool before this fixture
runs."* Loud, attributable, and non-destructive — the three properties the round-3 finding demands.

**(2) `live_worker` — the ORDERED QUIESCE. Makes the sanctioned path never trip the gate.**
The ONLY sanctioned way to start a real worker in this module. **It requests `_isolated_jobs`**, and
that dependency edge is the entire mechanism: pytest sets a requested fixture up first and finalizes in
**reverse** order, so teardown runs `live_worker` → `_isolated_jobs`. Quiesce, *then* delete. Always,
including on an aborted body.

```

    blockers: list[threading.Event] = []
    ...
    yield handle                        # handle.start(n), handle.blocker()
```

- `handle.blocker() -> threading.Event` — mints an Event, REGISTERS it with the fixture, and returns
  it. A test that blocks a handler must obtain its Event this way; that registration is what lets the
  fixture release a blocked worker on a path where the test body never got to.
- `handle.start(n=1)` — lazily `from app.queue import worker` (this plan is wave 2; `worker.py` does
  not exist until 16-07 — an import at module top would break this file on this plan's own wave), then
  `worker.start(n)`.
- `_quiesce_workers(blockers, *, grace: float = ...)` — a module-level plain function so 16-07 can
  drive it directly in a test. In order: **set every registered blocker** (an aborted test never got
  to its `release_me.set()`); `worker.stop(grace_seconds=grace)` (idempotent — safe even if the test
  already stopped cleanly); then wait, bounded, for `live_queue_worker_threads()` to empty. **If it
  does not empty inside the bound, `pytest.fail` naming the survivors** — never return quietly, and
  never let `_isolated_jobs` proceed to a delete.

> **⚠ The blocker's OWN `wait()` timeout must be comfortably LONGER than `_quiesce_workers`' join
> budget** (e.g. blocker waits up to 30s; quiesce budgets 5s). Otherwise a mutation that removes the

> against the bug it exists to catch.** State both numbers explicitly in the fixture, and state why.
