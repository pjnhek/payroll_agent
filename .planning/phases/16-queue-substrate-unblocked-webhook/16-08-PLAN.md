(`app/routes/runs.py:266-381`). The lowest-risk, already-manual path. Nothing on the money path moves.

**This is not a one-line `add_task` -> `enqueue_job` swap, and treating it as one is the trap.** Direct
source tracing this session confirmed that `retrigger()` today performs its ERROR/APPROVED CAS, its
stale-in-flight `load_run` + threshold check + second CAS, and its `clear_reply_context` across
**multiple separate, uncoordinated transactions** — every repo call defaults `conn=None` and opens its
own. QUEUE-02's atomicity requirement ("the enqueue lives inside the SAME transaction as the state
change that owes it — the CAS supplies exclusivity, the dedup_key UNIQUE supplies idempotency; you
need both, or you get a lost job or a phantom job") means consolidating all of it into ONE
caller-owned transaction. Budget real work for it.

`BackgroundTasks` and the queue coexist safely for exactly this phase, because each producer uses
exactly one mechanism. The other 7 `add_task` sites are untouched until Phase 19 (QUEUE-04).

Four things that are easy to get wrong and that you must get right:

- **The `dedup_key` MUST carry the epoch.** A naive `run_pipeline:{run_id}` means: run errors ->
  operator retriggers -> job runs, goes `done` -> run errors again -> operator retriggers again ->
  `ON CONFLICT DO NOTHING` against the OLD `done` row -> **the second retrigger silently does
  nothing.** `clear_reply_context` bumps `reply_epoch` on every retrigger and (as of 16-04) RETURNS
  it — that is the discriminator. Note the shape of the bug this also guards against: if
  `clear_reply_context` were to return `None`, every dedup_key would render as the identical string
  ending in `None` and you would reintroduce the swallowed-second-retrigger bug through the back door.
  A test must pin the epoch's presence in the key.

- **`wake.wake()` fires AFTER commit, never inside the transaction (D-09).** Wake inside and the woken
  worker races ahead of the commit, finds no row, and goes back to sleep — Retrigger then feels like
  it takes 20 seconds (the poll interval) instead of being instant. This file already has the
  discipline (`webhook.py`'s "Transaction committed. Everything below is post-commit" comment); follow
  it, and say so in a comment at the call site.

- **The `background_tasks: BackgroundTasks` parameter.** `retrigger` no longer schedules anything into
  process memory. Remove the parameter if nothing else in the function needs it. Do NOT remove
  `BackgroundTasks` from the OTHER route signatures in this file — the other producers stay on
  `add_task` until Phase 19.

- **Rewrite the stale comment.** The current comment at `:376-378` says `clear_reply_context` "opens
  its own committed transaction (conn=None): a durable unit that must NOT span the LLM-heavy
  run_pipeline_bg background task." That statement becomes FALSE. Replace it with the new contract: the
  CAS, the context clear, and the enqueue now commit together, and the LLM-heavy work happens in a
  worker thread against a job row that outlives this process.
