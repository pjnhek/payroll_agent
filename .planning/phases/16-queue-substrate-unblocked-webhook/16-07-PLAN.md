<objective>
Give the queue a running body: 2 daemon worker threads, owned by the app's **first-ever** FastAPI
`lifespan`, that wake instantly on the in-process signal, poll slowly otherwise, and — the part
ROADMAP criterion #4 is actually about — **release every lease they hold the moment the process is
asked to stop.**

Render free spins the service down after 15 minutes of no inbound traffic and redeploys are routine.
Without a graceful release, every in-flight retrigger stalls for the full `LEASE_SECONDS` (15 minutes)
before another worker can touch it. With it, the new instance picks the work up in seconds.

And **D-07**: `WORKER_COUNT + 2 <= max_size (5)` is asserted at boot and the app **refuses to start**
if violated. Not clamp-and-warn. A misconfigured deploy must fail loudly and visibly on Render rather
than booting into a state where every request silently hangs waiting for a connection the workers have
all checked out. Fail-fast on a startup-time config error is the money-system default — and this repo
already sets that precedent (`database_url: str  # no default — fails fast if unset`).

And **D-15** — the round-2 cross-AI review finding, and the one that would have shipped a queue with a
body and no pulse. The previous draft's `stop()` set a stop event that **nothing ever reset**, so the
next `start()` spawned threads that saw it already set and returned on their first iteration. On
Render — where a redeploy is routine and a 15-minute idle spin-down is the default — *every restart*
would have come back with workers that exist and never drain. The fix is one word wide (a **fresh
`threading.Event` per generation**, passed as a thread argument, never `.clear()`ed), which is exactly
why it needs a proof that a **restarted worker completes a real job**. `thread.is_alive()` is GREEN
against this bug; it is the vacuous twin, and it is prohibited here.

> ### ⚠ D-20 — ONE LIFECYCLE LOCK, HELD ACROSS THE WHOLE TRANSITION. NOT PER-FIELD.
>
> **Cross-AI review round 3, MEDIUM.** The previous draft said this state was "`threading.Lock`-guarded"
> and left it there. Guarding individual *field writes* is not the property that is needed and it is
> not what the word "idempotent" was quietly promising. Two `stop()` calls that interleave around the
> join/partition step (below) each read `_threads`, each join, each partition — and the second writes
> back `_threads` / `_orphans` / `_stop` from a **stale snapshot** taken before the first finished.
> `release_leases` is issued twice. A concurrent `start()` observes the pool mid-transition.
>
> **The mechanism:** `_LIFECYCLE_LOCK = threading.Lock()`, acquired for the **ENTIRE BODY** of `start()`
> and the **ENTIRE BODY** of `stop()` — prune, refusal check, generation bump, event mint and spawn
> (start); event set, wake, join, release, partition (stop). The state transition is the critical
> section, not the assignment.
>
> **`stop()`'s concurrency contract, made EXECUTABLE (the review's actual ask — "you already claim
> `stop()` is idempotent, so make that claim executable"):** a second concurrent `stop()` **BLOCKS on
> the lock**, and when it acquires it, finds `_stop is None` and `_threads == []` — the transition is
> already complete — and **returns as a genuine no-op**: it does not re-join, and it does **not** call
> `release_leases` again. So across N concurrent `stop()`s, `release_leases` is issued **exactly once**.
> That is `test_concurrent_stops_release_exactly_once`, and it is a real assertion rather than a claim.
>
> **THE DEADLOCK RULE — and it is the reason this lock is safe to hold across the joins:**
> **`_loop` NEVER acquires `_LIFECYCLE_LOCK`.** It reads its stop Event from its own thread **argument**
> (D-15), and it reads `_generation` as a **plain int load** — atomic under the GIL, and in any case
> only advisory (see the fence note in `_loop` below). Because no worker thread can ever be waiting on
> this lock, `stop()` holding it across `thread.join(...)` cannot deadlock. Nor can the token set:
> `drain.held_tokens()` takes *drain's own* lock, and nothing that holds drain's lock ever reaches for
> this one — so there is no cycle, in either direction.
>
> **Write this rule into the module docstring as a prohibition, not a description.** The single edit
> that reintroduces a deadlock is someone "tidying up" `_loop` to read `_stop`/`_generation` under the
> lock, and it will look like an improvement.

> ### ⚠ D-15 — THE STOP EVENT IS PER-GENERATION. READ THIS BEFORE YOU WRITE `start()`.
>
> **This is the cross-AI review's round-2 HIGH finding, CONFIRMED against the previous draft of this
> plan.** That draft said `_loop` runs "until the stop event is set", and that `stop()` **sets** it —
> and then never un-set it anywhere. So after a clean `start() → stop()`, the next `start()` would
> spawn threads whose **first loop iteration observes an already-set event and returns immediately.**
> Workers exist. Nothing ever drains. A restarted process is silently dead, and the plan's own restart
> test (which asserts a start after the orphan dies *succeeds*) would have "succeeded" at spawning
> corpses.
>
> **The fix is NOT `_stop.clear()` in `start()`.** That is the obvious one-liner and it is wrong:
> clearing a **shared** event **un-stops a straggler** — a worker that was told to stop but has not
> yet reached the loop boundary where it would notice. Task 3's Proof 4 constructs exactly that
> straggler on purpose (a handler blocked past a timed-out join). A `.clear()` would revoke its stop
> instruction and resurrect it on top of the new generation, holding a pooled connection against a
> `max_size=5` budget — **D-07's failure mode arriving through the back door.**
>
> **The mechanism:** `start()` mints a **brand-new `threading.Event()`** for the generation it is
> about to spawn, stores it in `_stop`, and **passes it to each thread as an argument**. A
> generation's event, once set by `stop()`, stays set **forever** — it belongs to that generation and
> to no other. A new generation begins with a fresh, unset object. The straggler can never be
> un-stopped, and the restarted worker can never start pre-stopped. Write this reasoning into the
> module docstring; a future reader WILL try to "simplify" it back to one shared event.

`start(n: int) -> None` — **the entire body runs under `_LIFECYCLE_LOCK` (D-20)**:
- First, **prune** `_orphans` of threads that have since died.
- **Second-start guard (this is a cross-AI review finding).** If `_threads` is non-empty, or any
  `_orphan` is still `.is_alive()`, **raise `RuntimeError`** naming the live count and the current
  generation. Do NOT silently spawn a second set. Fencing keeps the DB writes *safe* — a stale
  worker's `complete_job` is rejected on the lease token — but it does NOT keep the process from
  carrying **2 × WORKER_COUNT** live threads, each holding a pooled connection against a `max_size=5`
  budget. That is D-07's exact failure mode arriving through the back door, and D-07's whole posture
  is fail-fast, not clamp-and-hope. Refusing is consistent.
- `n <= 0` -> log "queue workers disabled" and return WITHOUT starting anything. This is the
  test/dev off switch (`WORKER_COUNT=0`, D-06/D-08) and it must be a genuine no-op, not a
  zero-length loop that still allocates state. (In particular: do NOT mint a stop Event on this path.)
- Otherwise: `_generation += 1`, capture it as `gen`; **`stop_evt = threading.Event()` — a FRESH
  object, never a `.clear()` of the old one (D-15)** — assign `_stop = stop_evt`; and spawn `n`
  `threading.Thread(target=_loop, args=(gen, stop_evt), daemon=True, name=f"queue-worker-{gen}-{i}")`.
  Each thread thus **closes over its own generation's event**, so it can never be signalled — or
  un-signalled — by another generation's lifecycle.
  Daemon threads, NOT a `ThreadPoolExecutor`: N self-driving claim loops need no dispatcher, and an
  executor adds a submission thread for zero safety gain. And NOT `asyncio.Task`s: every downstream
  pipeline call (LLM via `openai`, psycopg, Resend) is blocking synchronous I/O, so an async task
  running it would block the event loop — recreating QUEUE-01's exact defect one layer up.

`_loop(gen: int, stop_evt: threading.Event)` — each thread runs, until **its own** stop event is set
**or its generation is stale**. **It never acquires `_LIFECYCLE_LOCK` (D-20) — see the deadlock rule
above.**
  1. **`if stop_evt.is_set() or gen != _generation: return`** — the generation compare is a cheap
     second fence so a thread from a previous generation winds itself down at its next loop boundary
     instead of racing a new set. Costs one integer compare per iteration; read lock-free (a plain int
     load, atomic under the GIL). **Note the event is the thread's ARGUMENT, not a module global read**
     — that is the whole of D-15 and it is one word wide, so it is exactly the kind of thing a
     "cleanup" refactor deletes. Comment it.

  2. `worked = drain.drain_once()`
  3. `if worked: continue` — drain until the queue is empty before ever sleeping.
  4. `wake.clear()` then `wake.wait(timeout=settings.queue_poll_seconds)`.

  Wrap the body in a broad `try/except` that LOGS and continues — a worker thread that dies on a

`stop(grace_seconds: float = 10) -> None` — **the entire body runs under `_LIFECYCLE_LOCK` (D-20)**, in
exactly this order.

**IDEMPOTENT, and now with an executable definition (D-20):** a caller that finds `_stop is None` and
`_threads == []` — a never-started pool, or a second `stop()` that lost the race — **returns
immediately as a no-op, before step 1.** It does not set an event, does not join, and does **not** call
`release_leases` (not even with an empty list). Across N concurrent `stop()`s exactly ONE performs the
transition and exactly ONE `release_leases` is issued. A second caller BLOCKS on the lock rather than
interleaving with the first — which is what stops it from writing `_threads`/`_orphans`/`_stop` back
from a snapshot taken before the first finished.
  1. Set **the current generation's** stop event (`_stop.set()`). It stays set
     forever; nothing ever clears it. The next `start()` mints a new one (D-15).
  2. `wake.wake()` — break every thread out of its `wait()` immediately.
  3. `thread.join(timeout=grace_seconds)` for each.
  4. **`repo.release_leases(drain.held_tokens())` — UNCONDITIONALLY, even if a thread did not exit.**
     This is the whole point of criterion #4 and the part that looks wrong until you reason it through:
     releasing a lease out from under a still-running worker is CORRECT. That worker's eventual
     `complete_job` is fenced on the lease token, so it returns False, and the worker logs and drops —
     the documented zombie contract. Meanwhile the row is back at `pending` and the NEXT instance picks
     it up in seconds instead of waiting out a 15-minute lease. Write this reasoning into the docstring;
     a future reader WILL try to "fix" it by only releasing after a clean join.
  5. **Partition the threads, do NOT blindly clear the list.** Any thread that is STILL `.is_alive()`
     after its join timed out moves into `_orphans`; the rest are dropped. Then `_threads = []` and
     `_stop = None`. Dropping the module's reference to the event is safe **because each thread holds
     its own reference** — an orphan keeps observing the set event it was given and exits at its next
     loop boundary. That is precisely what a shared, `.clear()`-able event could not guarantee (D-15).
