<objective>
Give the queue a running body: 2 daemon worker threads, owned by the app's **first-ever** FastAPI
`lifespan`, that wake instantly on the in-process signal, poll slowly otherwise, and — the part
ROADMAP criterion #4 is actually about — **release every lease they hold the moment the process is
asked to stop.**

Render free spins the service down after 15 minutes of no inbound traffic and redeploys are routine.
Without a graceful release, every in-flight retrigger stalls for the full `LEASE_SECONDS` (15 minutes)
before another worker can touch it. With it, the new instance picks the work up in seconds.

And **D-07**: `WORKER_COUNT + 2 <= max_size (5)` is asserted at boot and the app **refuses to start**
if violated. Not clamp-and-warn. A misconfigured deploy must fail loudly and visibly on Render rather
than booting into a state where every request silently hangs waiting for a connection the workers have
all checked out. Fail-fast on a startup-time config error is the money-system default — and this repo
already sets that precedent (`database_url: str  # no default — fails fast if unset`).

Purpose: QUEUE-03.
Output: `app/queue/worker.py`, the lifespan wiring, and Proof 4.
</objective>

**`app/queue/worker.py`** — the thread pool, the loop, and the lifespan.

`POOL_BUDGET_RESERVE = 2` — the connections that must stay available for everything that is not a
worker (inbound webhook ingest, operator approve/reject, dashboard reads). Comment it as such.

`start(n: int) -> None`:
- `n <= 0` -> log "queue workers disabled" and return WITHOUT starting anything. This is the
  test/dev off switch (`WORKER_COUNT=0`, D-06/D-08) and it must be a genuine no-op, not a
  zero-length loop that still allocates state.
- Otherwise spawn `n` `threading.Thread(target=_loop, daemon=True, name=f"queue-worker-{i}")`.
  Daemon threads, NOT a `ThreadPoolExecutor`: N self-driving claim loops need no dispatcher, and an
  executor adds a submission thread for zero safety gain. And NOT `asyncio.Task`s: every downstream
  pipeline call (LLM via `openai`, psycopg, Resend) is blocking synchronous I/O, so an async task
  running it would block the event loop — recreating QUEUE-01's exact defect one layer up.

`_loop()` — each thread runs, until the stop event is set:
  1. `worked = drain.drain_once()`
  2. `if worked: continue` — drain until the queue is empty before ever sleeping.
  3. `wake.clear()` then `wake.wait(timeout=settings.queue_poll_seconds)`.
     Clear BEFORE the wait, not after: a `wake()` arriving in the gap is then not lost, because the
     next iteration's `drain_once()` finds the (already-committed) row regardless.
  Wrap the body in a broad `try/except` that LOGS and continues — a worker thread that dies on a
  transient DB blip and takes its poll loop with it is a silent, permanent capacity loss.

`stop(grace_seconds: float = 10) -> None`, in exactly this order:
  1. Set the stop event.
  2. `wake.wake()` — break every thread out of its `wait()` immediately.
  3. `thread.join(timeout=grace_seconds)` for each.
  4. **`repo.release_leases(drain.held_tokens())` — UNCONDITIONALLY, even if a thread did not exit.**
     This is the whole point of criterion #4 and the part that looks wrong until you reason it through:
     releasing a lease out from under a still-running worker is CORRECT. That worker's eventual
     `complete_job` is fenced on the lease token, so it returns False, and the worker logs and drops —
     the documented zombie contract. Meanwhile the row is back at `pending` and the NEXT instance picks
     it up in seconds instead of waiting out a 15-minute lease. Write this reasoning into the docstring;
     a future reader WILL try to "fix" it by only releasing after a clean join.
  5. Clear the thread list so `start()` is re-callable (tests do exactly that).

`lifespan(app)` — an `@contextlib.asynccontextmanager`. Before the yield:
  - **D-07 budget guard.** If `settings.worker_count + POOL_BUDGET_RESERVE > supabase.POOL_MAX_SIZE`,
    **raise** (a `RuntimeError` with a message naming the configured count, the reserve, and the pool
    max) so the app refuses to start. Use an explicit raise, NOT a bare `assert` — assertions are
    stripped under `python -O` and this guard must survive any interpreter flag. Do NOT clamp and warn:
    a clamped deploy boots into a state where requests silently hang on a connection the workers hold,
    which is exactly the invisible failure fail-fast exists to prevent.
  - `worker.start(settings.worker_count)`.
  After the yield: `worker.stop(grace_seconds=10)`.
  </action>
