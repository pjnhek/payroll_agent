<objective>
Give the queue a running body: 2 daemon worker threads, owned by the app's **first-ever** FastAPI
`lifespan`, that wake instantly on the in-process signal, poll slowly otherwise, and â€” the part
ROADMAP criterion #4 is actually about â€” **release every lease they hold the moment the process is
asked to stop.**

Render free spins the service down after 15 minutes of no inbound traffic and redeploys are routine.
Without a graceful release, every in-flight retrigger stalls for the full `LEASE_SECONDS` (15 minutes)
before another worker can touch it. With it, the new instance picks the work up in seconds.

And **D-07**: `WORKER_COUNT + 2 <= max_size (5)` is asserted at boot and the app **refuses to start**
if violated. Not clamp-and-warn. A misconfigured deploy must fail loudly and visibly on Render rather
than booting into a state where every request silently hangs waiting for a connection the workers have
all checked out. Fail-fast on a startup-time config error is the money-system default â€” and this repo
already sets that precedent (`database_url: str  # no default â€” fails fast if unset`).

And **D-15** â€” the round-2 cross-AI review finding, and the one that would have shipped a queue with a
body and no pulse. The previous draft's `stop()` set a stop event that **nothing ever reset**, so the
next `start()` spawned threads that saw it already set and returned on their first iteration. On
Render â€” where a redeploy is routine and a 15-minute idle spin-down is the default â€” *every restart*
would have come back with workers that exist and never drain. The fix is one word wide (a **fresh
`threading.Event` per generation**, passed as a thread argument, never `.clear()`ed), which is exactly
why it needs a proof that a **restarted worker completes a real job**. `thread.is_alive()` is GREEN
against this bug; it is the vacuous twin, and it is prohibited here.

Purpose: QUEUE-03.
Output: `app/queue/worker.py`, the lifespan wiring, Proof 4, and the D-15 restart proof.

> ### âš  D-15 â€” THE STOP EVENT IS PER-GENERATION. READ THIS BEFORE YOU WRITE `start()`.
>
> **This is the cross-AI review's round-2 HIGH finding, CONFIRMED against the previous draft of this
> plan.** That draft said `_loop` runs "until the stop event is set", and that `stop()` **sets** it â€”
> and then never un-set it anywhere. So after a clean `start() â†’ stop()`, the next `start()` would
> spawn threads whose **first loop iteration observes an already-set event and returns immediately.**
> Workers exist. Nothing ever drains. A restarted process is silently dead, and the plan's own restart
> test (which asserts a start after the orphan dies *succeeds*) would have "succeeded" at spawning
> corpses.
>
> **The fix is NOT `_stop.clear()` in `start()`.** That is the obvious one-liner and it is wrong:
> clearing a **shared** event **un-stops a straggler** â€” a worker that was told to stop but has not
> yet reached the loop boundary where it would notice. Task 3's Proof 4 constructs exactly that
> straggler on purpose (a handler blocked past a timed-out join). A `.clear()` would revoke its stop
> instruction and resurrect it on top of the new generation, holding a pooled connection against a
> `max_size=5` budget â€” **D-07's failure mode arriving through the back door.**
>
> **The mechanism:** `start()` mints a **brand-new `threading.Event()`** for the generation it is
> about to spawn, stores it in `_stop`, and **passes it to each thread as an argument**. A
> generation's event, once set by `stop()`, stays set **forever** â€” it belongs to that generation and
> to no other. A new generation begins with a fresh, unset object. The straggler can never be
> un-stopped, and the restarted worker can never start pre-stopped. Write this reasoning into the
> module docstring; a future reader WILL try to "simplify" it back to one shared event.

`start(n: int) -> None`:
- First, **prune** `_orphans` of threads that have since died.
- **Second-start guard (this is a cross-AI review finding).** If `_threads` is non-empty, or any
  `_orphan` is still `.is_alive()`, **raise `RuntimeError`** naming the live count and the current
  generation. Do NOT silently spawn a second set. Fencing keeps the DB writes *safe* â€” a stale
  worker's `complete_job` is rejected on the lease token â€” but it does NOT keep the process from
  carrying **2 Ă— WORKER_COUNT** live threads, each holding a pooled connection against a `max_size=5`
  budget. That is D-07's exact failure mode arriving through the back door, and D-07's whole posture
  is fail-fast, not clamp-and-hope. Refusing is consistent.
- `n <= 0` -> log "queue workers disabled" and return WITHOUT starting anything. This is the
  test/dev off switch (`WORKER_COUNT=0`, D-06/D-08) and it must be a genuine no-op, not a
  zero-length loop that still allocates state. (In particular: do NOT mint a stop Event on this path.)
- Otherwise: `_generation += 1`, capture it as `gen`; **`stop_evt = threading.Event()` â€” a FRESH
  object, never a `.clear()` of the old one (D-15)** â€” assign `_stop = stop_evt`; and spawn `n`
  `threading.Thread(target=_loop, args=(gen, stop_evt), daemon=True, name=f"queue-worker-{gen}-{i}")`.
  Each thread thus **closes over its own generation's event**, so it can never be signalled â€” or
  un-signalled â€” by another generation's lifecycle.
  Daemon threads, NOT a `ThreadPoolExecutor`: N self-driving claim loops need no dispatcher, and an
  executor adds a submission thread for zero safety gain. And NOT `asyncio.Task`s: every downstream
  pipeline call (LLM via `openai`, psycopg, Resend) is blocking synchronous I/O, so an async task
  running it would block the event loop â€” recreating QUEUE-01's exact defect one layer up.

`_loop(gen: int, stop_evt: threading.Event)` â€” each thread runs, until **its own** stop event is set
**or its generation is stale**:
  1. **`if stop_evt.is_set() or gen != _generation: return`** â€” the generation compare is a cheap
     second fence so an orphan from a previous generation winds itself down at its next loop boundary
     instead of racing a new set. Belt-and-braces with the `start()` refusal above; costs one integer
     compare per iteration. **Note the event is the thread's ARGUMENT, not a module global read** â€”
     that is the whole of D-15 and it is one word wide, so it is exactly the kind of thing a
     "cleanup" refactor deletes. Comment it.
  2. `worked = drain.drain_once()`
  3. `if worked: continue` â€” drain until the queue is empty before ever sleeping.
  4. `wake.clear()` then `wake.wait(timeout=settings.queue_poll_seconds)`.
     Clear BEFORE the wait, not after: a `wake()` arriving in the gap is then not lost, because the
     next iteration's `drain_once()` finds the (already-committed) row regardless.
  Wrap the body in a broad `try/except` that LOGS and continues â€” a worker thread that dies on a
  transient DB blip and takes its poll loop with it is a silent, permanent capacity loss.

`stop(grace_seconds: float = 10) -> None`, in exactly this order. **It must be IDEMPOTENT: calling it
twice, or on a never-started pool, is a safe no-op** (`_stop is None` -> nothing to set, no threads to
join, `release_leases([])` returns 0 without issuing a statement).
  1. Set **the current generation's** stop event (`if _stop is not None: _stop.set()`). It stays set
     forever; nothing ever clears it. The next `start()` mints a new one (D-15).
  2. `wake.wake()` â€” break every thread out of its `wait()` immediately.
  3. `thread.join(timeout=grace_seconds)` for each.
  4. **`repo.release_leases(drain.held_tokens())` â€” UNCONDITIONALLY, even if a thread did not exit.**
     This is the whole point of criterion #4 and the part that looks wrong until you reason it through:
     releasing a lease out from under a still-running worker is CORRECT. That worker's eventual
     `complete_job` is fenced on the lease token, so it returns False, and the worker logs and drops â€”
     the documented zombie contract. Meanwhile the row is back at `pending` and the NEXT instance picks
     it up in seconds instead of waiting out a 15-minute lease. Write this reasoning into the docstring;
     a future reader WILL try to "fix" it by only releasing after a clean join.
  5. **Partition the threads, do NOT blindly clear the list.** Any thread that is STILL `.is_alive()`
     after its join timed out moves into `_orphans`; the rest are dropped. Then `_threads = []` and
     `_stop = None`. Dropping the module's reference to the event is safe **because each thread holds
     its own reference** â€” an orphan keeps observing the set event it was given and exits at its next
     loop boundary. That is precisely what a shared, `.clear()`-able event could not guarantee (D-15).
     > **âš  The earlier draft said "clear the thread list so `start()` is re-callable" â€” full stop. That
     > is the bug the review found.** A blocked handler (exactly the situation Proof 4 constructs on
     > purpose) survives a timed-out join; clearing the list forgets it, and the next `start()` builds a
     > second worker set on top of a live one. Track it as an orphan instead, and let `start()` refuse
     > while it lives. `start()` stays re-callable for every clean stop â€” which is all the tests need.
