<objective>
Close GAP-1 (CR-1): `POST /runs/{run_id}/resolve` claims `NEEDS_OPERATOR → EXTRACTING` itself (main.py:859), then background-dispatches `_operator_resume` → `resume_pipeline(from_status=NEEDS_OPERATOR)`, which does a SECOND `claim_status(NEEDS_OPERATOR → EXTRACTING)`. That second CAS always fails (the status is already `EXTRACTING`), so `resume_pipeline` returns early doing nothing — the operator's resolution is silently dropped and the run is stuck in `EXTRACTING` forever, with no error, no processing, no path to approval.

Purpose: `resume_pipeline` already owns a correct, single, atomic CAS claim for every other caller (the webhook's reply-resume path never pre-claims). The fix is to make `/resolve` behave the same way — validate the POST, apply the override, then let `resume_pipeline` be the ONE and ONLY place that transitions `NEEDS_OPERATOR → EXTRACTING`.
Output: `resolve()` no longer calls `claim_status` — it unconditionally schedules `_operator_resume` after a valid POST; a new end-to-end test drives the REAL `resume_pipeline` (not mocked) from a `needs_operator` run all the way to `awaiting_approval`, proving the double-CAS strand is gone.
</objective>
