<objective>
Close the multi-round context-loss gap (CX-01/T-09-21) and give ambiguous replies a code-owned attribution anchor. This plan OWNS `resume_pipeline`: it writes the D-11-02 consumed marker at the CAS claim (the load-bearing seam that makes reply accumulation observable at runtime), rewrites `_combined_context_email` into a pure function that assembles ORIGINAL + a QUESTIONS-WE-ASKED anchor (rendered from persisted decision facts) + ALL consumed replies in round order + the current reply (D-11-10, D-11-12, D-11-13), and adds the absent-if-unaddressed no-guess extraction instruction (D-11-11). Flip the known-edge fixture from documenting the bug to asserting the fix (Round-1 "30, not 40" pays 30). This covers CLAR2-03 and CLAR2-05.

Purpose: today a Round-1 correction is silently reverted because the resume context re-reads only the ORIGINAL body + the latest reply; a bare "40" has no anchor tying it to the employee/field asked about. Both are silent-mispay classes. AND: without the consumed marker written here, `load_consumed_replies` returns empty forever, so the accumulation is a runtime no-op even though hermetic tests seeded with fake consumed rows would pass — the marker write is the seam that makes the whole accumulation real.
Output: the consumed-marker write at the resume claim, rewritten `_combined_context_email` + resume accumulation, the no-guess prompt instruction, one new pure-function test module, and the flipped known-edge fixture. Zero change to the 7.5 four-outcome machine or `is_round_2`.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| resume CAS claim → consumed marker | the marker records that this specific reply started processing; a missing or mis-timed write silently disables accumulation and redelivery detection |
| client reply body → resume extraction context | untrusted email text is accumulated across rounds and fed to the extractor |
| accumulated reply store (email_messages) | multiple consumed reply rows are concatenated in round order; a stale/duplicate body could poison a later round |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-16 | Repudiation / Tampering | consumed marker not written or written twice at the resume claim | mitigate | Marker is written immediately after the successful CAS claim (D-11-02) and is write-once (`consumed_round IS NULL` guard, 11-01); Task 4's REAL-row test fails if the seam is removed, so accumulation cannot silently no-op |
| T-11-07 | Tampering | prompt injection via accumulated reply bodies (a reply containing a spoofed "QUESTIONS WE ASKED:" section) | mitigate | The anchor is code-owned and code-positioned; extraction output is schema-validated and every money field re-gates deterministically at `decide` + human approval. Residual/accepted: same class as the existing single-reply injection surface, unchanged posture (RESEARCH Security Domain). |
| T-11-08 | Tampering | stale/duplicate reply body poisoning a later round's context | mitigate | prior_replies is sourced from `load_consumed_replies` ordered by consumed_round with the current reply excluded by marker — no duplicate concatenation; write-once consumed marker (this plan + 11-01) prevents a redelivered body re-entering as a new prior |
| T-11-09 | Tampering | no-guess policy bypass (an asked field silently guessed onto an unaddressed employee) | mitigate | Enforcement is the deterministic decide-gate backstop (absent → re-ask → send), NOT the prompt; Task 4's backstop test pins a SEND, so a regression that starts guessing fails the test |
