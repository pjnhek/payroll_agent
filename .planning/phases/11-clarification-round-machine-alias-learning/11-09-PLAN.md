<objective>
Close GAP-4 (CR-4): the D-11-15 bind-on-confirmation check computes two facts INDEPENDENTLY over the whole run's post-resume reconciliation — (a) the suggested employee id newly resolves SOMEWHERE, (b) the token is gone from unresolved SOMEWHERE — and binds when both happen to be true, even if they are unrelated. "No, Dave didn't work this period; David worked 5 hours separately" satisfies both facts via two completely different reconciliation entries, and silently binds `Dave → David` — the exact never-learn-from-inference class the guard exists to prevent.

Also close WR-1: `set_alias_candidates` is a full-column JSONB overwrite. With 2+ distinct tokens across 2+ rounds, the last writer erases prior candidates — a client-confirmed bind from an earlier round can be wiped by a later, unrelated capture/suggest write before `_write_aliases_if_safe` ever reads it at approval.

Purpose: tie the bind decision to the SAME reconciliation record the token itself produced (or the record for the suggested employee's own canonical full_name, since a legitimate confirming reply restates that name) — never two independently-satisfied whole-run facts. Make the write side additive so multiple tokens' evidence coexists correctly across rounds.
Output: a new `_bind_evidence_for_token` helper in orchestrator.py replacing the two independent set-membership checks; `set_alias_candidates` becomes a merge write; regression tests for both the exploit scenario and the two-tokens-across-two-rounds clobber scenario; the existing legitimate-confirmation and misname-guard tests continue to pass.
</objective>
