<objective>
Close GAP-5 (CR-5): a reply is linked to its run INSIDE the webhook's committed ingest transaction based purely on the RFC header chain (`in_reply_to`/`references`, attacker-controllable), with `consumed_round=NULL`. FIX-5's sender revalidation (`find_business_by_sender(from_addr) == run.business_id`) runs only POST-commit, inside `_finish_reply_resume`, on the FIRST delivery. If FIX-5 fails, the response is `sender_mismatch` — but the row stays linked and unconsumed. Neither the WR-04 redelivery re-schedule (webhook duplicate branch) nor the D-11-05 stranded-reply sweep (runs-list load) re-checks the sender before dispatching `_resume_pipeline` — both check only `consumed_round IS NULL` and run status. A spoofed reply that failed FIX-5 once can still drive the victim's payroll extraction via a redelivery or a later dashboard load.

Purpose: FIX-5 must be re-asserted at EVERY seam that can dispatch a resume from a linked-but-unconsumed reply, not just the first one. A single shared predicate makes this a structural guarantee rather than something each new re-schedule seam has to remember to re-derive.
Output: a new `_reply_sender_ok(row, run)` pure predicate in main.py; both the WR-04 duplicate-branch re-schedule and the D-11-05 stranded sweep call it before dispatching `_resume_pipeline`; two new regression tests proving a FIX-5-failed linked reply is never resumed by either seam.
</objective>
