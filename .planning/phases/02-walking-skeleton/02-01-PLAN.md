<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM provider → app | Untrusted model output (JSON content) crosses into the pipeline; never trusted as schema-valid until Pydantic-validated. |
| DB ← repo writes | All SQL is constructed in repo.py; parameter values originate from validated contracts. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | repo.py SQL (all writes) | mitigate | Parameterized psycopg `%s`/named placeholders only; never f-string SQL (PATTERNS Shared Patterns; RESEARCH Security Domain V5). |
| T-02-02 | Tampering | LLM JSON output → contracts | mitigate | `model_validate_json` on the real Pydantic contract is the field-correctness gate; `json_object` guarantees syntax only (RESEARCH Pitfall 1). |
| T-02-03 | Information disclosure | LLM client / gateway logging | mitigate | Do not log full email bodies at info level; log Message-IDs + status, not PII (RESEARCH Security Domain PII-safe logging). |
| T-02-04 | Spoofing | inbound email_messages | mitigate | `UNIQUE(message_id)` + `ON CONFLICT DO NOTHING` replay defense surfaced in repo.insert_inbound_email (consumed by Plan 02 webhook). |
| T-02-SC | Tampering | npm/pip installs | mitigate | No NEW package names this plan (openai/fastapi/uvicorn/python-multipart pre-vetted + pinned in Phase 1; RESEARCH Package Legitimacy Audit). A reply-parser, IF chosen later, requires the full legitimacy gate + a blocking human checkpoint. |
| T-02-05 | Repudiation | webhook signing-secret / SPF-DKIM | accept | Real-gateway signing-secret + SPF/DKIM verification are P6 (no real provider in Phase 2); documented, not implemented (per planning context — no high-severity item blocks Phase 2). |
</threat_model>
