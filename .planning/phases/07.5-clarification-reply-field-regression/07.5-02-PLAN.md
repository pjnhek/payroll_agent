**D-7.5-10 THREE-PHASE ORDERING (the structural change from Round 2):**
The round-1 R3-1 fix moved backfill BEFORE validate(). But validate() WAS the detector (it called detect_field_regression internally). So on first resume: backfill rewrites OT silence→snapshot value, then validate() compares snapshot(OT=2) vs backfilled(OT=2) → no drop seen → no field_regression issue → no asked → no clarification → carried value reaches operator with NO provenance. Detection was blind.

The fix (D-7.5-10): DETECT must run BEFORE BACKFILL, on the RAW extraction. Since detect cannot be inside validate() (which runs on backfilled data), detection must move OUT of validate() and into the orchestrator, as its own explicit step. This plan implements detect_field_regression as a PUBLIC pure function exportable by orchestrator. validate() no longer calls detect internally — it accepts PRE-COMPUTED drops via `raw_field_drops=` kwarg.

Ordering invariant (non-negotiable, D-7.5-10):
1. DETECT: orchestrator calls detect_field_regression(prior, RAW extracted, prior_matches, matches)
2. BACKFILL: orchestrator calls backfill_extracted(extracted, prior, matches, resolved_drops)
3. CALC: orchestrator calls validate(BACKFILLED extracted, ..., raw_field_drops=drops) → decide → line_items

This plan establishes steps 1 and 3 (detect as public function + validate() accepting pre-computed drops). Plan 03 wires the full sequence in _run_stages.
