# Phase 07.5: Code Review Report

**Reviewed:** 2026-06-28T00:00:00Z
**Depth:** standard
**Files Reviewed:** 14

## Summary

This phase (MONEY-03) implements the clarification-reply field-regression fix: the
three-phase DETECT-on-RAW → BACKFILL → CALC ordering (D-7.5-10), the D-7.5-11
classify-first-on-answered-rounds logic with the four ClarifiedFields outcomes, the
two-set detection/backfill separation, the write-once `pre_clarify_extracted` snapshot
CAS, the `clarified_fields` outcome column, and the N4 `purpose` CHECK migration.

The architecture is sound and the test coverage is unusually thorough. However, the
review found two BLOCKER defects on the money path: (1) the Round-2 classify lookup is
keyed against `prior_matches` only, so a client who restates a name in their reply
silently fails to classify the answer — `confirmed_dropped` and `client_supplied`
outcomes are missed, which on a later round can cause **overpay via re-backfill** of a
field the client explicitly zeroed; and (2) the N4 purpose CHECK DROP+RE-ADD block in
schema.sql is NOT wrapped in a transaction as its own comment claims (D-7.5-03a), so a
crash between DROP and ADD leaves the `purpose` column with no CHECK constraint at all,
permanently widening the allowlist that `get_outbound_message_id` depends on.

Seven warnings cover robustness gaps in the resume flow, a stale-snapshot edge,
detection running needlessly on suppressed fields, and a `FakeConnection` mismatch that
weakens the offline SQL-discipline tests.

## Critical Issues

### CR-01: Round-2 classify keys on `prior_matches` only — a restated name silently drops the answer and can overpay

**File:** `app/pipeline/orchestrator.py:374-413`

**Issue:** In the Round-2 classify-first block, the employee resolver used to attribute
the raw reply's answer to an `emp_id_str` is built EXCLUSIVELY from `prior_matches` (the
snapshot round's reconciliation):

```python
name_to_id_for_classify: dict[str, str] = {
    m.submitted_name: str(m.matched_employee_id)
    for m in prior_matches
    if m.resolved and m.matched_employee_id is not None
}
...
for raw_name, raw_e in raw_name_to_emp.items():
    if name_to_id_for_classify.get(raw_name) == emp_id_str:
        raw_emp = raw_e
        break
if raw_emp is None:
    continue  # field remains 'asked' — "safe no-op"
```

The raw reply uses the CURRENT submitted names. If the client restates a name in the
reply (the exact "M. Chen" → "Maria Chen" case that R3-3 / `detect_field_regression`
and `backfill_extracted` were specifically hardened against by employee-id keying on
BOTH sides), `name_to_id_for_classify` — built only from the prior names — will not
contain the new name, `raw_emp` stays `None`, and the asked field is never classified.

Consequences, all on the money path:
- The asked field stays `"asked"` forever → the run re-clarifies indefinitely (a
  clarify loop), OR
- Worse: because the field was never classified, it is absent from BOTH
  `suppress_detection` and `backfill_skip`. On this and every subsequent round it is
  treated as `carried_forward`, so backfill **re-fills it from the snapshot**. If the
  client's reply actually ZEROED that field (an intended `confirmed_dropped`), the
  snapshot value (e.g. OT=2) is silently restored → **the employee is overpaid** — the
  exact failure D-7.5-11's overpay guard exists to prevent.

This contradicts the design's own headline invariant (employee-id keying must survive a
restated name), and `detect_field_regression` / `backfill_extracted` already key on
BOTH prior and current matches — only the classify step regressed to prior-only.

**Fix:** Resolve the raw reply's names against the CURRENT roster (the same
`reconcile_names` call `_run_stages` will make), and use the current matches — unioned
with `prior_matches` — to map raw names to `emp_id_str`. Reconcile once up front and
thread it in:

```python
raw_submitted = [e.submitted_name for e in raw_extracted.employees]
current_matches = reconcile_names(raw_submitted, roster)
name_to_id_for_classify = {
    m.submitted_name: str(m.matched_employee_id)
    for m in (list(prior_matches) + current_matches)
    if m.resolved and m.matched_employee_id is not None
}
```

Pass the precomputed `current_matches`/`raw_extracted` into `_run_stages` (it already
accepts `extracted=`) so reconciliation is not duplicated, and add a regression test
where the Round-2 reply restates the name AND explicitly zeroes the asked field — assert
the outcome is `confirmed_dropped` and the paystub field is 0 (no re-backfill).

### CR-02: N4 `purpose` CHECK DROP+RE-ADD is not atomic — a crash leaves the column with no CHECK at all

**File:** `app/db/schema.sql:178-199`

**Issue:** The N4 migration block claims "atomic DROP+ADD in one transaction
(D-7.5-03a)" in its comment, but the `DO $$ ... $$` block does NOT establish a
transaction. A PL/pgSQL `DO` block runs each statement in the surrounding transaction
context; when `schema.sql` is applied statement-by-statement by the bootstrap runner
(the normal path — `python -m app.db.bootstrap`), the `DROP CONSTRAINT` and the
`ADD CONSTRAINT` are not guaranteed to be in one wrapping `BEGIN/COMMIT`. If the process
is interrupted (timeout, connection drop, Supabase pooler reset — all live failure modes
named in CLAUDE.md) after the DROP and before the ADD, the `email_messages.purpose`
column is left with NO CHECK constraint.

This is a real defect, not theoretical, because the allowlist is load-bearing in code:
`get_outbound_message_id` (repo.py:762) trusts that only
`{'clarification','confirmation','clarification_field_regression'}` can be persisted.
With the CHECK gone, any string can be written to `purpose`, silently widening the
`uq_email_run_purpose` semantics and the idempotency/threading guards that key on it.

A secondary issue: the conditional ADD uses a hardcoded name
`email_messages_purpose_check` for the existence test, but the DROP removes whatever
`conname LIKE '%purpose%'` matched (line 184-186). If the live constraint was named
differently (e.g. the inline `CHECK` from the original CREATE produces a system name),
the DROP removes it but the `IF NOT EXISTS (... conname = 'email_messages_purpose_check')`
re-add is keyed on a name that never existed — fine on the first run, but if a prior
partial run already added the literally-named constraint, a re-DROP via the LIKE could
remove it and the re-ADD path is correct only by luck of naming.

**Fix:** Wrap the DROP+ADD in an explicit transaction so it is all-or-nothing, and
verify the bootstrap runner executes `schema.sql` as a single script (one transaction)
rather than autocommitting per statement:

```sql
BEGIN;
DO $$
DECLARE _con_name TEXT;
BEGIN
    SELECT conname INTO _con_name FROM pg_constraint
    WHERE contype='c' AND conrelid='email_messages'::regclass AND conname LIKE '%purpose%';
    IF _con_name IS NOT NULL THEN
        EXECUTE 'ALTER TABLE email_messages DROP CONSTRAINT ' || quote_ident(_con_name);
    END IF;
    ALTER TABLE email_messages ADD CONSTRAINT email_messages_purpose_check
        CHECK (purpose IN ('clarification','confirmation','clarification_field_regression'));
END;
$$;
COMMIT;
```

(Confirm `bootstrap.py` runs the file as one `cur.execute(full_sql)` / single
transaction; if it splits on `;`, the `DO $$ ... $$` and any `BEGIN/COMMIT` will break —
that splitting strategy must be fixed regardless.)

## Warnings

### WR-01: Round-2 classify can mislabel `carried_forward` when the asked field is unresolvable, allowing overpay of a confirmed-zero field

**File:** `app/pipeline/orchestrator.py:396-413, 433-438`

**Issue:** When `raw_emp is None` the field is left `"asked"` and NOT added to
`newly_classified` (line 396-399). Because `backfill_skip` is built only from
`newly_classified` + prior terminals (line 433-438), an unclassified field is absent
from `backfill_skip`, so `backfill_extracted` will FILL it from the snapshot. For a
field the client meant to zero, this is a silent overpay. This is the same root failure
as CR-01 but also fires whenever the LLM omits the employee from the raw extraction
entirely (not just on a restated name). The "safe no-op" comment is incorrect — leaving
an asked field unclassified is NOT safe when the default downstream behavior is
backfill-fills.

**Fix:** Treat an unresolvable asked field conservatively: if it cannot be classified,
add it to `backfill_skip` (do NOT backfill) so the worst case is an under-fill that
re-clarifies, never an overpay. Better, fix the resolver per CR-01 so this branch
becomes genuinely rare, and add a test for the "employee missing from raw reply"
sub-case.

### WR-02: `clarified[emp_id_str][field]` write assumes the key exists; a defensively-built dict raises KeyError

**File:** `app/pipeline/orchestrator.py:405,408,411`

**Issue:** The classify loop iterates `for emp_id_str, field_outcomes in list(clarified.items())`
and then writes `clarified[emp_id_str][field] = ...`. This is safe only because it
iterates existing keys. But the parallel Round-1 path builds the dict with
`clarified.setdefault(str(current_emp_id_r1), {})[field_name_r1] = "asked"`
(line 333) — note `str(...)`. If any persisted `clarified_fields` key was ever written
without the `str()` normalization (e.g., a future code path, or JSON round-trip nuance),
the lookup `name_to_id_for_classify.get(raw_name) == emp_id_str` compares a freshly
`str(UUID)` against the stored key and the assignment target could diverge. The code is
correct today but brittle: the emp_id key type contract (always `str(UUID)`) is enforced
only by convention across three write sites.

**Fix:** Centralize the emp_id key normalization (a single helper `_emp_key(uuid) ->
str`) used at every read/write of `clarified` and `resolved_drops`, and assert the
invariant once. Reduces the chance a future edit reintroduces the UUID-vs-str mismatch
the N8 comment (validate.py:207-211) warns about.

### WR-03: `detect_field_regression` runs on confirmed-dropped fields every round; suppression is downstream-only

**File:** `app/pipeline/orchestrator.py:609` + `app/pipeline/validate.py:191-223`

**Issue:** `detect_field_regression` is always called on the raw extraction
(orchestrator line 609) and re-emits a `RawFieldDrop` for any field that went from
paid→unpaid versus the snapshot — including fields already terminally
`confirmed_dropped` in a prior round. Suppression happens only later in `validate()`
via the N8 `resolved_drops` check (validate.py:211). This works, but couples
correctness to two separate sets staying in sync: if `suppress_detection` ever fails to
include a prior `confirmed_dropped` pair (e.g., the restated-name bug in CR-01), the
drop is re-promoted to a `field_regression` issue and the run re-clarifies a field the
client already answered — the "stranding" bug this phase was built to kill.

**Fix:** Defense-in-depth: pass the prior-terminal set into `detect_field_regression`
so a confirmed-dropped/client-supplied (employee_id, field) is never re-emitted as a raw
drop in the first place, rather than relying solely on validate's N8 to filter it after
the fact. Detection-time suppression makes the invariant local.

### WR-04: First-resume snapshot can capture a regressed reply as the "original," breaking later backfill

**File:** `app/pipeline/orchestrator.py:342-351` (Round-1 deferred) + `_clarify` snapshot at orchestrator.py:712/828/844

**Issue:** `set_pre_clarify_extracted` is write-once (IS NULL CAS). The snapshot is
intended to hold the ORIGINAL good values so backfill can restore them. But the snapshot
is taken at the FIRST `_clarify`/AWAITING_REPLY transition. Consider a run whose FIRST
pause is itself a field-regression resume (Round-1): `_run_stages` persists the
**backfilled** extracted (orchestrator.py:637), `_clarify` is then called with
`persisted_extracted` (the backfilled version), and `set_pre_clarify_extracted` snapshots
it. On a first resume `prior=snapshot` is `None`, so no backfill occurred and the
persisted extracted equals the raw reply — which, on a regression, has the DROPPED value
(OT=None), not the original OT=2. If the original run had already snapshotted OT=2 this
is fine, but if the original pause did NOT snapshot (e.g., a path that reached
AWAITING_REPLY without going through `_clarify`'s snapshot calls), the first snapshot
captured here is the regressed reply, and every subsequent backfill restores the wrong
(dropped) value. The write-once guarantee then locks in the bad snapshot permanently.

**Fix:** Audit every code path that reaches `AWAITING_REPLY` and assert the snapshot is
taken from the ORIGINAL extraction (pre-any-regression). Add an integration test:
original run pauses (unresolved name, OT=2 present) → reply drops OT → assert snapshot
still holds OT=2 and the final paystub OT carries forward correctly.

### WR-05: `FakeConnection.execute` ignores `fetchone()` return scripting for chained `.execute().fetchone()`

**File:** `tests/conftest.py:144-155` + multiple repo helpers (e.g. repo.py:150-169)

**Issue:** Many repo helpers call `c.execute(sql, params).fetchone()` (single chained
call). `FakeConnection.execute` returns `self` (line 145-147) and `FakeConnection.fetchone`
pops the scripted queue (line 151-152) — OK. But `FakeCursor.execute` also returns
`self` (the cursor) and the cursor's `fetchone` delegates to the connection queue
(conftest line 97-98). The two surfaces share ONE queue but are scripted independently
in tests; a helper that opens a `cursor()` (dict_row reads) and another that uses the
bare connection both drain the same `_fetchone_q`. In tests that exercise multiple
helpers against one `fake_conn` the ordering coupling is implicit and fragile — a
reordered helper call silently consumes the wrong scripted row. This is a test-reliability
defect (flaky-by-construction), not a production bug, but it weakens the SQL-discipline
guards these fakes exist to protect.

**Fix:** Give `FakeCursor` its own fetch queue, or have `execute` record which statement
produced which result so `fetchone` returns the row scripted for THAT statement. At
minimum, document that one `fake_conn` must back exactly one helper call per test.

### WR-06: `get_outbound_message_id` raises `ValueError` on bad purpose, but `_clarify`/`_deliver` callers do not guard it

**File:** `app/db/repo.py:762-765` + `app/pipeline/orchestrator.py:702, 965`

**Issue:** `get_outbound_message_id` raises `ValueError` for an unrecognized `purpose`
(a good guard). `_clarify` is called with `purpose="clarification_field_regression"`
(orchestrator.py:350) which IS in the allowlist, so today it's fine. But the allowlist in
repo.py:762 is a hand-maintained tuple that must stay in lockstep with BOTH the schema
CHECK (schema.sql:152/170/196) AND every `purpose=` call site. There is no single source
of truth; the phase added the third value in three independent places. A future fourth
purpose added to the schema but missed in repo.py:762 would route a valid send through
the `ValueError` path, and since `_clarify` runs inside the D-A1-03 error-wrap, the run
would silently route to ERROR with reason `"ValueError"` (PII-safe but undiagnosable).

**Fix:** Define the purpose allowlist once (e.g., a `PURPOSES: frozenset` in one module)
and import it in repo.py, the schema-drift CI test, and anywhere a literal is used —
mirroring how `RunStatus` is the single source for the status CHECK.

### WR-07: `_run_stages` two-set semantics depend on caller never swapping the kwargs; no assertion guards the inversion

**File:** `app/pipeline/orchestrator.py:556-632`

**Issue:** `_run_stages(resolved_drops=..., suppress_detection=...)` carries two sets
with opposite meanings: `resolved_drops` → `backfill_extracted` (skip backfill);
`suppress_detection` → `validate(resolved_drops=...)` (suppress N8 re-flag). The Round-2
caller passes `resolved_drops=backfill_skip, suppress_detection=suppress_detection`
(orchestrator.py:452-453). If a future edit swaps these two kwargs, the failure is
SILENT and money-moving: `carried_forward` fields would be skipped by backfill (underpay)
and confirmed-dropped fields would be re-promoted (re-clarify/overpay). The prior commit
history shows this exact "set-conflation BLOCKER" was already hit once (commit 0b476b3).
The current code is correct but has no structural guard against re-conflation.

**Fix:** Make the two-set contract harder to misuse: pass a single typed object
(`@dataclass ResumeSuppression(backfill_skip: set, detection_suppress: set)`) instead of
two same-typed `set` kwargs, so a swap is a type/attribute error, not a silent inversion.
Add an explicit test asserting a `carried_forward` field is backfilled AND a
`confirmed_dropped` field is not, in the same run.

## Info

### IN-01: `RawFieldDrop` / `FieldDrop` duplication is dead-weight scaffolding

**File:** `app/models/contracts.py:124-145, 206-229`

**Issue:** `FieldDrop` (employee_id-keyed) is documented as "forward-compat scaffolding"
and "Nothing in Phase 7 emits this type." Phase 7.5 is now implemented and uses only
`RawFieldDrop`. `FieldDrop` appears unused by the reviewed code. Carrying two near-identical
records (the only difference is `submitted_name` vs `employee_id` key) invites confusion
about which is canonical.

**Fix:** If `FieldDrop` is still unreferenced after this phase, delete it; if it is the
intended public shape, wire it and drop `RawFieldDrop`. Per the structural pre-pass,
confirm there are no importers before removal.

### IN-02: `backfill_extracted` re-checks `emp_id_str is not None` after already gating on it

**File:** `app/pipeline/orchestrator.py:137-138, 154`

**Issue:** Line 137 computes `emp_id_str = name_to_id_current.get(...)` (may be None) and
line 138 sets `snap_emp = ... if emp_id_str else None`. Inside the field loop,
`if emp_id_str is not None and (emp_id_str, field) in _resolved_drops` (line 154) re-tests
`emp_id_str is not None` — but if `emp_id_str` were None, `snap_emp` would be None and the
`if snap_emp is None: continue` at line 140 already skipped the employee. The re-check is
dead-but-harmless.

**Fix:** Drop the redundant `emp_id_str is not None and` on line 154 for clarity.

### IN-03: `_field_regression_lines` will `ValueError` on a gate_reason with no dot

**File:** `app/pipeline/compose_email.py:43`

**Issue:** `submitted_name, field_name = qualified.rsplit(".", 1)` unpacks into two names
and will raise `ValueError: not enough values to unpack` if a `field regression: ` gate
reason ever lacks a `.`. Today `decide` always produces `"{submitted_name}.{field_name}"`,
so this can't fire — but `compose_clarification` is on the run path and an unexpected
gate_reason format would crash the draft (caught by the broad except → templated body,
so not fatal, but it would silently suppress ALL field-regression lines for that email).

**Fix:** Guard the split: `parts = qualified.rsplit(".", 1); if len(parts) != 2: continue`.

### IN-04: `insert_email_message` RETURNING fallback to `uuid.uuid4()` masks a real failure

**File:** `app/db/repo.py:746-748`

**Issue:** `return uuid.UUID(str(row[0])) if row else uuid.uuid4()` — the comment says
this only matters for the offline FakeConnection. In real Postgres `INSERT ... RETURNING`
always yields a row, so a `None` here would indicate the INSERT silently did nothing
(e.g., an `ON CONFLICT DO NOTHING` that unexpectedly conflicted). Returning a fresh random
UUID hides that the row was never written, and the caller treats a fabricated id as a
real persisted message id.

**Fix:** In production paths, treat `row is None` as an error (or log a warning); keep the
fabricated-id behavior strictly behind a test-only sentinel.

### IN-05: `is_round_2 = bool(clarified)` conflates "has any clarified entry" with "answered round"

**File:** `app/pipeline/orchestrator.py:301`

**Issue:** `is_round_2` is True whenever `clarified_fields` is non-empty. But a run could
have `clarified_fields` populated with only terminal (`confirmed_dropped`/`client_supplied`)
entries and zero `"asked"` entries — e.g., after a round fully resolved every asked field
but the run paused again for an unrelated reason. Entering the Round-2 classify block then
iterates with nothing to classify (the `if outcome != "asked": continue` guard handles it),
so it is currently harmless, but the round detection is a proxy, not a direct signal.

**Fix:** Make the round signal explicit: `is_round_2 = any(o == "asked" for fo in
clarified.values() for o in fo.values())`, or track a round counter, so the branch
condition names what it actually depends on.
