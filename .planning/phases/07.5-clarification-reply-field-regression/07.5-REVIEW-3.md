## Critical Issues

### CR-01: Two-extraction split fixes the classify label but still PAYS from the combined extraction â€” overpay/underpay when the two extractions disagree

**File:** `app/pipeline/orchestrator.py:349-352, 484-494` (and `_compute_line_items` consumes `raw_extracted` via `_run_stages`)

**Issue:** The round-2 CR-01 fix splits extraction into two: `raw_reply_extracted`
(reply-only) drives classify, and `raw_extracted` (combined) is passed as
`extracted=` to `_run_stages` and therefore feeds `_compute_line_items`. The
classify outcome is now read from the reply, but **the paid value is still read from
the combined extraction.** `backfill_skip` only prevents *snapshot restore* inside
`backfill_extracted`; it has no power over a value the combined extraction itself
reports. When the combined extraction returns the original section's value for the
asked field (the exact premise CR-01 is built on â€” "the original section's positive
value can survive into the extraction"), the two diverge and the paystub pays the
combined value while the audit trail records the reply-based outcome.

I proved three concrete divergences with a prompt-inspecting mock (reply-only returns
the client's value; combined adversarially returns the original section's value),
driving the real `resume_pipeline`:

| reply-only | combined | classify outcome | paystub OT | result |
|-----------|----------|------------------|-----------|--------|
| `0` (explicit zero) | `2` | `confirmed_dropped` | **2** | **OVERPAY** â€” pays the dropped OT |
| `5` (client supplies) | `2` | `client_supplied` | **2** | **UNDERPAY** â€” ignores the client's 5 |
| absent (reply omits employee) | `2` | stays `asked` (`_unresolvable_asked`) | **2** | pays OT=2 on a field marked unanswered, reaches AWAITING_APPROVAL, **never re-clarifies** |

This is a NEW defect introduced by the fix under review. At base `035535a` there was
a SINGLE extraction from `combined_email` used for BOTH classify and process, so the
classified value and the paid value were always the same object â€” they could not
diverge. Commit 89912e5 split them for classification but not for payment, relocating
the symptom (classify is now correct) without curing the disease (the paid value can
still be the eclipsed original). The third row also defeats the WR-01 "conservative
fail â€” under-fill that re-clarifies next round" invariant: the comment is only true
relative to snapshot restore; because the combined extraction independently carries
OT=2, the field is paid and the run advances with the field still `asked` and no
re-clarification.

For a money-moving, deterministic, auditable path the rule is "never silently
under/over-pays." This violates it for any field where the model's combined-body
extraction disagrees with its reply-only extraction on an asked field â€” which is the
very disagreement the fix's own rationale says is realistic.

**Fix:** Reconcile the paid value with the classify outcome instead of trusting the
combined extraction for asked fields. After classify, for every
`(emp_id_str, field)` in `newly_classified`, overwrite the corresponding field in
`raw_extracted` to the authoritative value the classify step decided:
`confirmed_dropped` â†’ set field to `None`/`Decimal('0')`; `client_supplied` â†’ set
field to the reply's `raw_val`; `carried_forward` â†’ set field to `None` so Phase-2
backfill fills it from the snapshot (instead of letting the combined value stand).
For an `_unresolvable_asked` field, do NOT let the combined value pass through â€” set
it to `None` so the conservative path genuinely under-fills (or re-defer the
clarification rather than processing a field still marked `asked`). Concretely, build
`raw_extracted` for the asked fields from the reply-derived values, e.g.:

```python
# after STEP 1 classify, before STEP 3 _run_stages:
reply_value_by_id = {}  # (emp_id_str, field) -> authoritative reply value
for emp_id_str, field in newly_classified:
    outcome = clarified[emp_id_str][field]
    raw_emp = _raw_emp_for_id(emp_id_str)  # via name_to_id_for_classify
    if outcome == "client_supplied":
        reply_value_by_id[(emp_id_str, field)] = getattr(raw_emp, field)
    elif outcome == "confirmed_dropped":
        reply_value_by_id[(emp_id_str, field)] = Decimal("0")
    elif outcome == "carried_forward":
        reply_value_by_id[(emp_id_str, field)] = None  # let backfill fill from snapshot
# _unresolvable_asked: force None so the combined value cannot leak through
for emp_id_str, field in _unresolvable_asked:
    reply_value_by_id[(emp_id_str, field)] = None
raw_extracted = _apply_reply_values(raw_extracted, matches_for_combined, reply_value_by_id)
```

Then the paid value is derived from the reply (the source of truth for an answered
field), the combined extraction is used only to retain *un-asked* originals
(its actual purpose, FIX 4), and `backfill_skip` stops the snapshot-restore overpay as
before. Add a regression test that drives `resume_pipeline` with the
**prompt-inspecting mock returning different values for the two extractions** and
asserts the **paystub line-item value** (not just the classify outcome) for all three
rows above.

### WR-01: Test 17 and the 10 updated Round-2 tests cannot detect classify/pay divergence â€” they assert the wrong thing or remove the disagreement

**File:** `tests/test_resume_pipeline.py:368, 411, 512, 677, 753, 864, 913, 975, 1059, 1125` (the `[_r2_*, _r2_*]` duplicated scripts) and `1277-1373` (Test 17)

**Issue:** The "update all 10 existing Round-2 tests for the doubled extraction call
count" change feeds the SAME scripted JSON to both extraction calls
(`mock_llm.script = [_r2_silent, _r2_silent]`, `[_r2_ot5, _r2_ot5]`, etc.), so the
reply-only and combined extractions can never disagree in any of these tests â€” the
exact precondition of CR-01 is scripted away. Test 17 (the new prompt-inspecting
mock) DOES make them disagree (reply `0`, combined `2`), and it correctly asserts the
classify outcome is `confirmed_dropped` (proving classify reads the reply) â€” but it
asserts only the outcome and the AWAITING_APPROVAL status, and **never asserts the
paystub OT value**. With the same mock I observed `PAYSTUB OT == Decimal('2')` while
the outcome was `confirmed_dropped`. So the suite's strongest money-safety test passes
while the run overpays. Test 15 (`test_answered_positive_uses_client_value`) asserts
paystub OT==5, but only because its mock returns 5 for both calls; under disagreement
it would observe the combined value, not the asserted client value. These tests give
false confidence on precisely the invariant they claim to protect.

**Fix:** In Test 17, add an assertion on the persisted paystub line item
(`fake_repo.load_line_items` â†’ `hours_overtime`) â€” it must equal the reply value
(`0` / `None`), NOT the combined `2`. Add the companion divergence tests for
`client_supplied` (reply `5`, combined `2` â†’ paystub must be `5`) and for the
reply-omits-employee `_unresolvable_asked` case. These assertions FAIL today and will
pass once CR-01 is fixed, making them genuine regression pins.

### WR-02: `_defer_field_regression_clarification` performs three `load_run` round-trips for data `_run_stages` already computed

**File:** `app/pipeline/orchestrator.py:621, 641` (plus the `name_to_id_post` build at 622-626)

**Issue:** The helper calls `repo.load_run(run_id)` at line 621 (for reconciliation),
then again at line 641 (for decision + extracted) â€” and the Round-2 caller already
has `stage.matches` in hand. The reconciliation it re-loads is the same `matches`
`_run_stages` just persisted; `name_to_id_post` could be built from `stage.matches`
without a DB round-trip, and the decision/extracted re-load duplicates state the
orchestrator just wrote. This is not a correctness bug (the persisted values are
authoritative and consistent within the single-threaded resume), but it adds three
redundant loads on the deferred path and couples the helper to DB read-back for data
it could receive as parameters. Given the project's DRY/explicit emphasis and that
`_RunStagesResult` already carries `matches`, threading the in-memory values is
cleaner and removes a class of "did the persisted copy drift?" reasoning.

**Fix:** Pass `stage.matches` into the helper and build `name_to_id_post` from it; the
decision/extracted re-load can stay (it is the persisted-state contract for
`_clarify`) or be replaced by passing the values `_run_stages` already has. Keep the
load-after-persist only where the persisted form is genuinely required.

## Info

### IN-01: WR-01 "conservative fail" comment overstates the guarantee

**File:** `app/pipeline/orchestrator.py:404-411, 447-456`

**Issue:** The comments on `_unresolvable_asked` claim "Worst case: under-fill that
re-clarifies next round" and "an unclassifiable asked field is NEVER re-backfilled
from the snapshot. Fail conservatively." As shown in CR-01 row 3, the field is not
under-filled when the combined extraction independently carries the original value â€”
it is paid at the original amount and the run advances to AWAITING_APPROVAL with the
field still marked `asked`, with no re-clarification. The comment describes the
intended invariant, not the actual behavior, which can mislead a future maintainer
into trusting a safety net that does not fire.

**Fix:** Once CR-01 is fixed (forcing the asked field to `None` so backfill_skip
genuinely under-fills, or re-deferring), the comment becomes accurate. Until then,
correct the comment to state the actual behavior, or implement the
force-to-None/re-defer so the comment holds.
