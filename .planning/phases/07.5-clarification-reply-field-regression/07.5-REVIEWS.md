# Cross-AI Plan Review — Phase 7.5

## Codex Review

**Summary**
Overall risk is **HIGH**. The plans understand the prior review findings, but Plan 03 reintroduces the core R3-1 failure: Round-2 backfill happens after `_run_stages` has already persisted, decided, and calculated line items in `orchestrator.py:277`. Plan 02 also does not actually satisfy the restated-name / same-`employee_id` diff requirement because detection is still keyed by `submitted_name` before employee-id mapping.

**Strengths**
- Plan 01 is a low-risk seam-prep change and preserves the current persist block/order.
- The plan set correctly identifies the major prior hazards: R3-1 ordering, R3-2 mixed issues, R3-3 name-vs-id diffing.
- `ClarifiedFields` typed-on-write JSONB is a pragmatic mitigation for safety-critical loop state.
- The test plan explicitly includes ordering, loop guard, tri-state, and mixed-issue coverage, which are the right properties to pin.

**Concerns**
- **HIGH: Plan 03 backfill is too late.** The Round-2 action calls `_run_stages(... prior=None ...)`, then classifies/backfills `stage.extracted` afterward. But `_run_stages` already calls `_compute_line_items` and `repo.replace_line_items` before returning on a process decision. Persisting backfilled extraction afterward does not recompute paystubs. This recreates R3-1.
- **HIGH: Plan 03 mixed-issue path still fails R3-2.** `_run_stages` defers `_clarify` only when `all_field_regression` is true. A mixed field-regression + missing/unresolved case calls `_clarify(... purpose="clarification")` immediately, before `asked` is written. Worse, an existing first clarification row can make `get_outbound_message_id(... purpose="clarification")` skip sending entirely.
- **HIGH: Plan 02 does not satisfy restated-name diffing.** `detect_field_regression` intersects original/resumed by `submitted_name`, so original `"M. Chen"` and reply `"Maria Chen"` never produce a raw drop. The required same-`employee_id` comparison must happen before diffing, using prior and current matches.
- **HIGH: Field-regression wording is only guaranteed in `_template_body`.** If the LLM draft returns non-empty text, `compose_clarification` returns it directly. The locked "Reply with the ... or 'none'" wording is not guaranteed on the real path.
- **MEDIUM: Plan 03 return shape is internally inconsistent.** `_RunStagesResult` is defined with `clarify_deferred` and `extracted`, but later uses `stage.decision`. It likely also needs `matches` and `issues`.
- **MEDIUM: `carried_forward` is overloaded.** A client replying with a positive value is classified as `carried_forward`, even though no carry-forward occurred. That weakens the provenance guarantee.
- **MEDIUM: Plan 04 claims hermetic rollback tests but describes live DB writes with manual cleanup.** Use the existing `seeded_db`/transaction fixture pattern or revise the claim.
- **MEDIUM: Approved-bytes test does not prove sent bytes.** Comparing DB before/after approval CAS does not exercise `_deliver`, confirmation composition, or PDF generation.
- **LOW: Schema CHECK migration should target the exact purpose constraint more defensively than `conname LIKE '%purpose%'`.**

**Suggestions**
- Reshape `_run_stages` around an actual pre-validate seam: extract/reconcile, optional backfill/classification, then validate/decide/persist/branch. Do not return to `resume_pipeline` for Round-2 backfill after calculation.
- Move field-regression diffing to employee-id keyed maps: reduce both prior and resumed extraction by resolved `employee_id`, then compare watched fields.
- Defer clarification when **any** field-regression issue exists, write `asked`, then send one clarification with `purpose="clarification_field_regression"` that includes all normal and field-regression asks.
- Make the required field-regression question deterministic: append it after any LLM draft or force it through a non-LLM template block.
- Add a terminal outcome like `client_supplied` or avoid showing "carried-forward" when the client supplied a positive replacement value.
- Make Plan 04 tests genuinely transactional/hermetic, and make the approved/sent invariant capture the actual outbound confirmation body/PDF source.

**Risk Assessment**
**HIGH** until Plans 02 and 03 are revised. As written, the phase can still silently compute paystubs from dropped fields, fail the restated-name requirement, and skip the mixed-issue follow-up email. Plan 01 is safe; Plan 04 has the right test intent but depends on fixing the orchestration design first.
