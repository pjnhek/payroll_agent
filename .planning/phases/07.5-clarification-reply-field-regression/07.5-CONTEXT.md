# Phase 7.5: Clarification-Reply Field-Regression (MONEY-03) - Context

**Gathered:** 2026-06-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver MONEY-03: a clarification *reply* that drops a money-affecting field the original
submission stated (original "40 + 2 OT" → reply "40", no OT) is **detected as a regression,
clarified exactly once** ("did you forget the overtime?"), then the original value is **carried
forward** into the computed paystub — or an explicit removal is **honored** — and the run
**never enters an infinite re-clarify loop**.

This phase is built on a **foundational refactor first**: `_run_stages` is split into
`(a) extract + reconcile` and `(b) validate + decide + persist + branch`, so the carry-forward
backfill can be injected *between* (a) and (b). Three cross-AI review rounds proved this is the
only correct seam (07-REVIEWS.md rounds 1–3; round-3 R3-1 confirmed against `orchestrator.py:277-289`).

**In scope:**
- Foundational split of `_run_stages` (+ updated `run_pipeline` / `resume_pipeline`), behavior pinned by existing suites BEFORE any feature code.
- `detect_field_regression` pure helper + `validate(prior=, resolved_drops=)` + `decide` Rule 2b + field-regression email copy.
- Two-inbound state machine: `pre_clarify_extracted` snapshot (snapshot-once), `clarified_fields` outcome column (`asked` / `carried_forward` / `confirmed_dropped`), `purpose="clarification_field_regression"`, asked-before-send durability, backfill-before-calc, loop guard.
- New JSONB columns + the `email_messages.purpose` CHECK-constraint expansion (live dev DB applied).
- Eval fixtures (16/17/18) for the judgment layer + integration tests for the state-machine layer.

**Out of scope (later v2 phases — cross-reference, do NOT double-build):**
- Full single-transaction wrapping of the resume sequence (pause-write + guard-write + persist) → **Phase 9** (Atomic Data Integrity). Phase 7.5 specifies the *ordering invariant* + the `IS NULL` snapshot guard only.
- Crash-idempotency + concurrency proof tests → **Phase 9 / Phase 10**. Phase 7.5 adds only a focused once-then-carry-forward loop-guard integration test for its own behavior.
- 401k / salary field-regression (D-08 excludes them — rarer, more false-positive surface).
- Sum (vs last-wins) multi-line reduction (last-wins is the locked default, D-12).

### Sequencing — the split-refactor seam (ROADMAP SC1)
- **D-7.5-01 (split is its own plan, pinned FIRST):** Plan A is a **pure refactor** — split `_run_stages` into `(a) extract+reconcile` and `(b) validate+decide+persist+branch`, update `run_pipeline` and `resume_pipeline`, and prove behavior-identical via the existing orchestrator/clarify/persistence suites — **committed GREEN before any MONEY-03 code lands.** No feature code interleaved with the refactor; cleanest bisect surface; directly satisfies ROADMAP SC1.
  - **D-7.5-01a (supersedes the seeds' approach):** The split moves the branch **OUT of `_run_stages`**, NOT a `StageResult`-returning single function that still branches internally. Round-3 R3-1 confirmed (against `orchestrator.py:277-289`) that a `StageResult` that still persists/computes/branches before returning leaves any post-return backfill too late → silent underpay survives. The seam MUST sit between reconcile and validate/decide/calc.
  - **D-7.5-01b (the seam is IN-MEMORY — no new persist between halves; verified 2026-06-28):** Source check of `orchestrator.py:269-289` confirms the four judgment stages (extract/reconcile/validate/decide) are **already pure/in-memory today**, and the FIRST DB write is `persist_extracted` at line 278 — *after* decide. There is **no intermediate persisted state** between reconcile and decide currently. Therefore the split MUST preserve that ordering: compute all pure stages → inject the carry-forward backfill **in the in-memory seam** → THEN run the existing persist block. **Do NOT persist extracted/reconciliation in stage (a) before stage (b) runs** — that would create a NEW crash window (extracted+reconciliation written, decision NULL, status still EXTRACTING) that cannot occur today, silently turning a "pure refactor" into a new partial-failure mode. The behavior-preservation test for Plan A is exactly: same DB writes, same order, same final state on every existing path.
  - **D-7.5-01c (crash-atomicity is Phase 9, not 7.5 — verified):** `_run_stages` is **already NON-atomic today** — it is 5 independent autocommit writes (`persist_extracted`, `persist_decision`, `persist_reconciliation`, `replace_line_items`, `set_status`×2; `orchestrator.py:278-287`, `repo.py:113-120` `_conn_ctx` opens a per-call transaction when `conn=None`). Wrapping the resume sequence in a single `with conn.transaction():` is **Phase 9 (DATA-01, which names `_run_stages` explicitly)** — do NOT add it here. 7.5 specifies only the *ordering invariant* (D-16/D-20) and the snapshot `IS NULL` guard (D-19). The split must not *worsen* atomicity (D-7.5-01b) but is not required to *fix* it.
  - **D-7.5-01d (consider the lighter shape — planning input, not locked):** Because the four stages are already pure and the first write is post-decide, the minimal viable fix may be an **inline pure `backfill(reconciled, snapshot) -> reconciled'` called between reconcile and validate in the EXISTING function** — no two-function restructure at all, zero change to the transaction/error boundary, full testability of the backfill as a pure money function. The planner should weigh this against the full two-way split: both satisfy D-7.5-01b's in-memory-seam requirement; the inline-pure-function shape has the smaller blast radius. A full declarative staged pipeline (`STAGES = [...]`, order-as-data) is the most robust against "ran too late" recurring but is a larger refactor than 7.5 should take on — note as a future option, do not adopt now.
- **Plans B+ (the feature, on the clean split):** detection → two-inbound state machine → eval fixtures + integration tests. Order follows D-30 priority: ship the two money-movers first (shared `_is_paid` predicate D-09 — already shipped in Phase 7; the carry-forward-vs-explicit-drop outcome D-13/D-14) before crash-safety hardening.

### Schema / migration mechanics
- **D-7.5-03 (DO-block migration + live apply):** `app/db/schema.sql` gets an **idempotent `DO $$` block**: `ADD COLUMN IF NOT EXISTS` for both `pre_clarify_extracted JSONB` and `clarified_fields JSONB`, plus an `email_messages.purpose` CHECK-constraint update to add `'clarification_field_regression'` (ADD COLUMN alone cannot alter an existing CHECK — this is the seed N4 fix). The migration MUST be applied to the **live dev Supabase DB** before integration tests run (Plan 04 is `autonomous: false` — `DATABASE_URL` user-setup gate).
  - **D-7.5-03a (migration safety — DROP+ADD in ONE transaction; from review):** The whole `DO $$` block runs in a single transaction so a failure can never leave the table with the purpose CHECK *dropped but not re-added* (a silent removal of a routing guard). **Declare the new CHECK explicitly** (full allowed-value list) rather than parsing the existing constraint text and reconstructing it — text-reconstruction is fragile. The introspect step finds the constraint *name* to drop; the re-add uses a literal declaration. Project convention is `schema.sql` as the source of truth (no Alembic — see CLAUDE.md); a versioned migration framework is out of scope for 7.5, but the single-transaction + explicit-declaration discipline is in scope.
  - **D-7.5-03b (`clarified_fields` is safety-critical state — validate on write; from review):** `clarified_fields` is BOTH the loop terminator AND the carry-vs-drop decision record — the state we least want free-form. Even though it lands in JSONB, it MUST be **constructed and validated through a Pydantic model on every write** (mirror the `FieldDrop`/outcome typing already in `contracts.py`), never raw `dict` assembly. A garbage/missing outcome doesn't get rejected by the DB; a mislabeled `carried_forward`→`confirmed_dropped` silently underpays. The `repo` setter for `clarified_fields` validates the shape before persisting. (Relational/CHECK-constrained columns would be even safer but are a larger schema change; typed-on-write JSONB is the in-scope mitigation — note relational as a future hardening.)

### Second-order partial drop — deferred (NOT "necessary for termination" — corrected after review)
- **D-7.5-06 (second-order partial drop — DEFERRED FOR SIMPLICITY, not a termination requirement):** The depth-1 design does **NOT** detect a *partial* second-order drop — a field the clarification *reply itself* newly drops while leaving other hours paid (e.g. reply restores OT but now omits holiday). **CORRECTION (peer review, 2026-06-28):** the earlier framing — "a deliberate trade for guaranteed loop termination" — was **wrong reasoning**, and the wrong label is what made this hole feel unavoidable. **Termination does NOT depend on comparing only against a round-1 snapshot.** It comes from the **per-field outcome lock (D-13):** each field transitions `none → asked → terminal` with no path back to `asked`; fields are a finite set; therefore total clarifications ≤ |watched hours fields| (a small constant) **regardless of what each reply is compared against.** A **running high-water-mark** comparison (compare each reply against the maximal set of values the client has asserted anywhere in the thread, not just round 1) would catch N-order drops with the *exact same* termination guarantee — the outcome lock still bounds it.
  - **Why depth-1 anyway (the HONEST reason):** implementation simplicity — high-water requires tracking asserted-value state for all watched fields across rounds, not just the originally-dropped field. That is a fine reason to ship depth-1 first. It is **not** "necessary for termination."
  - **Backstop in place:** MONEY-01's `_is_paid` zero-hours gate STILL runs on round 2 (only `resolved_drops` pairs are skipped), so a reply that leaves an employee with **no** paid hours is still gated — the silent-$0 path stays closed even at depth-1.
  - **The real guarantee is provenance, not detection depth (see D-7.5-08).** Detection is asymptotic — there is always an N+1-order case. What makes a missed drop *non-silent* is the operator seeing carried-forward provenance at the approval gate. Depth-1 detection + the zero-hours floor + provenance-at-gate + an informed human is the defensible system; deeper detection only changes how often the human gets a pre-filled answer.
  - **Future extension:** high-water-mark comparison (keyed off a `clarified_fields` generation) closes the residual with no termination cost — adopt if real usage shows chained partial drops. Out of scope for 7.5.

### Tri-state dependency (made explicit after review)
- **D-7.5-07 (silence-vs-explicit-zero rests on the extraction tri-state — VERIFIED, now explicit):** D-13/D-14's entire carry-vs-honor distinction depends on extraction reliably distinguishing three states for each hours field: *present with value* / *explicitly stated `0`* / *absent (not mentioned)*. **Verified against live source 2026-06-28:** the extraction model types each hours field `Decimal | None = Field(default=None, ge=0)` (`contracts.py:75-79`), the prompt explicitly instructs *"If the client did not mention a particular hours field… set it to null — NEVER 0 and NEVER omit it"* (`extract.py:31-32`, named "Pitfall 2" in the design), and `test_extract.py:58-60` proves absent→`None` survives extraction (never coerced to `0`). This dependency is **load-bearing and currently satisfied** — but it was implicit. The planner MUST (a) keep this dependency visible, and (b) ensure the eval/integration fixtures prove it **through the real extraction/serialization path** (D-24 / D-7.5-04c), so a future prompt or model change that collapses `0` and absent is caught by a failing test, not by a silent underpay. If that distinction ever regresses, D-13/D-14 collapse.

### Provenance at the approval gate (the review's best contribution — substrate already exists)
- **D-7.5-08 (a carried-forward value MUST be visibly labeled at the operator gate):** **Verified 2026-06-28:** the run-detail approval view is already a true 3-column "honest gate" — raw email | extracted data | computed paystubs side-by-side (`run_detail.html:104-193`) — and it already badges *name* resolution source (exact / stored-alias / unresolved, `run_detail.html:126-138`). What does NOT exist yet is **per-value provenance** on individual paystub/hours values. Since 7.5 introduces backfilled (carried-forward) values that the client did not state in the reply, the approval view MUST **label a carried-forward value as carried-forward** (and a `confirmed_dropped` field as honored-removal) so the operator approving never sees a backfilled number presented as if the client stated it. This turns the thesis from the un-achievable "detect every drop" into the achievable, already-architected **"no value reaches approval without visible provenance"** — and it is cheap because the 3-column substrate is built. Scope: a provenance tag/badge on the affected field(s) in the run-detail view + whatever minimal data the backfill must carry to render it. (Full per-field provenance for *every* value is a larger UI effort — 7.5 scope is the field-regression-touched values; note broader provenance as a future enhancement.)

### Ordering invariant — detect on RAW, then backfill, then calc (LOCKED after Codex review round 2, 2026-06-28)
- **D-7.5-10 (THREE-phase ordering — backfill must run AFTER detection AND before calc; and backfill must be employee_id-keyed):** Codex review round 2 (07.5-REVIEWS.md round 2) found that the round-1 R3-1 fix *overcorrected*. The fix moved the carry-forward backfill into the in-memory seam **before `validate()`** — but `validate()` is ALSO where field-regression detection runs (`detect_field_regression(prior, extracted)`, D-7.5-02/Plan 02). Verified against live source (`validate.py:81-140` inspects `extracted` directly; the 7.5 design calls `detect_field_regression` inside `validate()`): if `extracted` is **already backfilled** when `validate()` runs, detection compares the snapshot (OT=2) against the *backfilled* extraction (OT=2) → sees **no drop** → emits **no `field_regression` issue** → writes **no `asked`** → sends **no clarification** → the carried-forward value reaches calc and the operator **with NO provenance**, presented as client-stated. That silently defeats the phase's central thesis (D-7.5-08: "no value reaches approval without visible provenance") and SC2/SC3. **The root error of the entire thread, now named:** detection, backfill, and calc are **THREE ordered phases, not two.** Every prior revision argued only about "backfill vs calc"; the missing constraint is "backfill vs **detection**." The locked invariant:
  1. **DETECT on the RAW (pre-backfill) resumed extraction** — `detect_field_regression` runs against the unmodified reply so the OT `2 → None/0` drop is still visible. This produces BOTH the `field_regression` ValidationIssue(s) AND the set of (employee_id, field) pairs eligible for carry-forward.
  2. **BACKFILL** — apply carry-forward to the silence fields (employee_id-keyed, see below), AFTER detection has already recorded what was dropped, respecting `resolved_drops` (confirmed_dropped / client_supplied are NOT backfilled, D-14).
  3. **CALC** — `decide` → `_compute_line_items` see the backfilled extraction (R3-1 still satisfied: the carried value reaches the paystub).
  - **Implication for the seam:** detection must NOT be buried inside the same `validate()` call that consumes the backfilled `extracted`. Either (a) run `detect_field_regression` on the raw extraction as its own step BEFORE `backfill_extracted`, then run the rest of `validate()` on the backfilled data; or (b) pass the raw extraction to detection and the backfilled extraction to the missing/over-40 gates explicitly. The planner picks the shape; the invariant is non-negotiable: **the regression is detected from raw reply data, never from backfilled data.** A test MUST pin this: a first-resume OT `2 → None` case asserts a `field_regression` issue IS emitted and `asked` IS written (i.e. detection fired) AND, on the round that carries forward, the paystub shows OT=2 — proving detect-then-backfill, not backfill-then-blind.
  - **D-7.5-10a (backfill must be employee_id-keyed — completes the R3-3 fix):** Codex round 2 also found `backfill_extracted` still looks up the snapshot employee by `submitted_name` (`name_to_snapshot_emp.get(emp.submitted_name)`). Detection and `resolved_drops` were made employee_id-keyed (D-11/the iter-2 fix), but the **carry-forward lookup itself was not** — so "M. Chen" (snapshot) → "Maria Chen" (reply), same employee_id, finds no snapshot match → does not carry the value forward. The backfill's snapshot lookup MUST resolve both sides to `employee_id` (via prior_matches for the snapshot, current matches for the reply) and match on employee_id — identical keying to detection. The restated-name carry-forward is then proven by the same `test_restated_name...` integration test asserting OT=2 reaches the paystub (not just that the drop is detected).
  - **D-7.5-10b (`carried_forward` vs `client_supplied` must be classified from RAW reply, not backfilled data):** Codex round 2 (MED): the Round-2 outcome classifier reads the already-backfilled persisted extraction and labels `raw_value == snap_val` as `carried_forward` — so a client who explicitly re-supplies the SAME positive value is mislabeled `carried_forward` (a provenance error, D-7.5-08). Classification MUST read the RAW reply value (present-positive → `client_supplied`; silence/None → `carried_forward`; explicit-0 → `confirmed_dropped`) — the same raw extraction detection uses (step 1), captured BEFORE backfill mutates it. This is the same root cause as D-7.5-10: provenance/classification decisions read raw, calc reads backfilled.

### Answered-round ordering — classify FIRST, then suppress/backfill/detect/calc (LOCKED after Codex review round 3 + code spike, 2026-06-28)
- **D-7.5-11 (on a round where the client ANSWERED, outcome classification must run BEFORE detect/backfill/decide/calc — not after):** Codex review round 3 (07.5-REVIEWS.md round 3) found that the D-7.5-10 three-phase reorder, while correct for *first* detection (Round-1), strands the run on Round-2 (the answered round). **Verified against Plan 03 AND proven by a throwaway code spike** (`scratchpad/spike_round2_ordering.py`, run against live `Extracted`/`_is_paid`/`_HOURS_FIELDS`; do NOT merge the spike per the throwaway-spike note below):
  - The suppression set `resolved_drops` (`_resolved_by_name`) is built ONLY from TERMINAL outcomes (`confirmed_dropped`/`client_supplied`) — it EXCLUDES `"asked"` (Plan 03 Step E2). On Round-2 the just-answered field is still `"asked"` (it only becomes terminal in the classifier that runs AFTER `_run_stages`). So it is NOT suppressed.
  - Therefore `_run_stages` re-detects the drop (snapshot OT=2 vs reply OT=None), emits a `field_regression` issue, `decide` → `request_clarification`, the run DEFERS and skips calc. The classifier then writes `carried_forward`/`confirmed_dropped` — but there is NO second validate/decide/calc pass. The run re-clarifies (stranded at `AWAITING_REPLY`); the "AWAITING_APPROVAL" claim is false. **Spike confirmed:** current ordering → `request_clarification`/paystub=None; proposed → `process`/paystub OT=2.
  - **Explicit-zero overpay edge:** `backfill_extracted` treats silence-OR-explicit-zero as backfillable unless the field is already terminal in `resolved_drops`. With the answered field still `"asked"`, a naive "also suppress on asked" fix would let an explicit-zero the client just removed be RE-BACKFILLED → overpay. **Spike confirmed:** proposed ordering keeps explicit-zero OT=0 (NOT re-backfilled, `confirmed_dropped`).
  - **The locked invariant (the missing constraint, one level above D-7.5-10):** on an answered round (`clarified` non-empty for the field), the sequence is:
    1. **CLASSIFY the RAW reply** into a per-(employee_id, field) outcome FOR THE ASKED FIELDS ONLY (iterate the `clarified`/`asked` set — NOT every snapshot-paid field; the spike's all-fields shortcut mislabels untouched fields, e.g. `hours_regular`): present-positive → `client_supplied`; explicit `Decimal('0')` → `confirmed_dropped`; silence/`None` → `carried_forward`. Read from the RAW reply (D-7.5-10b), before any backfill.
    2. **DERIVE** from that classification: (a) the detection-suppression set = ALL answered asked fields (a field being resolved THIS round must not re-emit a `field_regression` issue → no re-clarify); (b) the backfill set = ONLY `carried_forward` fields (silence); `confirmed_dropped`/`client_supplied` are NOT backfilled (honor removal / use client value).
    3. **RUN** detect → backfill → validate → decide → calc ONCE with those sets. Detection is suppressed for all answered fields (no re-clarify), backfill applies only to `carried_forward`, the run reaches `AWAITING_APPROVAL` in a single pass.
    4. **PERSIST** the terminal outcomes (already computed in step 1, not after decide).
  - **Why this is locked, not discretionary:** it is the FOURTH consecutive ordering bug found in this thread (R3-1 backfill-vs-calc; R2-1 backfill-vs-detection; this: whole-pipeline-vs-classification). The recurring failure mode is "a decision computed too late relative to the pipeline that consumes it." The classify-FIRST ordering on answered rounds is the fix and was proven by code, not asserted in prose.
  - **Tests MUST pin (real `resume_pipeline`, DATABASE_URL):** (a) answered-silence → run reaches `AWAITING_APPROVAL` (NOT a second clarification) with the carried value in the paystub; (b) answered-explicit-zero → reaches `AWAITING_APPROVAL`, paystub field = 0, NOT re-backfilled (overpay guard); (c) answered-positive → uses the client value, labeled `client_supplied`.
