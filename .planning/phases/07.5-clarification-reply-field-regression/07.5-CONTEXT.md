# Phase 7.5: Clarification-Reply Field-Regression (MONEY-03) - Context

**Gathered:** 2026-06-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver MONEY-03: a clarification *reply* that drops a money-affecting field the original
submission stated (original "40 + 2 OT" → reply "40", no OT) is **detected as a regression,
clarified exactly once** ("did you forget the overtime?"), then the original value is **carried
forward** into the computed paystub — or an explicit removal is **honored** — and the run
**never enters an infinite re-clarify loop**.

This phase is built on a **foundational refactor first**: `_run_stages` is split into
`(a) extract + reconcile` and `(b) validate + decide + persist + branch`, so the carry-forward
backfill can be injected *between* (a) and (b). Three cross-AI review rounds proved this is the
only correct seam (07-REVIEWS.md rounds 1–3; round-3 R3-1 confirmed against `orchestrator.py:277-289`).

**In scope:**
- Foundational split of `_run_stages` (+ updated `run_pipeline` / `resume_pipeline`), behavior pinned by existing suites BEFORE any feature code.
- `detect_field_regression` pure helper + `validate(prior=, resolved_drops=)` + `decide` Rule 2b + field-regression email copy.
- Two-inbound state machine: `pre_clarify_extracted` snapshot (snapshot-once), `clarified_fields` outcome column (`asked` / `carried_forward` / `confirmed_dropped`), `purpose="clarification_field_regression"`, asked-before-send durability, backfill-before-calc, loop guard.
- New JSONB columns + the `email_messages.purpose` CHECK-constraint expansion (live dev DB applied).
- Eval fixtures (16/17/18) for the judgment layer + integration tests for the state-machine layer.

**Out of scope (later v2 phases — cross-reference, do NOT double-build):**
- Full single-transaction wrapping of the resume sequence (pause-write + guard-write + persist) → **Phase 9** (Atomic Data Integrity). Phase 7.5 specifies the *ordering invariant* + the `IS NULL` snapshot guard only.
- Crash-idempotency + concurrency proof tests → **Phase 9 / Phase 10**. Phase 7.5 adds only a focused once-then-carry-forward loop-guard integration test for its own behavior.
- 401k / salary field-regression (D-08 excludes them — rarer, more false-positive surface).
- Sum (vs last-wins) multi-line reduction (last-wins is the locked default, D-12).

</domain>

### Sequencing — the split-refactor seam (ROADMAP SC1)
- **D-7.5-01 (split is its own plan, pinned FIRST):** Plan A is a **pure refactor** — split `_run_stages` into `(a) extract+reconcile` and `(b) validate+decide+persist+branch`, update `run_pipeline` and `resume_pipeline`, and prove behavior-identical via the existing orchestrator/clarify/persistence suites — **committed GREEN before any MONEY-03 code lands.** No feature code interleaved with the refactor; cleanest bisect surface; directly satisfies ROADMAP SC1.
  - **D-7.5-01a (supersedes the seeds' approach):** The split is into **two functions with the branch moved OUT of `_run_stages`**, NOT a `StageResult`-returning single function that still branches internally. Round-3 R3-1 confirmed (against `orchestrator.py:277-289`) that a `StageResult` that still persists/computes/branches before returning leaves any post-return backfill too late → silent underpay survives. The seam MUST sit between reconcile and validate/decide/calc.
- **Plans B+ (the feature, on the clean split):** detection → two-inbound state machine → eval fixtures + integration tests. Order follows D-30 priority: ship the two money-movers first (shared `_is_paid` predicate D-09 — already shipped in Phase 7; the carry-forward-vs-explicit-drop outcome D-13/D-14) before crash-safety hardening.

### Seed-plan handling

  - **DISCARD from the seeds:** post-return backfill (R3-1), `clarify_deferred`-only-when-all-field_regression mixed-issue gap (R3-2), build-diff-by-`submitted_name` then filter-by-id (R3-3).

### Evidence gate (D-23 two layers)
- **D-7.5-04 (integration tests must PASS with live DB at the phase gate):** `skipif(not DATABASE_URL)` is acceptable **during development**, but the **phase-completion gate requires running WITH `DATABASE_URL` set** so the state-machine integration tests (snapshot-once, loop-guard, `confirmed_dropped` no-reflag, asked-before-send ordering, single-`_run_stages`-call invariant, D-26 explicit-drop) actually **execute and pass.** Skip ≠ evidence (D-23 / seed Finding #9). Matches the live-test pattern established in earlier phases (keys in shell env, not just `.env`).

### Phase-7 scaffolding reality check (verified against live source, 2026-06-28)
- **D-7.5-05 (what Phase 7 actually shipped vs. what 7.5 must create):** Phase 7 shipped `FieldDrop` (public model, `app/models/contracts.py:124`, now with `ge=0` money fields), the `field_regression` `ValidationIssue` Literal value (`app/models/roster.py:211`), and `_is_paid` (`app/pipeline/validate.py:38`). **`RawFieldDrop` was NOT shipped** — the re-scoped Phase 7 explicitly left it as a Phase 7.5 internal implementation detail (Plan 07-01 frontmatter). **Phase 7.5 creates `RawFieldDrop` itself**, alongside `detect_field_regression`. Confirm all three Phase-7 artifacts exist before building on them (they do, as of this CONTEXT).

### Re-affirmed accepted limitation
- **D-7.5-06 (second-order partial drop — ACCEPTED, re-affirmed):** The two-inbound design does **NOT** detect a *partial* second-order drop — a field the clarification *reply itself* newly drops while leaving other hours paid (e.g. reply restores OT but now omits holiday). Round 2 (the answer to "did you forget the OT?") runs without a prior, so it does not field-regression-detect the reply's own new drop. **Backstop in place:** MONEY-01's `_is_paid` zero-hours gate STILL runs on round 2 (only `resolved_drops` pairs are skipped), so a reply that leaves an employee with **no** paid hours is still gated to clarification — the silent-$0 path stays closed. The residual is a *partial* second-order drop. This is a **deliberate trade for guaranteed loop termination** (chasing drops inside a clarification answer risks the infinite re-clarify the phase exists to prevent). Document clearly in the writeup. Natural extension if real usage demands it: a bounded "one more round" guard keyed off a `clarified_fields` generation counter — out of scope for 7.5.

## Existing Code Insights

### Reusable Assets
- **`_is_paid(v)` (validate.py:38, shipped Phase 7):** the load-bearing shared predicate. `detect_field_regression` is its third call site (D-09) — uses it so `OT 2→0` is a *drop*, not a silent change.
- **`FieldDrop` (contracts.py:124, shipped):** public field-regression record with `ge=0` money fields and `extra="forbid"`. `validate()` maps raw drops → `FieldDrop` by `employee_id`.
- **`field_regression` ValidationIssue Literal (roster.py:211, shipped):** lets `validate()` emit a regression as "just another validation issue" so `decide.py` (mostly) already gates it.
- **`alias_candidates` scratch-column pattern:** `clarified_fields` (D-13) mirrors this existing nullable-JSONB scratch pattern on `payroll_runs`.
- **`claim_status` CAS (orchestrator.py:124):** double/duplicate replies are already serialized (`AWAITING_REPLY→EXTRACTING`, loser drops cleanly) — D-22 says do NOT re-add a concurrency guard here.

### Established Patterns
- **`decide.py` gates on any issue:** a single stray `field_regression` issue on pass 2 = infinite loop — so backfill MUST precede `validate(..., prior=snapshot)` so pass-2 detect emits nothing (D-20 termination ordering).
- **One `PaystubLineItem` per extracted entry, no dedup (calculate.py:286-289):** the diff must reduce-by-`employee_id` (last-wins, D-12) on BOTH snapshot and resumed extraction *before* diffing, or it manufactures/misses drops by line ordering.
- **Live-test discipline:** integration tests that touch the real DB need keys in the **shell env**, not just `.env` (established in earlier phases); the phase gate runs them for real (D-7.5-04).

### Integration Points
- The carry-forward backfill is injected in the **new seam** between split-stage (a) and (b) in `resume_pipeline` — the whole reason for Plan A.
- `pre_clarify_extracted` is written at the `awaiting_reply` pause (snapshot-once `IS NULL` guard, D-19); read-only on resume.
- The field-regression clarification send uses `purpose="clarification_field_regression"` (new CHECK value) and writes `clarified_fields[emp][field]="asked"` durably BEFORE the send (R3-2 / N2).

<specifics>
## Specific Ideas

- The headline scenario the whole phase is judged on: original **"40 + 2 OT"** → reply **"40"** (no OT) → detect → clarify once ("did you forget the overtime?") → carry OT=2 forward (silence) OR honor `confirmed_dropped` (explicit "remove it" / explicit `0`).
- The restated-name case MUST be covered: `"M. Chen" → "Maria Chen"`, both resolving to the SAME `employee_id`, is still diffed as a same-employee drop (R3-3 — reduce-by-`employee_id` before diff, not name-first).
- The discipline that paid off in Phase 7's review thread: **every cross-AI finding was verified against live source before being trusted.** Re-verify line numbers and `_run_stages` behavior against actual code when planning 7.5 — plan-text reasoning about this function has repeatedly drifted from reality.
