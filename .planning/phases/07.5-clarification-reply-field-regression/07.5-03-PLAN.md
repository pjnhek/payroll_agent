- **Finding 3 (HIGH — R3-2 mixed-issue path):** Clarification is now deferred whenever ANY `field_regression` issue exists in issues (not only when ALL are field_regression). When deferred, a single email with `purpose="clarification_field_regression"` is composed that bundles BOTH the normal unresolved/missing asks AND the field-regression ask. The `get_outbound_message_id` idempotency check uses the new purpose value so a pre-existing plain `"clarification"` row does NOT suppress the field-regression send.

- **Finding 5 (MEDIUM — _RunStagesResult shape):** The return shape carries `clarify_deferred: bool`, `matches: list[NameMatchResult]`, and `issues: list[ValidationIssue]` — exactly what `resume_pipeline` needs for Round-1 outcome classification. The `extracted` object is NOT returned (it is already persisted by `_run_stages` itself; callers that need it reload from DB to avoid stale references).

- **Finding 6 (MEDIUM — `carried_forward` overloaded):** A fourth outcome `"client_supplied"` is added to `ClarifiedFieldOutcome` for the case where the client replied with a POSITIVE replacement value. `"carried_forward"` now means ONLY that the value came from the snapshot (the client was silent). Three distinct outcomes: `carried_forward` (silence → backfill from snapshot), `confirmed_dropped` (explicit zero/none), `client_supplied` (positive replacement value). All three are loop-terminal (no path back to `"asked"`).

- **Finding 9 (LOW — schema CHECK migration matcher):** The `pg_constraint` lookup uses both `contype = 'c'` (CHECK) and `conrelid = 'email_messages'::regclass` to narrow the search, making the `LIKE '%purpose%'` pattern less likely to hit a wrong constraint on schemas with similarly-named tables.
