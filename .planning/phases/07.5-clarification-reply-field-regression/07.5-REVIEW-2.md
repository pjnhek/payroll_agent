## Critical Issues

### CR-01: Round-2 classify-first extracts from the COMBINED body — real extraction can restore the original value and OVERPAY on explicit-zero

**File:** `app/pipeline/orchestrator.py:364` (and `app/pipeline/orchestrator.py:586-599`, `app/llm/prompts/extract.py:38-56`)

**Issue:**
The entire D-7.5-11 classify-first overpay guard rests on this premise (stated in the
code comments at lines 367-432): the raw reply extraction reflects ONLY what the client
said in the *reply*, so silence → `None` → `carried_forward`, and an explicit `0` →
`Decimal('0')` → `confirmed_dropped`.

But the value actually classified is extracted from `combined_email`, not the reply:

```python
# orchestrator.py:364 (Round-2 else branch)
raw_extracted = extract(combined_email, roster, **extract_kwargs_r2)
```

`combined_email` is built by `_combined_context_email` (lines 586-599):

```python
combined_body = (
    "ORIGINAL PAYROLL EMAIL:\n"
    f"{original_body}\n\n"          # <-- still contains "Maria Chen 40 regular, 2 overtime"
    "CLARIFICATION REPLY FROM CLIENT:\n"
    f"{reply.body_text}"            # <-- "40 regular, 0 overtime this week"
)
```

The extraction prompt (`app/llm/prompts/extract.py`) is given this combined body with
NO instruction to prefer the reply, NO notion of "this is a clarification reply, take the
latest value." Its only rule is "extract each named employee's reported hours." Given a
body that literally states `2 overtime` in the original section, a correct extraction can
legitimately return `hours_overtime=2`.

Then `classify` (lines 422-432) reads `raw_val = getattr(raw_emp, 'hours_overtime')`:
- If the model returns `2` (from the original section) instead of `0` (from the reply),
  the field is classified `client_supplied` (positive). It is added to `backfill_skip`,
  but the raw extracted already carries `OT=2`, so `_compute_line_items` computes the
  paystub with **OT=2**.
- The client explicitly said "0 overtime." The system pays for 2. This is the **OVERPAY**
  the whole two-set model (`confirmed_dropped` in `backfill_skip`) was designed to prevent
  — and it is defeated upstream of the guard, at the extraction step.

For the silent-reply carry-forward case the money outcome happens to be the same (OT=2
either way), but the outcome label is still wrong (`client_supplied` instead of
`carried_forward`), corrupting the audit trail. The dangerous case is explicit-zero.

This is exactly the "fixture-vs-reality gap" pattern in MEMORY.md
(`live-gate-dateless-email-bug`, `review-traces-argflow-not-prose`): the deterministic
classify logic is correct in isolation, but it is fed a value from a context that violates
its precondition, and the test suite hand-feeds the precondition so the gap is invisible.

**Fix:**
Make the classify step read the reply *in isolation*, not the combined body. Two viable
approaches:

```python
# Option A (preferred): classify from a reply-only extraction; keep the combined
# extraction (which retains original employees/hours) ONLY for backfill/process.
reply_only_email = reply.model_copy(update={"body_text": reply.body_text})  # uncombined
raw_reply_extracted = extract(reply_only_email, roster, **extract_kwargs_r2)
# classify asked fields from raw_reply_extracted (silence on OT => None => carried_forward;
# explicit "0 overtime" => Decimal('0') => confirmed_dropped)
# ...then run _run_stages with the COMBINED extraction for the lossless process path.
```

```python
# Option B: teach the extraction prompt the reply semantics, e.g. a dedicated
# "extract ONLY values stated in the CLARIFICATION REPLY section; set every field the
# reply does not mention to null even if the ORIGINAL section stated it" instruction,
# and add a live/non-mock test that proves a combined body with original OT=2 + reply
# "0 overtime" classifies as confirmed_dropped (paystub OT=0), not client_supplied.
```

Whichever path is chosen, add at least one test that does NOT hand-script the extraction
to a reply-isolated result — score the real extract() over a genuine combined body and
assert the explicit-zero case lands at paystub OT=0.

### CR-02: A Round-2 reply that introduces a NEW field regression strands the run at `extracting` — no email sent, no recovery

**File:** `app/pipeline/orchestrator.py:491-507`

**Issue:**
The Round-2 branch calls `_run_stages` and then unconditionally assumes the run reached
`AWAITING_APPROVAL`:

```python
stage = _run_stages(run_id, combined_email, roster, ...)   # line 491
# D-7.5-11 STEP 4: persist the terminal outcomes ...
repo.set_clarified_fields(run_id, clarified)               # line 506
# Fall through to STEP C/D alias-diff (the run is at AWAITING_APPROVAL)   <-- assumption
```

But `_run_stages` returns `clarify_deferred=True` whenever `decision.final_action ==
"request_clarification"` AND any `field_regression` issue exists (lines 695-704), and in
that case it does NOT call `_clarify` and does NOT advance the status — it leaves the run
at `extracting` (set by `claim_status` at resume start, line 235), expecting the *caller*
to write 'asked' and send. The Round-1 branch honors this contract (lines 316-352); the
Round-2 branch IGNORES `stage.clarify_deferred` entirely.

Reachable scenario (single restated employee, two fields):
- Snapshot: Maria Chen `hours_overtime=2`, `hours_holiday=8`.
- Round-1 asked only about `hours_overtime` (the only field that dropped that round).
- Round-2 reply answers OT but now drops `hours_holiday` (8 → absent).
- `suppress_detection` contains only `(CHEN_ID, hours_overtime)`. `detect_field_regression`
  on the snapshot vs raw reply emits a NEW drop for `hours_holiday`, NOT suppressed.
- `validate` promotes it to a `field_regression` issue → `decide` → `request_clarification`
  → `_run_stages` sets `clarify_deferred=True`, sends NOTHING, leaves status `extracting`.
- The Round-2 branch falls through, persists `clarified`, runs the alias diff, and returns.

The run is now stuck at `extracting` with no clarification email, no `awaiting_reply`, and
no `awaiting_approval`. This violates INGEST-05 / D-A1-03 "nothing silently hangs" on a
money-moving path. The only recovery is the 5-minute stale-retrigger
(`main.py:560-596`), which restarts `run_pipeline` — re-extracting the ORIGINAL source
email only (`_run`/`load_inbound_email` reads the source email, `main.py` does not re-pass
the reply), silently DISCARDING the client's reply.

None of the 15 hermetic tests exercise a Round-2-introduces-new-regression case — every
scenario uses a single asked field (OT only), so the gap is untested.

**Fix:**
Mirror the Round-1 deferred-clarification handling inside the Round-2 branch:

```python
stage = _run_stages(run_id, combined_email, roster, ..., extracted=raw_extracted)

if stage.clarify_deferred:
    # A NEW field regression appeared this round. Write 'asked' for the newly-dropped
    # field(s) BEFORE the send (N2 ordering), then clarify under the field-regression
    # purpose — same as the Round-1 path (orchestrator.py:316-352).
    post_run = repo.load_run(run_id)
    name_to_id_r2 = {
        m["submitted_name"]: m["matched_employee_id"]
        for m in (post_run.get("reconciliation") or [])
        if isinstance(m, dict) and m.get("matched_employee_id")
    }
    for issue in (stage.issues or []):
        if issue.issue_type == "field_regression":
            parts = issue.field.rsplit(".", 1)
            if len(parts) == 2:
                sn, fn = parts
                eid = name_to_id_r2.get(sn)
                if eid:
                    clarified.setdefault(str(eid), {})[fn] = "asked"
    repo.set_clarified_fields(run_id, clarified)
    run_row = repo.load_run(run_id)
    persisted_decision = Decision.model_validate(run_row["decision"]) if run_row and run_row.get("decision") else None
    persisted_extracted = Extracted.model_validate(run_row["extracted_data"]) if run_row and run_row.get("extracted_data") else None
    if persisted_decision is not None and persisted_extracted is not None:
        _clarify(run_id, combined_email, persisted_decision, roster, persisted_extracted,
                 llm=llm, purpose="clarification_field_regression")
    return  # run is at AWAITING_REPLY — do not run the alias diff

# else: process path — persist terminals and fall through to alias diff.
repo.set_clarified_fields(run_id, clarified)
```

Add a test: snapshot with two positive fields, asked only one, Round-2 answers the asked
field and drops the other — assert the run reaches `awaiting_reply` (not `extracting`) and
a `clarification_field_regression` outbound row exists.

## Warnings

### WR-01: `name_to_id_for_classify` merge order contradicts its own comment (latent precedence bug)

**File:** `app/pipeline/orchestrator.py:383-387`

**Issue:**
The comment at lines 379-380 states "Current matches go FIRST so a restated name takes
the current resolution; prior_matches are the fallback." But the construction is a dict
comprehension over `list(current_matches_for_classify) + list(prior_matches)`, and a dict
comprehension is **last-wins**: for any `submitted_name` present in BOTH lists,
`prior_matches` (appended LAST) overwrites `current_matches`. The comment describes
first-wins precedence; the code does the opposite.

This is harmless in the common case (a restated same name reconciles to the same
employee_id in both rounds, so the override changes nothing) and the same misleading
comment is mirrored in the passing tests (`test_cr01_classify_union.py:133-137`), which is
why it survived. But it becomes a real misroute if the roster changed between rounds such
that one `submitted_name` resolves to a different employee in `current` vs `prior` — the
stale `prior` resolution would silently win.

**Fix:** Either make the code match the comment (current wins):
```python
name_to_id_for_classify = {
    m.submitted_name: str(m.matched_employee_id)
    for m in (list(prior_matches) + list(current_matches_for_classify))  # current LAST = current wins
    if m.resolved and m.matched_employee_id is not None
}
```
or correct the comment to state prior-wins and justify why prior is authoritative. Pick
one deliberately; do not leave code and comment disagreeing on a money-path lookup.

### WR-02: The same first-vs-last precedence bug exists in the R2-2 backfill merge

**File:** `app/pipeline/orchestrator.py:483-488`

**Issue:**
`prior_matches_for_backfill = list(snapshot_matches) + list(prior_matches)` is consumed by
`backfill_extracted`'s `name_to_id_prior` dict (orchestrator.py:110-114), which is again
last-wins. The comment (lines 483-487) correctly reasons that `prior_matches` should
override `snapshot_matches` for shared names and notes "both point to the same
employee_id" — so here the code and comment agree and the behavior is safe FOR THE
DOCUMENTED CASE.

The residual risk is the same as WR-01: if `snapshot_matches` (fresh reconcile of snapshot
names against the *current* roster) and `prior_matches` (the persisted snapshot-round
reconciliation) disagree on a shared name's employee_id because the roster changed between
rounds, the merge silently prefers the persisted prior. That is arguably the safer choice
(prior is the authoritative snapshot-round record), but it is undocumented as a deliberate
conflict-resolution rule and untested.

**Fix:** Add a one-line note that the prior-wins merge is the *intended* conflict
resolution (prior is authoritative for the snapshot round) and add a regression test where
`snapshot_matches` and `prior_matches` disagree, asserting `prior_matches` wins and no
cross-employee carry-forward occurs.

### WR-03: Eval `_score_fixture` omits the BACKFILL phase — the eval does NOT honor the production three-phase ordering

**File:** `eval/run_eval.py:178-188`

**Issue:**
The review brief asks to confirm "the three-phase ordering is honored in eval the same as
production." It is not. Production `_run_stages` runs **detect → backfill → validate(on
BACKFILLED) → decide** (orchestrator.py:646-680). The eval runs only **detect →
validate(on RAW, with raw_field_drops) → decide** (run_eval.py:178-188) — there is no
`backfill_extracted` call, and `validate` is passed neither `prior=` nor `prior_matches=`.

For the one committed field-regression fixture (`18_field_drop_clarify_carry_coastal.json`)
this produces the correct score because that fixture is a Round-1 detect-and-clarify case
(OT 2→None, expected `request_clarification`) where backfill is irrelevant to the decision.
But it means the eval has ZERO coverage of the Round-2 money path — no fixture exercises
`carried_forward` carry-forward into a paystub, `confirmed_dropped` zero-honoring, or the
`suppress_detection` two-set model. The "real, legible eval chart" the project optimizes
for therefore does not measure the most defect-prone code this phase added.

Additionally, because the eval's `any_hours` missing-gate runs on the RAW (non-backfilled)
extraction, a future fixture modeling a carried-forward employee (all hours None in the
reply, value only in the snapshot) would be scored as `missing` by the eval while
production processes it cleanly — a silent eval/production divergence.

**Fix:** Either (a) add the backfill phase to `_score_fixture` so the eval mirrors
production (`extracted = backfill_extracted(...)` between detect and validate, threading
`prior=`/`prior_matches=` into validate), and add a Round-2 carry-forward fixture; or
(b) explicitly document that the eval intentionally scores only the Round-1 detect path and
that the Round-2 money behavior is covered solely by the (currently LLM-mocked) integration
tests — and accept the gap consciously rather than implicitly.

### WR-04: Eval `--record` mode crashes on the new field-regression fixtures

**File:** `eval/run_eval.py:675-678`

**Issue:**
`_record_extraction` strips only `("expected", "fixture_category")` from the fixture before
`InboundEmail.model_validate(email_fields)`:

```python
email_fields = {
    k: v for k, v in raw.items() if k not in ("expected", "fixture_category")
}
email = InboundEmail.model_validate(email_fields)   # extra="forbid"
```

But `_load_fixture` (the scoring path) strips four keys including the MONEY-03 additions:
`{"expected", "fixture_category", "prior_extracted", "prior_matches"}` (run_eval.py:105).
Fixture `18_field_drop_clarify_carry_coastal.json` carries `prior_extracted` and
`prior_matches`. In `--record` mode those keys survive into `email_fields`, and
`InboundEmail` is `extra="forbid"`, so `model_validate` raises `ValidationError` and the
re-record run aborts on fixture 18.

`--record` is the only path that regenerates the extraction caches the whole eval depends
on, so this silently breaks cache regeneration for the new fixtures.

**Fix:** Use the same strip set as `_load_fixture`:
```python
_EVAL_KEYS = {"expected", "fixture_category", "prior_extracted", "prior_matches"}
email_fields = {k: v for k, v in raw.items() if k not in _EVAL_KEYS}
```
Better: extract the strip-set into one module constant and reuse it in both
`_load_fixture` and `_record_extraction` (DRY — the divergence is the bug).

### WR-05: 15 "hermetic integration" tests assert classify outcomes against a hand-scripted extraction that bypasses the real combined-body extraction (false confidence)

**File:** `tests/test_resume_pipeline.py:365-366`, `:910-912`, `:1050-1052`, `:1114-1116` (representative)

**Issue:**
Every Round-2 test feeds `mock_llm.script` a pre-isolated extraction, e.g.
`_extraction_json([{"submitted_name": "Maria Chen", "hours_regular": "40"}])` for the
"silent reply" case and `... "hours_overtime": "0"` for the explicit-zero case. The mock's
`_MockCompletions.create` (conftest.py:647-650) pops these verbatim regardless of the
prompt, so the tests never exercise `extract()` over the actual `combined_email` (original
+ reply). They therefore PROVE the deterministic classify/backfill logic given a perfectly
reply-isolated extraction — which is exactly the precondition CR-01 shows production does
NOT guarantee.

The tests are not wrong about the deterministic logic, but they create false confidence:
`test_answered_explicit_zero_not_rebackfilled` (line 1016) asserts no overpay, yet the
real overpay risk (model reads OT=2 from the original section) is scripted away. The
hermetic cleanup (function-scoped `fake_repo`, FIFO `mock_llm` reset) is sound; the issue
is the test boundary, not the fixtures' lifecycle.

**Fix:** Keep the deterministic-logic tests, but add at least one test that drives the
REAL `extract()` over a genuine combined body (original OT=2 + reply "0 overtime") — either
gated behind `ALLOW_LIVE_LLM` like the eval `--record` path, or with a mock whose
`create()` actually inspects the prompt and returns the reply-only value only when the
prompt instructs reply-isolation. This is the test that would have caught CR-01.

### IN-02: `_combined_context_email` does not bound `original_body` length before re-prompting

**File:** `app/pipeline/orchestrator.py:586-599`

**Issue:** `_combined_context_email` concatenates `original_body` (the full original
cleaned inbound) with the reply and re-submits the whole thing to extraction on every
resume. There is no length cap. The webhook caps inbound bodies for the demo composer
(`main.py:786`, 4000 chars) but the resume path has no equivalent guard, so a pathological
original body could grow the re-extraction prompt unbounded across rounds. Not a
correctness bug (extraction is single-shot, not accumulative) and out of strict v1 scope,
but worth a defensive cap given the body is re-sent each round.

**Fix:** Apply the same length ceiling used elsewhere (or `max_tokens`-aware truncation of
the ORIGINAL section) before building `combined_body`.
