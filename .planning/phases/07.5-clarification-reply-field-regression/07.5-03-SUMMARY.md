## Decisions Made

- D-7.5-11 classify-first (not post-decide reclassification): classifying after `_run_stages`/`decide` would leave the just-answered field as "asked" in `_resolved_by_name`, causing `_run_stages` to re-detect the drop and `decide → request_clarification`, stranding the run — classify-first is the fix
- TWO-SET MODEL (not single set): using one set for both N8 suppression and backfill would conflate "suppress re-detection" with "skip backfill", causing carried_forward fields to be skipped by backfill (silent underpay) while confirmed_dropped fields could be re-backfilled (overpay) — two separate sets with different membership rules prevent both failure modes
- CR-01 union lookup (current + prior matches): prior-only keying in the classify step missed restated names (e.g. "M. Chen" in snapshot, "Maria Chen" in reply, same employee_id) — without current matches, raw_emp stayed None, field stayed "asked", absent from backfill_skip, snapshot's positive value silently restored (overpay). The union with current_matches_for_classify covers all reply-side names; current goes first so a restated name takes current resolution, prior is fallback
- WR-01 conservative fail: when classify cannot resolve an employee in the raw reply even after the union lookup, the field is staged in `_unresolvable_asked` and absorbed into `backfill_skip` — worst case is under-fill that re-clarifies next round, never overpay from snapshot restore
- N1 logical invariant (two mutually-exclusive branches): the plan's N1 "single _run_stages call" invariant is satisfied at runtime — the Round-1 and Round-2 `_run_stages` calls are in mutually exclusive `if not is_round_2: / else:` branches; only one fires per invocation. The AST literal count is 2 (one per branch), which is architecturally required by the Round-1/Round-2 split
