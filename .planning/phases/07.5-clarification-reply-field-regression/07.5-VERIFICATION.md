## Post-Initial-Verification History

The initial verification (2026-06-28T12:00:00Z) returned `passed`. A second adversarial code review (07.5-REVIEW-2.md) subsequently found TWO REAL money-path blockers that the hermetic tests missed because they hand-scripted the LLM extraction and bypassed the real extraction call:

**CR-01 (overpay):** The classify-first step read `extract(combined_email, ...)` — the combined body still contains the original section with `hours_overtime=2`. A real LLM could legitimately return `OT=2` from the original section even when the reply said "0 overtime", classifying the field as `client_supplied` instead of `confirmed_dropped`. Paystub would compute `OT=2` — the exact overpay the confirmed_dropped guard was built to prevent.

**CR-02 (strand):** The Round-2 branch called `_run_stages` but never checked `stage.clarify_deferred`. If a Round-2 reply introduced a NEW (not-previously-asked) field regression, `_run_stages` returned `clarify_deferred=True` but the Round-2 branch fell through to `set_clarified_fields` + alias diff and returned — leaving the run at `extracting` with no clarification email sent. INGEST-05 violation: nothing silently hangs.

Both were fixed and pinned by new tests that drive the actual gap (not scripted around it). The full suite re-passed at 509/0 after fixes.
