DECLARE

## Key Notes for Planner

### D-7.5-01d inline-vs-split decision
The planner must choose between:
- **Inline pure `backfill()` function** called between `matches = reconcile_names(...)` and `issues = validate(...)` inside the EXISTING `_run_stages` — zero change to the transaction/error boundary, full testability as a pure money function. Recommended for minimum blast radius (D-7.5-01d).
- **Two-function split** (`_run_stages` returns `(extracted, matches)`, caller injects backfill). Larger structural change; use if the inline approach makes the function too opaque.

Both satisfy D-7.5-01b (in-memory seam, no new persist between halves). The inline option is explicitly the lower-risk choice per D-7.5-01d.

### D-7.5-03b `clarified_fields` validated on every write
The `set_clarified_fields` setter docstring already flags this requirement. The planner must add a `ClarifiedOutcome` or `ClarifiedFields` Pydantic model (mirroring `FieldDrop` style: `extra="forbid"`, typed outcome `Literal["asked", "carried_forward", "confirmed_dropped"]`) and ensure the orchestrator constructs/validates through it before calling `repo.set_clarified_fields`. This is safety-critical: a garbage outcome silently underpays.

### D-7.5-04a ordering test
The evidence-gate requires a test that fails IFF backfill runs after calc — specifically: a `carried_forward` case asserting the FINAL PAYSTUB LINE ITEM reflects the carried-forward OT value (e.g. OT=2, not OT=absent). This is stronger than outcome-only tests. Add to `test_resume_pipeline.py`.

### D-7.5-07 tri-state dependency
`detect_field_regression` and the `confirmed_dropped` classification BOTH depend on the extraction tri-state (present / explicit 0 / absent) preserved by `extract.py:31–32` ("null for absent, NEVER 0") and proven by `test_extract.py:58–60`. Do not regress this. Fixture 18's `prior_extracted` must be produced via `model_dump_json()` to ensure the `Decimal('0')` vs `None` distinction survives serialization (D-24 / D-7.5-04c).

### N6 snapshot identity check
When `prior_matches` is supplied, validate() must cross-check that the `submitted_name` from a `RawFieldDrop` resolves to the **same** `employee_id` in BOTH `prior_matches` and `matches`. A re-resolution to a different employee is NOT a regression — skip it (D-11). The `prior_matches=None` path is an accepted no-op (the N6 cross-check is bypassed when prior_matches is not threaded through — a deliberate bounded scope, pinned by a test).
