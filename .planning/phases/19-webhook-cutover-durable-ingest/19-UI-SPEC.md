## Design System

| Property | Value |
|----------|-------|
| Tool | Existing hand-authored CSS in `app/static/style.css`; no framework or build step |
| Preset | not applicable |
| Component library | Existing FastAPI + Jinja2 templates; no component package |
| Icon library | none; Phase 19 introduces no icon-only controls |
| Font | Inter with the existing system sans-serif fallback stack |

Phase 19 preserves the current Linear/Stripe-style dashboard language: pale gray page background, white table/card surfaces, compact pill badges, low-opacity elevation, and indigo primary actions. It introduces no SPA, JavaScript framework, dependency, third-party registry, or operations dashboard.

## Typography

Phase 19 additions declare exactly four sizes and two weights: regular (400) and semibold (600). Existing unrelated components keep their current styling.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.55 |
| Label / queue badge | 12px | 600 | 1.4 |
| Section heading | 18px | 600 | 1.3 |
| Page heading | 24px | 600 | 1.2 |

Queue state must remain visually subordinate to the primary payroll status: use the 12px label role for `Queued`, `Retry queued`, and `Running`; never render queue state at heading size.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#F7F8FA` | Page background and breathing room |
| Secondary (30%) | `#FFFFFF` | Navigation, tables, cards, form surfaces, and detail content |
| Accent (10%) | `#4F46E5` | Primary demo CTAs, keyboard focus rings, existing action links, and the soft `Running` queue indicator only |
| Destructive | `#DC2626` | Existing Reject action and bounded enqueue-failure presentation only |

Accent reserved for: `Run pipeline`, `Send Test Email`, keyboard focus rings, existing action links, and the `Running` queue badge. `Queued` and `Retry queued` use the existing neutral badge treatment so queue transport state never competes with payroll business state. Existing green/amber/red payroll semantics remain semantic colors, not additional decorative accents.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `Run pipeline` |
| Empty state heading | `No payroll runs yet` |
| Empty state body | `Use the Send Test Email button below to fire a demo fixture through the pipeline.` |
| Error state | `We couldn't queue this demo run. Please try again.` |
| Destructive confirmation | `Reject`: `Reject this payroll run? This cannot be undone.` |

Additional exact copy:

| State | Copy |
|-------|------|
| Curated fixture CTA | `Send Test Email` |
| Immediate pending job | `Queued` |
| Delayed pending job | `Retry queued` |
| Leased job | `Running` |
| Open-job durability note | `This action is durably saved; you can safely leave this page.` |
| Superseded operator submission | `An earlier resolution was already accepted. This submission was recorded but not applied.` |

Copy constraints:

- The demo enqueue error appears on the surface that owns the failed action and provides only the retry path above. It never includes exception text, database/provider details, job IDs, or PII.
- The superseded-submission notice appears after redirect on run detail and contains no submitted names, mappings, employee identifiers, or diagnostics.
- Polling timeout is not described as failure and exposes no recovery CTA. Auto-polling simply stops after 120 seconds; the page remains manually refreshable and the durable job remains owed.
- Do not show job IDs, attempt counts, raw diagnostics, sender addresses, or business payloads in queue badges, notices, or status JSON.

## Visual Hierarchy

### Runs list

1. The existing primary payroll-status badge remains the first and strongest status signal.
2. If a run has an open job, render one compact secondary queue badge beside it: `Running` if any open job is leased; otherwise `Queued` if pending work is immediately due; otherwise `Retry queued` if pending work is delayed.
3. Do not add queue data to the Summary column, add new table columns, or expose Phase 21 operations metrics.
4. Hide the secondary queue badge as soon as no pending or leased job remains.

### Run detail

1. Keep `Run Detail` and the primary payroll-status badge as the focal point.
2. Place the secondary queue badge beside or immediately below the primary badge, at the smaller label role.
3. While an open job exists, place the exact durability note immediately below the heading/status cluster using muted body text; it is informative, not a banner competing with payroll decisions.
4. Bounded redirect notices appear below the heading cluster and above existing decision/error banners.
5. Existing raw email, extracted data, paystub, conversation, approval, rejection, and retrigger hierarchy remains unchanged.

### Demo surfaces

- A successful composer or curated-fixture action redirects directly to `/runs/{run_id}`; the queue badge and durability note provide immediate confirmation on that real run.
- A failed atomic enqueue returns to the action-owning demo/runs surface and shows the bounded error copy above the relevant form.
- No modal, spinner overlay, progress stepper, job-details drawer, or new page is introduced.

## Interaction Contract

| Interaction | Contract |
|-------------|----------|
| Queue projection | Queue state is a separate safe projection and never changes or replaces `payroll_runs.status`. |
| Poll cadence | Poll `/runs/{run_id}/status` every 2 seconds while the payroll status is in flight or an open-job queue projection exists. |
| Poll cap | Stop after 60 attempts / 120 seconds. Timeout performs no enqueue, retrigger, or business-state mutation. |
| Detail update | Update the primary and queue badges from bounded status JSON; reload once when a meaningful payroll or open-job state change requires new server-rendered content. |
| List update | Update primary status, queue badge, and existing bounded failure projection in place without resetting form controls or scroll position. |
| Open-job exit | Hide the queue badge and durability note when no pending or leased job remains; existing success or safe-failure presentation becomes authoritative. |
| Competing resolution | After a losing valid submission, redirect to run detail with the exact bounded notice; never echo the submitted form values. |

Meaningful queue-state precedence is fixed: `Running` wins over both pending labels; among pending work, immediately due produces `Queued`, otherwise delayed produces `Retry queued`.

## Accessibility and Safety

- Every queue state is conveyed by exact text as well as color. Never rely on the badge dot or hue alone.
- Queue badge and durability-note containers use `aria-live="polite"` so in-place polling updates are announced without interrupting form entry.
- Preserve existing keyboard focus rings and labeled buttons. Phase 19 adds no icon-only actions.
- Bounded notices are ordinary server-rendered text with Jinja autoescaping; never render query parameters or diagnostic strings with `safe`.
- Preserve forms and scroll position during list polling. Detail may reload only on meaningful state change, never on every two-second tick.
- Queue state is secondary transport information. It must not be announced, styled, or worded as a payroll outcome.
