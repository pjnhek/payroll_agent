<objective>
Close the clarify-round hours-safety hole found in live run e6fa8643, in two parts:

1. BUG FIX. `detect_field_regression` is structurally blind to the very employee a
   name-clarification is about, because it builds its original-side identity map from
   `prior_matches` filtered to `resolved` — and the clarified employee was BY DEFINITION
   unresolved in the prior round. A dropped OT line on the clarification reply is silently
   accepted. Fix by repairing `prior_matches` itself (the actual defect), not by patching
   each consumer.

2. NEW DETECTION, DISPLAY-ONLY. There is no detection of a cross-round paid->paid VALUE
   CHANGE (20 -> 40 regular, 10 -> 2 OT). That change is arithmetically applied and never
   shown to the human who approves it. Surface it at the operator gate.

Purpose: every money-moving change a client makes across a clarification round is either
gated by code (a drop -> clarify) or visible to the human (a change -> banner). Neither is
silently swallowed.

Output: an identity bridge in alias_learning.py, a pure `detect_hours_changes` in
validate.py, a `hours_changes` JSONB column, an operator banner, and three RED-first tests.
</objective>

<the_bridge>
The one thing this plan must get right, stated explicitly because a naive reading gets it
wrong.

The prior round's token ("Sandy") and the reply's token ("Sandra Kim") are DIFFERENT
STRINGS. Three candidate bridges, two of which do NOT work — verified against live source:

- **Re-reconcile the prior token against the roster.** FAILS. `alias_learning.write_aliases_if_safe`
  is called from `delivery.deliver`, i.e. at the APPROVAL gate — so at resume time "Sandy"
  is still absent from `employees.known_aliases` and `reconcile_names("Sandy")` still
  returns unresolved. The alias does not exist yet.

- **Whole-run set-difference** ("employee S newly appears resolved AND the token vanished").
  FORBIDDEN. This is precisely the two-independent-facts inference that
  `alias_learning.bind_evidence_for_token`'s docstring exists to reject: the reply
  "No, Dave didn't work; David worked 5 hours separately" satisfies both facts and binds a
  match the client explicitly DENIED.

- **THE BRIDGE WE USE: same-record confirmation evidence, reusing `bind_evidence_for_token`
  verbatim.** `payroll_runs.alias_candidates` already persists `{token: {"suggested": <id>,
  "bound": <id>|None}}`, written by `clarification.clarify` before the clarification is sent.
  `bind_evidence_for_token(token, suggested_id, suggested_full_name, current_reconciliation)`
  returns True only when ONE reconciliation record ties both facts together: its normalized
  `submitted_name` equals the token's own text OR the suggested employee's canonical
  `full_name`, AND that same record is `resolved=True` with `matched_employee_id ==
  suggested_id`.

Why this is sound and not an LLM decision: the LLM's suggestion only PROPOSES employee S.
The deterministic resolver (`reconcile_names`, pure code over the roster) is what CONFIRMS
it. This is the exact same evidence standard already accepted for PERMANENTLY writing an
alias to the roster — a strictly higher-stakes action than diffing two hours values. Reusing
it here is strictly weaker-stakes, and reuses a function that is already money-audited.

Where the bridge is applied: to `prior_matches` ITSELF, once, before it is handed to its
three consumers (`detect_field_regression`, the new `detect_hours_changes`, and
`backfill_extracted`). The defect is that `prior_matches` is missing an identity — so fix
`prior_matches`. Seeding each consumer separately would let the three disagree about who the
snapshot employee is, which is the exact class of bug `validate.py`'s own `is_paid` docstring
warns about ("Keeping it shared is what stops the two rules from disagreeing").

Consequence to accept deliberately: `backfill_extracted` now also sees the bridged employee,
so a field the client went silent on in the confirming reply carries forward from the
snapshot. That is the LOSSLESS behavior the snapshot exists for, and it is what already
happens for every non-clarified employee today. It is a widening of correct behavior, not a
new risk: backfill only ever fires on a field whose CURRENT value is not paid.
</the_bridge>
